# Grade 5 — real 2026–2027 curriculum: production load (replicable)

Loads the real Grade 5 curriculum into a linked Supabase project and archives any
prior mock/demo curriculum. **Every step is idempotent — safe to re-run**, and
nothing is ever deleted (archiving is done via `is_active=false`, fully
reversible).

Source of truth for the data is the Grade 5 Curriculum folder; the SQL is
generated, not hand-written.

## 0. (Optional) regenerate the data SQL

```bash
python scripts/gen-grade5-real-data.py
# → rewrites scripts/_grade5-real-data.sql (35 units, 185 lessons, 108 standards)
```

## 1. Load (linked project — no psql / DB password needed)

Run in order with the Supabase CLI against the **linked** project:

```bash
supabase db query --linked -f supabase/seed-cloud.sql        # baseline: school, Grade 5, 8 subjects, 2025–2026 year
supabase db query --linked -f scripts/_grade5-real-data.sql  # real 2026–2027 curriculum (idempotent upserts)
supabase db query --linked -f scripts/grade5-finalize.sql    # make 2026–2027 the ONLY active year; archive prior year(s)
```

For an **atomic** load, concatenate the three files inside a single
`BEGIN; … COMMIT;` block and run once via `--file`.

## 2. Verify

```sql
select label, is_active from school_years order by is_active desc;  -- 1 active row: 2026–2027
select count(*) from units where school_year_id
  = '00000000-0000-0000-0000-0000000000c2';                         -- 35
select count(*) from master_core_lesson_events m
  join units u on u.id = m.unit_id
  where u.school_year_id = '00000000-0000-0000-0000-0000000000c2'
  and m.deleted_at is null;                                         -- 185
```

## Notes

- **Archived curriculum:** the prior 2025–2026 year is left **intact but
  inactive** — its units/lessons remain so it can be viewed as an archived
  curriculum once the app reads from Supabase
  (`NEXT_PUBLIC_PLANNER_USE_SUPABASE=1`).
- **Reset / re-run:** because every insert is `on conflict do nothing/update`,
  re-running the three files is a no-op on already-loaded rows.
- **Backup:** before the first prod load a JSON snapshot of the affected tables
  was taken; `supabase db dump` needs Docker (unavailable on the Windows box), so
  a logical `db query` snapshot was used instead.

## Day-level rebuild (2026-06-12)

Stage 2 replaces the 185 all-on-Sunday **week-summary** lessons above with
**1,054 real daily lessons** parsed from the teacher workbook
("Grade 5 Weekly Plans 2026-2027.xlsx" → `scripts/_grade5-day-extract.json`
via `scripts/extract-grade5-days.py`). Nothing is hard-deleted.

What `scripts/_grade5-daily-load.sql` does (generated by
`scripts/gen-grade5-daily-sql.py`; idempotent, safe to re-run):

1. **Subjects** — upserts the three planner subjects the daily plan needs
   (`spelling` …5d5, `ufli` …5d6, `sel` …5d8; seed values, no-ops in prod).
2. **Units** — 14 new unit rows for spelling/ufli/sel (one per thematic band
   that has lessons; `unit_id` is NOT NULL on lessons).
3. **Soft-delete** — sets `deleted_at = now()` on the 170 week-1-34 summary
   lessons, by explicit id list (read from `_grade5-real-data.sql`). The
   `deleted_at is null` guard makes re-runs no-ops. The whole file is **one
   transaction**, and this step first locks the 170 rows (aborting if any are
   missing) and **aborts the entire load** if any teacher-attached row
   references them — forks (`personal_core_lesson_event_copies`),
   `completion_status`, `boards`, `lesson_sections`, `comments`, or
   `resources` (all 0 at pre-flight; the guard re-checks at apply time so
   rows attached since the pre-flight can't be orphaned). Note the guard
   cannot prevent a write that lands *after* commit — today nothing can
   produce one (planner/Teach Supabase writes are not enabled, and the app
   never surfaces soft-deleted lessons), but apply during a quiet window
   anyway; schema-level active-parent guards are Phase 1B hardening.
4. **Daily lessons** — 1,054 upserts (`on conflict (id) do update` rewrites
   every content column and clears `deleted_at`, so re-runs heal partial
   loads). Literacy cells are split into reading / writing / grammar /
   spelling / ufli lessons by their inline headings; SEL rows combine into one
   `sel` lesson per day; math / explorers load as-is.
5. **Weeks 35-37** (Unit 7 Spiral Review, no workbook content) — the 15
   summary lessons are kept and spread one subject per day
   (math→sun, reading→mon, writing→tue, grammar→wed, explorers→thu), each
   update guarded by `day_of_week='sun'` so manual teacher moves are never
   clobbered.

### Dry run (nothing committed)

The load file already contains `begin; … commit;` — do **not** wrap it in
another `BEGIN` (the inner `commit;` would still apply it). Instead, swap the
final `commit;` for `rollback;`:

```bash
python - <<'EOF'
content = open('scripts/_grade5-daily-load.sql', encoding='utf-8').read()
head, sep, tail = content.rpartition('\ncommit;')
assert sep, 'commit; not found'
open('scripts/_grade5-daily-load.dryrun.sql', 'w', encoding='utf-8').write(
    head + '\nrollback;' + tail)
EOF
supabase db query --linked -f scripts/_grade5-daily-load.dryrun.sql
```

### Apply

```bash
supabase db query --linked -f scripts/_grade5-daily-load.sql
```

### Rollback (fully reversible)

```bash
# emits scripts/_grade5-daily-rollback.sql from the load file:
#  - ABORTS if teachers have already attached forks / completions / boards /
#    sections / comments / resources to the new daily lessons (review those
#    rows first; to force the rollback anyway, delete the guard block from the
#    generated file)
#  - un-soft-deletes the 170 week-summary lessons (section-2 id array)
#  - soft-deletes the new daily lessons (section-3 insert ids)
#  - moves the 12 spread unit-7 lessons (explicit section-4 ids) back to Sunday
python - <<'EOF'
import re
sql = open('scripts/_grade5-daily-load.sql', encoding='utf-8').read()
soft = re.search(r'ids uuid\[\] := array\[(.*?)\]::uuid\[\];', sql, re.S).group(1)
old_ids = re.findall(r"'[0-9a-f-]{36}'", soft)
new_ids = re.findall(r"insert into master_core_lesson_events\s*\n\s*\([^)]*\)\s*\n"
                     r"\s*values \('([0-9a-f-]{36})'", sql)
u7_ids = re.findall(r"where id = '([0-9a-f-]{36})' and day_of_week = 'sun'", sql)
assert len(old_ids) == 170 and len(new_ids) >= 1000 and len(u7_ids) == 12, \
    (len(old_ids), len(new_ids), len(u7_ids))
new_arr = "array[%s]::uuid[]" % ", ".join("'%s'" % i for i in new_ids)
guard = """do $$
declare ids uuid[] := %s; n bigint;
begin
  select (select count(*) from personal_core_lesson_event_copies where master_core_lesson_event_id = any(ids))
       + (select count(*) from completion_status where core_lesson_event_id = any(ids))
       + (select count(*) from boards where master_core_lesson_event_id = any(ids))
       + (select count(*) from lesson_sections where owner_kind = 'master' and owner_lesson_id = any(ids))
       + (select count(*) from comments where anchor_type = 'core_lesson_event' and anchor_id = any(ids::text[]))
       + (select count(*) from resources where owner_event_type = 'core_lesson_event' and owner_event_id = any(ids))
       into n;
  if n > 0 then
    raise exception 'rollback aborted: %% teacher-attached row(s) reference the daily lessons — review them first', n;
  end if;
end $$;""" % new_arr
out = ["begin;", guard,
       "update master_core_lesson_events set deleted_at = null, updated_at = now()"
       " where id in (%s);" % ", ".join(old_ids),
       "update master_core_lesson_events set deleted_at = now(), updated_at = now()"
       " where deleted_at is null and id = any(%s);" % new_arr,
       "update master_core_lesson_events set day_of_week = 'sun', updated_at = now()"
       " where day_of_week <> 'sun' and id in (%s);"
       % ", ".join("'%s'" % i for i in u7_ids),
       "commit;"]
open('scripts/_grade5-daily-rollback.sql', 'w', encoding='utf-8').write("\n".join(out) + "\n")
print(f"rollback: restore {len(old_ids)} summaries, archive {len(new_ids)} daily lessons, "
      f"re-Sunday {len(u7_ids)} unit-7 lessons")
EOF
supabase db query --linked -f scripts/_grade5-daily-rollback.sql
```

### Notes / provenance

- **Id scheme:** new lessons use slugs `d-w<week>-<subject>-<day>` through the
  same `slug_uuid("lesson", …)` bridge as the prior load — namespaced so they
  can never collide with the old `g5-2627-…-w<week>` ids.
- **SEL day cells** are not materialized in `_grade5-day-extract.json` (its
  `unmapped` block only summarizes them), so the generator re-walks the
  workbook for the Morning Meeting / SE Learning / Closing Circle rows only.
- **Standards:** codes are parsed with the prior generator's parser; only
  codes already present in the prior load's 108 `standards` rows are tagged as
  uuids. Any standards text that doesn't fully resolve is kept verbatim in
  `notes` ("Std: …") — never dropped (the workbook's prose-style standards
  rarely match the prior load's code spellings).
- **Pre-flight checks run 2026-06-12 (read-only):** all 170 ids exist in prod
  on `sun`, active; 0 rows reference them in any dependent table
  (`personal_core_lesson_event_copies`, `completion_status`, `boards`,
  `lesson_sections`, `comments`, `resources`); the 15 unit-7 ids exist on
  `sun`; `grade_level_id` on lessons is trigger-derived from `unit_id` (the
  load correctly does not set it).
- **SEL source caveat:** the generator re-reads SEL rows from the live
  workbook while math/literacy/explorers come from the frozen Stage-1 JSON.
  The structural drift guard fails closed on re-organized workbooks, but a
  content-only edit could skew SEL newer than the rest; the supported
  regeneration path is always `extract-grade5-days.py` **then**
  `gen-grade5-daily-sql.py` in the same sitting.
