# MyCurricula UI/UX Audit Report

**Site:** https://mycurricula.app
**Pages audited:** /weekly, /daily, /subject
**Date:** 2026-05-21
**Stack observed:** Next.js (CSS-modules + Tailwind utilities), Supabase auth, Cloudflare in front
**Tester:** Claude (Chrome MCP + computer-use)
**Viewport:** Tested at 1296×830 inner. Attempts to drop to mobile (375/414/480px) failed because the OS-level window minimum on this machine clamps the inner viewport at 1296. Recommendations for mobile are based on inspection of the responsive media queries already in the bundle (breakpoints found: 440, 480, 540, 640, 720, 768, 920, 1024, 1280, 1536) but were not visually verified on a true mobile viewport — please re-run the mobile sweep with DevTools device-emulation enabled.
**Audience for this report:** Claude Code agent + front-end design plugin. Each finding has a stable ID, severity, exact CSS-module hook (when available), the verified source file (`Appendix A`), and a concrete fix sketch so the items can be filed straight into the implementation queue.

**Repo root:** `C:\Users\losey\Projects\mycurricula.app` (Next.js app router; `app/`, `components/`, `lib/`, `styles/`, `supabase/`).

## 1. Executive Summary — Top Findings

## 2. Critical Bugs (must fix before any public launch)

### BUG-001 · Right-rail widget covers grid columns and lesson banner

- **Where:** All three pages — most visible on `/weekly` (Wednesday/Thursday cards) and `/daily` (lesson banner clock).
- **Source:** `components/daily/RightRail.tsx` + `components/daily/RightRail.module.css`; affected views: `components/grid/WeeklyGrid.tsx`, `components/daily/DailyView.tsx`; orchestration: `components/weekly/WeeklyShell.tsx`.
- **Selectors:** `.RightRail_rail__LR0jk` (rail), `.WeeklyGrid_grid__0RluM` (grid), `.DailyView_lessonSubjectLabel__ZeHTB` (banner).
- **Measured:** Grid bounding rect `{x: 76, width: 1082, right: 1158}`. Rail bounding rect `{x: 960, width: 320, right: 1281}`. Overlap: 198 px at 1296 px inner viewport.
- **Symptom 1 (user-reported clock issue):** On `/daily`, the lesson banner renders `MATH | Equivalent fractions warm-up | 8::` because the time `8:10–9:10` is painted on the right side of the banner directly under the rail.
- **Symptom 2:** On `/weekly`, the Wednesday column is half-covered. Even when the user collapses the Resources widget via the `‹` chevron, the rail's 320 px footprint stays — only the inner widget bodies hide.
- **Steps to reproduce:** Load `/weekly` or `/daily` at any viewport ≤ 1400 px logical width.
- **Expected:** Grid reflows to the space remaining beside the rail; OR rail is a true overlay that auto-hides at narrower widths; OR there is a clear "hide rail" toggle that removes the 320 px footprint entirely.
- **Fix sketch:** Wrap the planner area + rail in a CSS Grid container: `grid-template-columns: 1fr 320px`. Replace the rail's `‹` chevron with a true hide control that toggles a `--rail-w` custom property between `320px` and `32px`. Below 1280 px CSS width, hide the rail and surface it as a bottom sheet via the existing top-bar to-do / comments icons.
- **Acceptance:** Rail no longer overlaps grid at viewports 1024–1920; verified by `getBoundingClientRect()` on `.RightRail_rail__LR0jk` vs `.WeeklyGrid_grid__0RluM` not overlapping, and the `/daily` lesson banner reads full `8:10–9:10` with no clipping.

### BUG-002 · Title-band button overlap after a card is moved

- **Where:** `/weekly`. Triggered any time a card has more than 3 status elements simultaneously.
- **Source:** `components/weekly/weekly-lesson-card.tsx` + `components/weekly/weekly-lesson-card.module.css`.
- **Selectors:** `.weekly-lesson-card_band__1STK9` (band), `.weekly-lesson-card_affordance__h_CpE` (drag handle + ⋯).
- **Measured:** Band width 173.8 px; H3 title inside takes 150.8 px; only ~18 px remains for any right-side affordance set.
- **Trigger combinations seen:**
  - Sunday Math card (clean): `MA | 8:10 | ⋮⋮ | 0 | ⋯` → renders `MA 8:10-⋮⋮0 ⋯` (handle squashed between time and counter).
  - Tuesday Math card (with checkbox): `MA | ✓ | 8:10 | ⋮⋮ | 0 | ⋯` → even tighter.
  - Wednesday Math card (modified + swap): `MA | ⇄ | 10:00 | ⋮⋮ | 0 | ⋯` → renders `MA ⇄-10-⋮⋮0 ⋯`; swap icon and time digits run together.
  - Monday Math card (modified-only): `MA | MODIFIED | ⋮⋮` → time and counter disappear entirely behind the MODIFIED pill.
- **Root cause:** Flex children inside the band have no `min-width` or priority ordering, and the band itself has a hard width tied to the grid cell. When a card is moved, the MODIFIED pill or `⇄` icon is inserted into the same flex row and pushes everything else into ~6 px of negative space.
- **Fix sketch (preferred — 2-row title band):** Restructure the band as two rows: row 1 = `[subject pill] [time] ... [drag] [⋯]`; row 2 (only when modified/has counter) = `[MODIFIED] [counter] [status icons]`. Keeps the visual rhythm of the existing card and stops the right-side affordances fighting the time.
- **Fix sketch (minimal — flex priority):** Give the band `display: flex; gap: 4px;`. Order children: subject pill (`flex: 0 0 auto`), status badges (`flex: 0 1 auto; min-width: 0`), spacer (`flex: 1`), drag/more (`flex: 0 0 auto`). Set the MODIFIED pill to `position: absolute; bottom: -10px` so it never competes for the title row, OR hide the time when MODIFIED is present and surface it in the expanded view only.
- **Acceptance:** Across all card states (clean / modified / swap / has-counter / has-checkbox), no two interactive elements in the title band overlap; verified by `getBoundingClientRect()` non-intersection for every pair, and a Storybook story per state asserts visual layout.

### BUG-003 · Sticky subject row labels ghost over scrolled content

- **Where:** `/weekly` when the grid is scrolled horizontally (Sun–Thu won't all fit at 1296 px).
- **Source:** `components/grid/WeeklyGrid.tsx` + `components/grid/WeeklyGrid.module.css`.
- **Selectors:** `.WeeklyGrid_cornerCell__tDuO3` and the subject row-head cells.
- **Symptom:** Subject icon (`Ma`/`Re`/`Wr`) and label remain pinned to the left as cards scroll past, but the cells' background is `transparent`. Sunday column cards bleed through behind the labels (visible "ions", "rer", "ess re" letterforms behind the Math/Reading/Writing pills).
- **Fix sketch:** Give the sticky column an opaque background-token (`var(--surface-canvas)` or whatever neutral the rest of the canvas uses) and a hairline right border to match the card column separators.
- **Acceptance:** With the grid scrolled to reveal Thursday, the row-head cells render fully opaque; verified visually and by computed `background-color` ≠ `rgba(0,0,0,0)` / `transparent`.

### BUG-004 · STANDARDS filter list stays Math-flavoured after switching to Reading

- **Where:** `/subject`, left filter rail under "STANDARDS".
- **Source:** `components/shell/left-filter-panel.tsx` (filter rail); standards data from `lib/mock/standards.ts`; bound through `lib/app-state.ts`.

### BUG-005 · Right Resources pane does not respond to lesson selection on /weekly (user-reported)

- **Where:** `/weekly`. Repro: click any lesson card (e.g. Sunday → Math → "Equivalent fractions warm-up"). The card expands inline and shows its own resources (Fraction Basics · Video, What is a Fraction? · Link, Fractions Overview · Doc, Khan Academy · Link, Fraction Wall…). But the right-rail `Resources · Week 12` header and count (`34`) do not change.
- **Source:** `components/daily/RightRail.tsx` (rail shell), `components/daily/ResourcesPanel.tsx` (Resources widget body — **this is the file that needs lesson-scoping**), `components/weekly/WeeklyShell.tsx` (passes scope to rail), `lib/planner-store.ts` (add `selectedLessonId`).

- **Why it matters:** With the rail anchored at the week scope, a teacher previewing Sunday's math lesson sees a 34-item resource list — most from other subjects/days. Signal-to-noise is ~5/34 for a typical lesson.
- **Fix sketch:**

  - Expose a "Back to week" link in the header to revert to aggregate.
  - On click outside / Esc / scroll to a different week, clear `selectedLessonId`.
- **Acceptance:** On `/weekly`, clicking a lesson card changes the right-rail Resources title to that lesson's title and the count to that lesson's resource count within ≤ 100 ms; clicking the canvas / pressing Esc reverts to the week aggregate.

### BUG-006 · Resources defined inline on weekly/daily lessons are not surfaced in the rail Resources pane (user-reported)

- **Where:** `/weekly` and `/daily`. The Sunday Math card lists 5 inline resources; `/daily` for the same lesson shows a richer inline RESOURCES grid plus a MORE RESOURCES list (Fraction Wall Poster · PDF, Anchor Chart Template · DOCX, Fraction Examples · PDF). The right-rail Resources widget should aggregate these into the lesson event's resource pane when the lesson is selected (per BUG-005), but currently the inline lists and the rail list are independent stores.
- **Source:** `lib/types.ts` (canonical `Lesson` / `Resource` types — needs `LessonEvent.resources[]` array), `components/daily/ResourcesPanel.tsx`, `components/weekly/weekly-lesson-card.tsx`, `components/daily/LessonDetail.tsx`.
- **Expected (per user):** Resources attached at the weekly card and on the daily lesson should both flow into the single canonical "Resources for this lesson event" view in the rail. One bag of resources per lesson event — no surface-specific stores.
- **Fix sketch:**
  - Single `resources[]` array on the `LessonEvent` model in `lib/types.ts`.
  - Every surface that lists resources for that lesson event reads from the same array.
  - The `+ Add resource` affordance on each surface writes to the same array.
  - If a resource is week-scoped (not lesson-scoped), make the distinction explicit (`this week` pill vs `this lesson` pill).
- **Why it matters:** Resource fragmentation is the #1 reason teacher-planning tools accumulate "junk drawer" data. One canonical list per lesson event is the highest-leverage data-model decision in this product.
- **Acceptance:** Adding a resource via any surface (weekly card, daily lesson detail, rail) makes it appear in the other two surfaces immediately (verified by an integration test that adds a resource from the weekly card and asserts the daily detail's MORE RESOURCES list contains it).

## 4. UX Findings — Heuristic Evaluation

### 4.1 Visibility of system status
**Strong:** Active week labeled `THIS WEEK`; "This week" button disables when already on current week; card-pagination footer shows "1 of 2"; Master mode shows a red "Heads up — changes here affect the whole team" banner.

**Gap:** No persistent save indicator. After editing a lesson title in place, the user has no visual confirmation that the change was saved/synced. Suggest a tiny "Saved · 09:42" timestamp pinned to the right of the page title, mirroring Notion/Google Docs.

**Gap:** Undo/Redo icons are correctly disabled when there's nothing to undo, but they live in the top bar between a date-nav cluster and a Personal/Master cluster — they read as part of navigation. Group them visually with the search or in a "History" mini-cluster.

### 4.2 Match between system and the real world
**Strong:** Vocabulary is teacher-native: "I CAN", "Standards", "Directions", "Notes", "Lesson plan", "Resources", "Day Shoutbox". Card sub-pagination "1 of 2 / 1 of 3" maps to how teachers think about same-day multiple sessions.

**Confirmed by team:** Default working week is per-teacher (chosen during onboarding, supports A-day / AB-day / other rotations). The Sun–Thu default is just the test account.

### 4.3 User control and freedom
**Strong:** Undo/Redo present in the top bar and verified working (Mark-done can be undone). Cards expand/collapse individually and via "Expand all / Minimize all". Escape correctly cancels in-place title edits.

**Gap:** No confirmation before destructive actions (untested — but the `⋯` menu deliberately omits Delete, which is a positive signal that destructive actions go through a different path).

### 4.4 Consistency and standards
**Strong:** Card colours match subject (Math = blue, Reading = green, Writing = purple, Grammar = orange, Spelling = pink, UFLI = peach). Same palette is used in `/daily` lesson list and `/subject` sidebar.

**Gap:** `By UnitBy Week` and `Personal• Master` toggles look like typos (no separator). The rest of the app uses pills with clear boundaries. Top-bar tabs are mixed-case ("Weekly", "Daily", "Subject") while sub-headers are all-caps ("WEEKLY PLAN", "LESSONS", "TODAY'S EVENTS"). Pick one convention. Resource file-type chips use uppercase (`DOC`, `LINK`, `PDF`, `SLIDES`) but `Video` filter button is mixed-case.

### 4.5 Error prevention and recovery
**To verify:** Destructive actions in the card `⋯` menu — none exposed in this audit, which is good (deliberate omission).

**Gap:** BUG-004 (stale STANDARDS filter on subject switch) is a recovery problem — users see empty results and think the page is broken.

### 4.6 Recognition rather than recall
**Strong:** Resource types are tagged inline so the user doesn't have to remember.

**Gap:** The icon rail on the far left ("Today / Schedule / To-dos / Year / Voice note / Settings") is icon-only and the icons toggle visual state on click but most don't navigate. Add labels on hover, or render text labels at viewports ≥ 1024 px.

**Gap:** The MODIFIED badge tells the user a card was moved, but nothing tells them when, why, or by whom. A tooltip on MODIFIED with "Moved from Mon → Tue by Tim, 9:15 AM" would be high-value at low cost.

### 4.7 Flexibility and efficiency of use
**Strong:** Search input in the top bar exists; per-card shortcuts ⌘D (Duplicate) and ⌘P (Print this lesson) shown in the `⋯` menu.

**Gap:** No global command palette (tested `?`, `/`, `Cmd+K` — none opened anything). Teachers using this daily will outgrow the mouse fast. Suggest:
- `Cmd+K` → command palette / search
- `[` / `]` → previous / next week
- `T` → jump to today
- `1` / `2` / `3` → Weekly / Daily / Subject
- `E` → expand all, `M` → minimize all
- `/` → focus the search input

**Gap:** No bulk operations visible. Selecting multiple cards (Shift-click? Cmd-click?) to move or duplicate a whole day to next week would be a teacher's biggest weekly time save.

### 4.8 Aesthetic and minimalist design
**Strong:** Lots of whitespace; card colour-coding does most of the visual work; chips do the labeling.

**Gap:** Card title band is too dense (BUG-002) — six tap targets in a single ~173 px row. Right rail with three stacked widgets is heavy; consider tabs across the top of the rail so only one widget is full-height at a time.

### 4.9 Help users recognise, diagnose, recover from errors
**Untested:** No error states were intentionally triggered (no network throttle, no rapid double-submit). Strongly recommend a sweep with offline / 500-response mocks to verify error UI exists.

### 4.10 Help and documentation
**Gap:** No `?` or `Help` affordance found in the top bar or icon rail. Wire `?` keyboard shortcut to a Shortcuts overlay; add a `Help` button to the Settings icon's menu. The team has confirmed that per-page onboarding guidance will be built throughout.

### 4.11 Accessibility (WCAG 2.2 AA spot-checks)

**Strong:** ARIA labels are systematic on every icon button (`Previous week`, `Mark "Equivalent fractions warm-up" — current status not_done`, `Open comments panel (3 unread)`, etc.). Grid uses `role="grid"` with `role="columnheader"` on day headers.

- **A11Y-001:** Day header aria-label `"SundaySun"` runs together — see POLISH-005. **Source:** `components/grid/WeeklyGrid.tsx`.
- **A11Y-002:** Body text inside expanded cards (light-grey-on-tinted-pastel) was not contrast-measured but visually looks borderline. Run axe-core / Storybook a11y addon over each card variant. **Source:** `components/weekly/weekly-lesson-card.tsx`.
- **A11Y-003:** Daily lesson drag handles (`⋮⋮`) are < 12 px wide — below WCAG 2.5.5 target-size minimum. **Source:** `components/daily/LeftRail.tsx`.
- **A11Y-004:** No skip-to-content link detected. **Source:** `app/(planner)/layout.tsx` (or root layout).
- **A11Y-005:** Focus order through the top bar was not stepped manually. Verify Tab order: skip → logo → tabs → page-nav → undo/redo → edit-mode → view-mode → search → utility icons. **Source:** `components/shell/top-bar.tsx`.
- **A11Y-006:** No `prefers-reduced-motion` test performed. The bundle has a `@media (prefers-reduced-motion: reduce)` block (good) — sanity-check that the move animations on `/weekly` honour it. **Source:** `components/weekly/weekly-lesson-card.tsx`, `components/grid/WeeklyGrid.tsx`.

## 5. Cross-Page Comparison: Weekly ↔ Daily ↔ Subject

### Navigation consistency
All three pages share the same top bar; Schedule / Unit / Year SOON tabs are present on all three. The right rail (Resources / To-do / Day Shoutbox) appears on `/weekly` and `/daily` but not `/subject`. This is correct intent (subject view is focused), but the *transition* is jarring — the page width visibly snaps back to use the full canvas when entering `/subject`. Either keep the rail on `/subject` as a "Subject resources" widget, or animate the transition.

### Data flow
Switching from `/weekly` (Week 11) to `/daily` preserves the week; the Daily date picker correctly defaults to the current day inside that week. **Key finding:** the Daily page already scopes the right-rail Resources to the selected lesson (changes `7 → 2` when switching Math → Reading). The Weekly page should match this pattern — BUG-005. Editing a card title on `/weekly` would propagate to `/daily` lesson list — full title preserved in DOM but the column is too narrow (POLISH-007).

### Visual styling
Subject colours are consistent across the three pages. Typography sizes are consistent (`H3` for card titles, `H1` for page titles). Spacing is *not* consistent: `/subject` uses a denser 8 px rhythm, `/weekly` uses 16 px, `/daily` uses 24 px. This is fine if intentional, but a teacher switching pages will feel the shift.

### Conceptual model
Week → Day → Subject is a clean drill-down. The MODIFIED pill on `/weekly` is a recognisable signal that something was moved. From `/subject`, there's no direct path back to "this lesson on /weekly" — add a "View on calendar" link or contextual menu in the lesson rows.

## 10. Deep-link URL pattern (team-confirmed)

The team has confirmed they want every focused view to be a bookmarkable URL. Apply this pattern globally, not just on `/subject`:

| View | Today | Proposed deep-link route |
| ---- | ----- | ------------------------ |
| Weekly view of current week | `/weekly` | `/weekly` ✅ (current week implicit) |
| Weekly view of a specific week | `/weekly` (state only) | `/weekly/2026-w12` or `/weekly?w=2026-w12` |
| Daily view of today | `/daily` | `/daily` |
| Daily view of a specific date | `/daily` (state only) | `/daily/2026-05-18` |
| Daily, focused on a specific lesson | none | `/daily/2026-05-18/<lesson-id>` |
| Subject overview | `/subject` | `/subject/math` |
| Subject + Unit | `/subject` | `/subject/math/unit-3-fractions-on-a-number-line` |
| Subject + Lesson | `/subject` | `/subject/math/unit-3-…/equivalent-fractions-warm-up` |
| Lesson event (canonical) | none | `/lesson/<lesson-event-id>` (URL chat/to-do/resources widgets can link back to) |
| Shared resource viewer (future) | none | `/resource/<resource-id>` |

Next.js dynamic routing handles all of these via `app/<segment>/[slug]/page.tsx`. Each segment should be human-readable (kebab-case slug derived from the entity title), with the entity ID either appended (`equivalent-fractions-warm-up-l8x4`) or kept in a query parameter for unique resolution. Keep the canonical `lessonEvent.id` as the source of truth and treat URL slugs as a presentation layer over it.

When you implement BUG-005 (right-rail Resources scopes to the selected lesson on `/weekly`), the selection state should also push a URL hash like `/weekly#lesson=l8x4` so a teacher can paste that link in chat and the receiver lands on the same weekly view with the same lesson focused.
