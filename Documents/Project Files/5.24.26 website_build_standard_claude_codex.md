# Website Build Standard: Precisely Designed, Faithfully Implemented, and Consistently Adaptable

## Purpose

This document defines the build standard for this website or application.

The goal is not merely to make the site "look nice." The goal is to translate the intended design into a real, durable, responsive product that works across pages, screen sizes, devices, and content states.

The standard is:

> Precisely designed, faithfully implemented, and consistently adaptable.

This means the website should preserve the visual intent of the design while also functioning well in real browser environments, including desktop, laptop, tablet, iPad, and phone screens.

---

## 0. Non-Negotiable Core Requirements

Two requirements take precedence over every other guideline in this document. These are not preferences. They are mandates.

### Requirement A: Tri-Device Support Is Mandatory

Every page, every feature, every component, every modal, every form, every data view, and every interactive element must work on all three device classes:

```txt
Computer (desktop and laptop)
Tablet (iPad portrait, iPad landscape, generic tablet)
Phone (portrait and landscape)
```

There are no exceptions. A feature is not considered built until it works on all three. "Works" means usable, readable, navigable, and visually consistent with the rest of the product.

If a feature genuinely cannot be reproduced on a smaller device, a deliberate, well-designed fallback is required (see Section 9). A page that is broken, squeezed, or unusable on one of these device classes is treated as incomplete, regardless of how polished it looks on the others.

### Requirement B: Phone Defaults to List View

On phone screens, the default layout for any collection, grid, table, calendar, card array, kanban board, or multi-column display is a single-column vertical list view.

This rule applies to:

```txt
Lesson grids
Unit cards
Resource galleries
Calendars
Tables
Dashboards
Search results
Notification feeds
Class rosters
Student lists
Library views
Any other collection of items
```

The list view is not a degraded fallback. It is the intended phone presentation. It must be designed with the same care as the desktop layout.

A phone list view must:

1. Stack items vertically in a single column.
2. Show the most important information first (title, status, primary metadata).
3. Use full-width tap targets so the entire row is clickable where appropriate.
4. Hide or collapse secondary information behind a tap, drawer, or detail view.
5. Maintain consistent row height rhythm or use intentional variable heights.
6. Use clear visual separators (dividers, spacing, or card boundaries).
7. Support quick scanning, not dense data analysis.

Anything more complex than a list view on phone (such as a two-column grid, a horizontal scroller, or a small calendar) must be a deliberate exception, justified by the content, and explicitly designed. Default to the list.

## 1. Precisely Designed

### Meaning

"Precisely designed" means the visual language of the site should not be approximate, random, or invented page by page. The design should be governed by a clear system.

Every repeated pattern should have intentional rules.

This includes:

- Colors
- Spacing
- Typography
- Borders
- Shadows
- Radius
- Layout width
- Sidebar behavior
- Card styles
- Button styles
- List row styles (phone)
- Hover states
- Selected states
- Empty states
- Loading states
- Error states
- Mobile states
- Tablet states
- iPad states

The design should feel like one coherent product, not a collection of unrelated screens.

### Practical Requirements

When implementing a page, identify and preserve the design system behind it.

Ask:

1. What colors are being used, and what do they mean?
2. What spacing rhythm is being used?
3. What is the hierarchy between titles, labels, body text, badges, buttons, and metadata?
4. What elements are meant to feel primary, secondary, muted, active, or completed?
5. What elements should feel clickable?
6. What elements should feel selected or depressed?
7. What elements should be visually quiet?
8. What elements should draw attention?
9. How does this content collapse into a single-column list on phone?

Do not make one-off visual decisions unless there is a clear reason.

### Visual Consistency Rules

Use consistent values for:

```txt
Border radius
Card padding
List row padding (phone)
List row height (phone)
Grid gaps
Button height
Sidebar width
Header height
Font sizes
Font weights
Line heights
Icon sizes
Status colors
Selected states
Hover states
```

A user should be able to move from one page to another and immediately feel that the pages belong to the same application, on any device.

## 2. Faithfully Implemented

### Meaning

"Faithfully implemented" means the code should honor the actual visual and functional intent of the design. Do not simplify the UI so much that it loses the intended feel. Do not introduce new styling that conflicts with the established direction.

The implementation should match the design in spirit, structure, spacing, hierarchy, and interaction.

Faithful implementation does not mean fragile pixel perfection. It means the final website should clearly look and behave like the intended design, while still being technically sound, on every device.

### Practical Requirements

For each page or component, check the following:

1. Does the final page look like the intended design on desktop?
2. Does the final page look like the intended design on tablet?
3. Does the final page present as a clean, well-designed list on phone?
4. Are the colors accurate across devices?
5. Is the spacing accurate?
6. Are the buttons, cards, sidebars, panels, and headers styled consistently?
7. Are selected states visibly clear?
8. Are hover states useful but not distracting (and absent on touch where appropriate)?
9. Are icons sized and aligned properly?
10. Are text labels readable?
11. Is the layout balanced?
12. Does the page still look good with realistic content?

### Do Not

Avoid these common implementation problems:

- Do not replace carefully styled elements with generic default components.
- Do not ignore spacing details.
- Do not let cards stretch unpredictably unless that behavior is intentional.
- Do not let long text break the layout.
- Do not create inconsistent button styles across pages.
- Do not use random colors outside the design palette.
- Do not make mobile layouts as an afterthought.
- Do not ship a phone view that is just a squeezed desktop layout.
- Do not skip the list view on phone in favor of a cramped grid.
- Do not leave empty states, loading states, or error states unstyled.

### Content Reality Check

The design must work with real content, not only ideal sample content, on every device.

Test with:

- Short titles
- Long titles
- Missing descriptions
- Multiple status badges
- Empty sections
- Long unit names
- Long lesson names
- Many resources
- No resources
- Alerts or notifications
- Completed and incomplete items
- Active and inactive items

If the layout breaks when real content is inserted on any device, the implementation is not faithful yet.

## 3. Consistently Adaptable

### Meaning

"Consistently adaptable" means the design responds well to different devices, screen widths, and content conditions without losing its identity.

A website is not a static image. It must adapt.

The desktop version, tablet version, iPad version, and phone version may not be identical, but they must clearly belong to the same design system. The phone list view, in particular, should feel like the same product as the desktop grid, just optimized for the phone form factor.

Adaptability means the site remains:

- Usable
- Readable
- Visually coherent
- Functional
- Navigable
- Touch-friendly
- Stable under different content conditions

### Practical Requirements

Every page must have a clear responsive plan, written or implicit, that covers all three device classes.

For each page, define what happens on:

```txt
Desktop
Laptop
Tablet
iPad portrait
iPad landscape
Phone portrait (list view default)
Phone landscape (list view default)
```

Do not assume that a desktop layout will automatically work on smaller screens. Do not assume that a tablet layout will automatically work on phone.

### Responsive Behavior Questions

For every page, check:

1. Does the sidebar collapse, hide, or become a bottom navigation?
2. Do grids become fewer columns on tablet?
3. Do grids collapse to a single-column list on phone?
4. Do cards stack properly?
5. Does text remain readable?
6. Do buttons remain easy to tap?
7. Do modals fit on small screens?
8. Do tables become scrollable, stacked, or converted to list rows?
9. Do panels still have enough breathing room?
10. Are important actions still visible?
11. Does the page avoid horizontal scrolling unless intentionally designed?

---

## 4. Required Device Checks

Every page must be checked on desktop, tablet, iPad, and phone layouts.

This is not optional. A page that has not been checked on all three device classes is not ready to ship.

### Required Viewport Checks

Test each page at the following approximate widths:

```txt
1440px: desktop
1280px: laptop
1024px: iPad landscape or small laptop
834px: iPad portrait
768px: tablet
430px: large phone
390px: standard phone
360px: small phone
```

### Page Review Requirement

For every page, all four statements must be true:

```txt
The page works on desktop and laptop.
The page works on tablet and iPad (portrait and landscape).
The page works on phone as a single-column list view (or has an intentional, justified alternative).
The page handles realistic content on all of the above.
```

If a page does not meet all four conditions, it is not complete. Either fix it or create an intentional, well-designed fallback. Silent breakage on phone or tablet is treated as a defect.

## 5. Mobile and Tablet Standards

### Phone Standard

On phone screens:

- The default layout for any collection is a single-column vertical list view.
- The layout should not feel like a squeezed desktop page.
- Primary actions should remain easy to access (consider a bottom action bar, sticky header action, or floating action button).
- Text should not be too small (16px minimum for body text in most cases).
- Tap targets should be at least 44 by 44 pixels.
- Cards and list rows should stack clearly with consistent spacing.
- Side panels should become drawers, sheets, accordions, or separate views.
- Tables and dense grids should be converted to list rows or, where appropriate, made scrollable in a controlled way.
- Modals should fit within the viewport, often as full-screen sheets.
- The user should not have to zoom in to use the page.
- Horizontal scrolling should be intentional only (such as a deliberate carousel).

### Phone List View Standard (Mandatory Default)

When a phone screen displays any collection of items, use a list view by default. The list view is the canonical phone presentation, not a degraded version of the desktop layout.

A well-built phone list view follows these rules:

```txt
[ ] Single column, full viewport width (minus consistent horizontal padding).
[ ] Each row contains the title, key status indicator, and one or two pieces of essential metadata.
[ ] Secondary metadata moves into a detail view, drawer, or expandable row.
[ ] Rows have a consistent vertical rhythm or intentional variable height.
[ ] Visual separation between rows is clear (divider line, spacing, or card edge).
[ ] The whole row is tappable when the row represents a single navigable item.
[ ] Status, completion, or selection is shown with a clear visual cue (badge, icon, color accent).
[ ] Long titles wrap or truncate predictably, not unpredictably.
[ ] Empty list states are explicitly designed.
[ ] Loading list states are explicitly designed (skeleton rows or spinner).
[ ] Scrolling is smooth, with no layout shift as content loads.
```

If a designer or developer wants to deviate from list view on phone (for example, a two-column thumbnail grid for images), this must be a deliberate, justified choice. The reason should be obvious from the content (such as visual browsing of images). Otherwise, default to the list.

### Tablet and iPad Standard

On tablet and iPad screens:

- The layout should take advantage of the larger screen.
- Do not automatically fall back to the phone list view if a better tablet layout is possible.
- Sidebars, split panels, and two-column layouts often work well.
- Touch targets still need to be comfortable (44 by 44 pixels minimum).
- Avoid cramped three-column layouts unless the content genuinely fits.
- Check both portrait and landscape orientations.
- iPad portrait is often the trickiest width and deserves its own pass.

Tablet and iPad layouts often need their own treatment. They should not be treated as either "just desktop" or "just phone."

## 6. Page-Level Implementation Checklist

Use this checklist for every page.

### Visual Design

```txt
[ ] Page follows the established color system.
[ ] Page follows the established spacing system.
[ ] Typography hierarchy is clear.
[ ] Cards, panels, buttons, and inputs match the design language.
[ ] List rows on phone match the design language.
[ ] Selected states are clear.
[ ] Hover states are present where useful.
[ ] Empty states are styled (for both grid and list views).
[ ] Loading states are styled (for both grid and list views).
[ ] Error states are styled.
[ ] Notification states are styled.
```

### Layout

```txt
[ ] Layout works at desktop width.
[ ] Layout works at laptop width.
[ ] Layout works at tablet width.
[ ] Layout works on iPad portrait.
[ ] Layout works on iPad landscape.
[ ] Layout collapses to a single-column list view on phone portrait.
[ ] Layout works on phone landscape.
[ ] No accidental horizontal scrolling.
[ ] Important actions remain visible on all devices.
[ ] Content does not overflow awkwardly.
```

### Content

```txt
[ ] Long titles do not break the layout on any device.
[ ] Missing content does not leave awkward gaps.
[ ] Empty sections have useful empty states.
[ ] Dense content remains readable.
[ ] Realistic data has been tested on desktop, tablet, and phone.
[ ] Truncation is intentional.
[ ] Wrapping is intentional.
[ ] Scrolling areas are intentional.
```

### Interaction

```txt
[ ] Buttons are clearly clickable.
[ ] Touch targets are at least 44 by 44 pixels.
[ ] List rows on phone are fully tappable where appropriate.
[ ] Navigation is clear on all devices.
[ ] Modals, drawers, popovers, and dropdowns work on phone.
[ ] Keyboard navigation is reasonable.
[ ] Focus states are visible.
[ ] Forms are usable on phone.
[ ] Required fields and errors are clear.
```

### Tri-Device Sign-Off

```txt
[ ] Verified on desktop (1440px or 1280px).
[ ] Verified on tablet or iPad (834px and 1024px).
[ ] Verified on phone (390px, in list view).
```

A page cannot be marked complete until all three sign-off boxes are checked.

## 7. Component-Level Rules

Reusable components should be built once and used consistently.

Examples:

```txt
PrimaryButton
SecondaryButton
IconButton
Sidebar
TopNav
BottomNav (phone)
PageHeader
Card
ListRow (phone default)
LessonCard
LessonListRow
UnitBadge
StatusBadge
ProgressBar
EmptyState
ResourcePreview
NotificationBadge
Modal
Sheet (phone modal)
Drawer
Popover
Tabs
SegmentedControl
```

Every collection component should ship with both a grid/card presentation and a corresponding list row presentation, so that the same data can be rendered as a grid on desktop and tablet, and as a list on phone, without duplicating logic.

Each reusable component should have consistent states:

```txt
Default
Hover
Active
Selected
Disabled
Loading
Error
Completed
Empty
```

Do not recreate the same component differently on different pages.

If a design change is needed, update the component system rather than creating a local exception.

## 8. Design Fidelity Versus Adaptation

Some parts of the design should remain highly consistent across devices:

```txt
Color palette
Typography style
Button styling
Card and list row radius
Icon style
Status colors
Selected state logic
General visual tone
```

Other parts must adapt:

```txt
Number of columns (collapses to 1 on phone)
Sidebar behavior (drawer or bottom nav on phone)
Panel placement
Table display (becomes list rows on phone)
Calendar density (becomes agenda list on phone)
Navigation position
Modal size (full-screen sheet on phone)
Amount of visible metadata (less on phone)
```

The goal is not to force the desktop layout onto every device.

The goal is to preserve the product identity while making the layout work well in each context. On phone, that almost always means a clean, well-designed list view.

## 9. Responsive Fallbacks

If a complex feature does not fit naturally on phone, create a deliberate mobile fallback built on the list view pattern.

Examples:

### Desktop

```txt
Full calendar grid
Sidebar visible
Resource panel visible
Multi-column lesson layout
Kanban with multiple columns
Data table with many columns
```

### Tablet or iPad

```txt
Reduced column count
Collapsible resource panel
Sticky top controls
Scrollable lesson area
Two-column kanban or stacked columns
Data table with fewer visible columns
```

### Phone (list view default)

```txt
Calendar becomes agenda list (grouped by day)
Kanban becomes a status-grouped list (sections by column)
Data table becomes a list of rows, one record per row, with key fields stacked
Multi-column lesson layout becomes a single list of lessons
Resource panel becomes a sheet opened from the lesson detail
Drawer navigation or bottom nav
Bottom action bar for primary actions
Tap to open full details
```

A simplified phone layout based on a list view is the standard, expected approach. It is not a compromise. It is the right design for the device.

A broken, squeezed desktop layout on phone is not acceptable.

---

## 10. Final Review Standard

Before considering a page complete, review it with these questions:

> Does this page preserve the intended design while working responsibly across desktop, tablet, and phone?
>
> On phone, does the page present as a clean, deliberate list view (or an intentional, justified alternative)?
>
> Has the page been tested with realistic content on all three device classes?

A page is complete only when the answer to all three is yes.

The final site should be:

```txt
Precisely designed:
It follows a clear, intentional visual system.

Faithfully implemented:
It looks and behaves like the intended design on every device, not a generic approximation.

Consistently adaptable:
It works across desktop, tablet, iPad, and phone without losing usability or visual identity, with phone defaulting to a well-designed list view.
```

---

## 11. Instruction for Claude or Codex

When modifying or generating code for this project:

1. Treat tri-device support (desktop, tablet, phone) as a hard requirement, not a nice-to-have. Every change must work on all three.
2. On phone, default any collection, grid, table, or calendar to a single-column list view. Deviation from the list view on phone requires a justified reason.
3. Preserve the existing design direction unless explicitly instructed otherwise.
4. Look for the design system behind the page before changing components.
5. Reuse existing components and styling patterns where possible. If a collection needs both a grid and a list presentation, build both as part of the same component family.
6. Avoid introducing visual inconsistencies.
7. Check desktop (1440px or 1280px), tablet/iPad (834px and 1024px), and phone (390px) layouts for every change. State explicitly which viewports were verified.
8. Do not leave phone or tablet views broken. If a feature cannot fit on phone, build a list-view-based fallback rather than shipping a squeezed layout.
9. Test with realistic content, including long text and empty states, on every device.
10. Ensure selected, hover, active, disabled, loading, error, and empty states are handled, including in the phone list view.
11. Prioritize a polished, coherent, durable product over a fragile pixel-perfect copy.
12. When in doubt about a phone layout, choose the list view. It is almost always the right answer.

The goal is not just to build pages.

The goal is to build a product that feels intentionally designed, carefully implemented, and dependable on every device a real user will actually hold in their hands.
