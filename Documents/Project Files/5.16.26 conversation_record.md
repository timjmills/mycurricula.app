# Grade 5 Curriculum Planner — Conversation Record

> Working notes from the design conversation between the teacher (project lead) and Claude. This document is a faithful record of what was asked, what was answered, what was decided, and what remains open. It is the source material for the formal planning document.

---

## 1. The Original Problem

The Grade 5 teaching team currently runs their yearly planning across **five separate documents**, which creates duplication, drift, and friction:

1. **Padlets** — one per unit per subject, holding resources displayed to students each week.
2. **Week-by-week Google Doc** — a day-by-day grid; each day lists every subject taught with the lesson, resource links, etc.
3. **Week-by-week Focus Doc** — general focus statements for math, literacy (grammar, reading skills, writing), Explorers, UFLI, Social Emotional Learning, and Daily Notes.
4. **Standards Document** — Common Core standards per subject for the full year.
5. **Personal Teacher Copies** — each teacher keeps their own copy of the week-by-week plan to mark edits, pacing differences, or substitutions.

The goal is to consolidate all five into **one web application** that the team can use as their single source of truth.

---

## 2. Feature Wishlist (Initial Ask)

The teacher's original feature list, captured verbatim from the conversation:

- Week-by-week view of the standard curriculum, searchable, with drag-and-drop to move things around or reassign to different weeks.
- Daily view that also stores resources (documents, PowerPoints, Google Slides, pictures, etc.).
- Subject view that filters all resources for a single subject.
- Ability to add or remove subjects.
- Subject view should work weekly, unit-based, monthly, or full year.
- Standards mapped to each assignment, viewable on mouseover.
- Function somewhat like Padlet — but with filtering by time period, unit, or subject.
- Print / download / annotate a resource and display it from the app.
- Print plans daily, weekly, or by unit, including standards.
- Unit summary view for each unit.
- Teachers can copy a daily / weekly / unit plan, edit it, and check items off as they complete them.

## 3. Key Clarifications (Round 1 Q&A)

### Editing model
- **Master plan** holds all official lessons and resources.
- Teachers default to editing their own personal copy.
- Editing the master requires entering an explicit "Editing mode."
- Personal alterations have two flavors:
  - Small **personal notes** attached to a master lesson, AND
  - A full **personal copy** that the teacher can freely edit.

### Hosting
- Currently using **Squarespace** for the public website (with MathQuest app already deployed via GitHub).
- Open to **Cloudflare** for app hosting.
- Final stack decision: **GitHub (code) + Cloudflare Pages (frontend) + Supabase (backend/DB/auth) + Squarespace (public-facing site, linking out to the planner)**.

### Resources
- **Phase 1:** Links only, with auto-hyperlinking.
- **Phase 2:** Actual file storage (Padlet-style), once a free-storage solution is in place.
- Priority file types: PDFs, images, Word docs, YouTube video links, hyperlinks to websites / Slides / Google Docs.

### Annotation
- PDF annotation tool AND image annotation tool, with a whiteboard mode.
- Confirmed as a Phase 3 feature (heaviest single piece).

### Calendar structure
- **40 weeks** in the school year.
- Each week of each subject is tied to a **unit**.
- Unit length changes each year — must be flexible.

### Flexibility requirements
- Subjects and units must be addable/removable/editable any time.
- Unit duration must be adjustable.
- Daily assignments must move freely between days, weeks, and units.
- Items must be quickly reassignable within and between units.

### Copy-and-edit behavior
- When a teacher copies a unit, and the master is later updated, the teacher should receive a **notification** listing what was added/removed, with the option to accept or reject the change.

### Standards
- Many assignments already carry CCSS tags.
- **Automatic tagging** is desirable as a future feature.
- Assignments can be tagged with one or more standards, or remain untagged.
- Vertical alignment (Grade 4 / Grade 6 view) is nice-to-have, not Phase 1.

### Default view
- Default view is **per-teacher preference**, saved.
- All views must be easy to switch between.
- **Color** must be used aggressively for instant subject/unit recognition.
- Most-important views: **40-week weekly view** (with resource links) and **subject views**.
- Resource integration (file hosting) can come later.

## 4. Hosting Discussion

The teacher initially asked whether GitHub + Squarespace (the MathQuest setup) could host this project too. Claude explained:

- **GitHub Pages** serves static files only. MathQuest works because its state is per-user/local. This planner needs **shared state** across teachers — a database somewhere on the internet.
- Static hosting cannot do shared data, authentication, file uploads, or master-change notifications.
- The right split:
  - **GitHub** stores the code.
  - **Cloudflare Pages** serves the frontend (free, fast, integrates with GitHub).
  - **Supabase** hosts the database, authentication, and (later) file storage. Free tier covers the school's scale comfortably.
  - **Squarespace** keeps the existing public website, linking out to the planner via subdomain or external link.

The teacher confirmed this stack.

## 6. Daily Notes

Per-day reminder banner, NOT a subject row.

- **Three priority levels:**
  - 🔴 **Urgent** (pulsing red) — e.g., "Fire drill at 10:15"
  - 🟡 **Important** (solid yellow) — e.g., "Picture day, kids in uniform"
  - 🔵 **FYI** (subtle blue) — e.g., "Sarah's birthday treat at lunch"
- **Two scopes:**
  - **Shared** — team-visible (stacked on top)
  - **Personal** — only visible to author (stacked underneath)
- Shared notes carry a team icon; personal carry a "just you" indicator.

## 7. Completion Tracking

Surfaced as a high-priority feature mid-conversation: the catch-up filter.

### Status model
Each assignment has a per-teacher status (not global):
- ⬜ Not done (default)
- ✅ Done
- ⏭️ Skipped (intentional)
- 🔄 Carried over (missed, rescheduled)
- ⚠️ Partial

### Catch-up filter
- Appears in every view (daily, weekly, monthly, subject, unit).
- Toggle: "Show only uncovered."
- Weekly grid shows a "🔥 N items not covered" badge.
- Unit dashboard shows % completion per subject and a "view what's missing" action.

### Privacy
- Completion data is **per-teacher**, and the privacy setting is **per-teacher's preference**: each teacher can elect to keep theirs private OR share with the team.

### Auto-carryover (decided)
- Do NOT auto-flip stale ⬜ items to 🔄.
- Catch-up filter surfaces them; teacher can bulk-action (mark all of last week as done / skip / reschedule).

## 8. Schedule View (added)

A sixth view type, distinct from the others.

- Follows the teacher's actual **daily timetable** (time slots), not the curriculum grid.
- Auto-slots lessons from the week's daily/weekly plan into their scheduled time blocks.
- Also displays **non-academic blocks**: lunch, recess, PE, Arabic. These accept **notes only**, not curriculum content.
- Default source: **personal copy if it exists**, falling back to master if not.
- Edits made in Schedule View write to **personal only** (auto-forks from master if no personal copy exists yet).
- Master is NEVER edited from Schedule View.

## 10. Master vs. Personal — Final Locked Rules

After several rounds of refinement, the rule set converged to a clean, predictable model:

### Rule 1: Default view behavior
- Every view (daily, weekly, monthly, unit, subject, schedule) shows **personal copy where it exists, master where it doesn't**.
- A teacher with a personal copy basically always sees their version unless they explicitly switch to master view.

### Rule 2: Write target
- **Master mode views** (entered explicitly via master-mode toggle + "Now editing master" button) — edits write to master.
- **All other views, all other edits, everywhere else in the app** — edits write to personal.
- If no personal copy exists when an edit is made, the system **auto-creates a personal copy from the master** and applies the edit. (Lazy forking — no upfront "make a copy" step required.)
- Master can ONLY be edited from master-mode views, never from a personal-mode view.

### Rule 3: Side-by-side toggle
- Personal version lives toggleable side-by-side with master equivalent in the same view.

### Rule 4: Master-change notifications
- When master is changed, all existing personal copies that diverge from the new master receive a notification.
- Notification lists what was added/removed; teacher accepts or rejects the change per-item.

### Rule 5: Year reset / reuse
- Plans must be archivable and rollable forward to next year with minimum manual work.
- (Detailed mechanism TBD in planning doc — flagged as open question.)

### Rule 6: Visual indicator
- Lessons displayed in a default view must visually indicate whether they are "mine" (personal copy) or "master" (unedited from master) — a subtle dot, tint, or icon so the teacher can tell at a glance.

## 12. Doability Discussion (Quick Summary)

The teacher asked how doable the project is in Claude Design (artifacts) and Claude Code.

- **Claude Design / artifacts:** Not the production app. Artifacts are great for **prototyping individual screens** (weekly grid, standards picker, unit dashboard, print preview) to nail UI decisions before committing to code. No real auth, no shared backend, 5MB localStorage limit.
- **Claude Code:** Yes, very doable on the chosen stack. Estimates given:
  - MVP (replacing 2 of 5 documents) — 3–5 weeks.
  - Full feature set (including annotation, copy-and-edit, multi-teacher version control, print templates) — 3–6 months.
- Hardest pieces: annotation, PowerPoint preview, smart fork/merge semantics. Scoping annotation to PDFs-only and forks to "frozen snapshots" makes the project dramatically more tractable. The full master→personal sync model (Rule 4) is more sophisticated and was retained.

## 14. To-Do Lists (added)

A late addition to scope: a to-do list feature for tasks that aren't lessons (photocopies, parent emails, supply orders, bulletin-board updates, conference scheduling, etc.).

### Structure
- **One running personal list per teacher** — only visible to that teacher.
- **One running team list** — shared across the grade-level team, visible to all.

### Date model
Items have an optional date attribute. The UI bucketizes them via filter:
- **Date-specific** — pinned to an actual date
- **Today** — items due today
- **This week** — items due in the current week
- **No date** — items with no date attached

### Tags and color
- Teacher-defined **tags** with colors (e.g., "copies" = blue, "emails" = green, "supplies" = amber, "events" = purple).
- Lists are **filterable by tag** and by date bucket.
- Tags on the team list are team-defined; tags on personal lists are per-teacher.

### Reasonable defaults applied (not explicitly confirmed but flagged for review)
- **Shared list checkboxes:** anyone can mark a team item complete; UI shows "✅ done by Sarah" attribution.
- **Recurring tasks:** out of scope for Phase 1.
- **Location in UI:** primary access via a right-side slide-out panel from any view; also a full-screen To-Do view for power management.
- **Linking:** a to-do item can optionally link to a lesson, unit, or resource for context ("prep copies for Math Unit 3 Lesson 5").

## 15. Two-Pane Daily Layout (added)

A late refinement applied to daily-style views (the Daily view and the Schedule view). The Personal/Master distinction is handled by the **top-bar toggle**, which works in all views — there are no separate "Daily Master" and "Daily Personal" views.

### Layout
- **Left pane:** clickable list of the day's subjects (Daily view) or time blocks (Schedule view). Acts as a table-of-contents for the day. Each item shows title + completion checkbox.
- **Right pane:** lesson detail for whatever is selected on the left — directions, notes, resources, standards, completion controls.

### Specifics
- **Applies to:** Daily view and Schedule view.
- **Highlighting in Schedule view:** the current/next time block is highlighted, but the highlight is **per-teacher** — based on the individual teacher's own schedule, not a team-wide clock indicator.
- **Right pane has one job at a time** (see §17 Design Refinement). It is either showing lesson detail, a notes editor (for non-academic blocks), or — when nothing is selected — a read-only Today dashboard.
- **Today dashboard (right pane empty state):** a read-only summary showing today's daily notes (shared on top, personal underneath), today's to-dos (slice of items dated today or in the "Today" bucket, both personal and team), daily completion summary, and quick-add inputs for notes and to-dos. Full to-do management happens in the slide-out panel.
- **Default selection on open** (assumed; flag if different is preferred):
  - Schedule view → auto-select the time block that matches the current time of day.
  - Daily view → auto-select the first not-yet-done subject.
- **Mobile/narrow screens:** the two panes stack — left becomes a header dropdown, right fills the screen.

### Why this matters
- In-the-moment usability: scan left, tap, right fills in.
- Classroom projection friendly: right pane can be displayed without the navigation clutter.
- Today-dashboard default state turns the daily view into a useful landing page rather than an empty canvas.

## 16. Multi-Grade Scalability (added)

The app should be designed so additional grade levels can be activated later without re-architecture. Decision: **Option A — multi-grade single-school**, with Grade 5 the only active grade at launch.

### What this means
- Multiple grade levels (K through 6 or higher) can coexist in the same database.
- Each grade level has its own teaching team, master plan, units, subjects, standards, to-do lists, daily notes, and lead teachers.
- Teachers can be assigned to one or more grade levels and switch between them via a top-bar grade selector.
- Lead-teacher role is granted **per grade** (a teacher could be lead for Grade 5 but not for Grade 4).

### Standards implication (confirmed)
- Standards are **grade-scoped**. A Grade 5 teacher's standards picker shows CCSS.5.* codes; a Grade 3 teacher's shows CCSS.3.*.
- A teacher viewing Grade 5 will not see Grade 3 standards unless they switch grades.

### Not included
- Multi-school / multi-tenant support (would require true tenant isolation, billing, etc.).
- Cross-grade visibility (Grade 5 viewing Grade 4's plans) — out of scope.
- Vertical CCSS alignment is still Phase 3+ and grade-aware when added.

### Cost
- Baking in multi-grade now adds ~2–3 days to Phase 1.
- Retrofitting later would cost weeks. Worth doing upfront.

## 17. Design Refinement — Consolidation (added)

Mid-design review pass. Three issues surfaced and were resolved:

### Issue 1: Three daily-style views was one too many
Original conversation treated "Daily Master" and "Daily Personal" as separate views, alongside "Schedule." But Daily Master and Daily Personal are the *same view* showing the *same information shape* — they differ only in which version of the data they pull. That's a mode, not a view. And the Personal/Master toggle in the top bar already does this work for every other view (Weekly, Unit, Subject, Year).

**Resolution:** One Daily view, one Schedule view. The Personal/Master top-bar toggle handles the data-version switch in both. No "Daily Master" view, no "Daily Personal" view as separate UI surfaces.

### Issue 2: Right pane was being asked to do too many jobs
Originally spec'd to show (a) lesson detail when something is selected, AND (b) a daily dashboard when nothing is selected, AND now potentially to-do management with week/month/team/personal/tag filtering. That's three different surfaces fighting for the same real estate, and creates a confusing mode-y panel where teachers don't know what they'll see.

**Resolution:** The right pane has **one job at a time**:
- When a lesson is selected → lesson detail only
- When a non-academic block is selected (Schedule view) → notes editor only
- When nothing is selected → read-only Today dashboard (notes + today's to-dos + completion summary + quick-add)

No mixed-purpose pane. No filtering controls in the right pane.

### Issue 3: To-dos had two homes
The slide-out panel and the right pane both wanted to be to-do surfaces. This duplicates the UI and confuses teachers about where to manage tasks.

**Resolution:** Clear division of labor:
- **Right pane Today dashboard:** read-only slice of today's to-dos. Shows what's on the docket today. Quick-add only.
- **To-do slide-out panel (top-bar trigger):** the full to-do management surface. Mine / Team tabs, scope filter chips (Today / This week / This month / No date / All), tag filter, full list with sorting, bulk actions, edit/delete, link-to-lesson, etc.

To glance at today's tasks → look at the dashboard. To manage the to-do system → open the slide-out.

### New principle locked in
**Each UI surface has one clear job.** Modes and toggles change the *content* of a surface, not its purpose. Filtering and management live in dedicated surfaces (slide-outs, settings), not in primary view panes.

## 18. Lesson Card Interaction Patterns (added)

Two specific interaction behaviors clarified after the consolidation pass:

### View-dependent expansion
- **Weekly view:** clicking a lesson card **expands it inline within its grid cell** (directions, notes, resources, standards visible in place). Adjacent cards in the column reflow. Click again, click another card, or Esc to collapse. Only one card expanded at a time.
- **Daily view + Schedule view:** clicking a lesson card opens detail in the **right pane** (the two-pane layout established earlier). No inline expansion in daily-shaped views.
- **Subject view + Unit view:** inline expansion within the list (same pattern as weekly).
- **Why the split:** weekly is grid-oriented and benefits from staying-in-context inline expansion; daily is selection-oriented and benefits from a dedicated detail surface. Different ergonomics for different use cases.

### Right-click context menu
- Available on every lesson card across all views.
- Menu items: Move to (week / day / unit) · Duplicate · Copy to my personal (or Reset to master) · Mark status (5 statuses) · Add to to-do list · Delete (master mode only, confirm) · See standards · Print this lesson.
- For touch and keyboard users, a "⋯" hover/focus affordance opens the same menu — the right-click and the affordance are equivalent.
- The "Move to" submenu is the fastest way to relocate a lesson between weeks/days/units without dragging — important because drag-and-drop on a 40-week grid can require a lot of scrolling.

## 19. Lesson Card Preview and Sticky Expansion (added)

Two refinements to lesson card behavior:

### Preview paragraph in collapsed state
The collapsed (default) state should show a **meaningful 2–3 line preview paragraph**, not just the title. The preview is auto-drawn from the first portion of the lesson's directions field, truncated with ellipsis at ~140–200 characters.

**Why this matters:** a title-only card forces the teacher to click every lesson to figure out what's in it — too many clicks on a 40-week grid. A preview paragraph lets teachers scan the week visually without expanding anything.

### Sticky expansion (Weekly, Subject, Unit views)
Once a card is expanded, it **stays open until explicitly closed**. Opening one card does NOT close others. Multiple cards can be expanded simultaneously across the grid, and expansion state persists **across the teacher's devices** (laptop → iPad → laptop) via a server-side `TeacherUIState` record that syncs in realtime, with writes debounced ~500ms.

**Why this matters:**
- Teachers often need to compare lessons across days (e.g., Monday math + Wednesday writing).
- Auto-closing other cards when a new one opens (the usual accordion pattern) is the wrong default here — it would force teachers to constantly re-open the card they were just looking at.
- Teachers switch between devices throughout the day — plan at home on a laptop, reference in the classroom on an iPad. Filters, expanded cards, and the active view should follow them across devices, not reset on every device switch.
- Closing is explicit: collapse caret, click header, or Esc.

### Disclosure tiers (full picture)
The lesson card now has three clear content tiers:
- **Collapsed (default):** title + 2–3 line preview paragraph + checkbox + standards badge + resource icons.
- **Expanded (after click, sticky):** full directions, all resources, all standards, accessible notes.
- **Deepest:** hover-revealed notes (click to open and expand further).

### Implication for content authoring
The directions field's **first sentence or two should be a self-contained summary**. Teachers writing lessons should think of the opening as the preview, not as a lead-in. This is a content guideline, not a UI constraint — the UI just shows what's there.

### Cross-device UI state (added)
A new server-side entity, `TeacherUIState`, holds one row per teacher and stores ephemeral UI state: expanded card IDs, active filters per view, last-active view, last-active grade, panel open/closed states. This syncs via Supabase realtime so a teacher's UI state follows them across devices (laptop ↔ iPad ↔ phone). Writes are debounced ~500ms after the last interaction to avoid spamming the database on rapid clicks.

This is the right design because teachers routinely shift devices throughout the day. localStorage would have lost state every time they switched device, which is the wrong behavior for this audience.

## 20. Curriculum Exports for School and Governing-Authority Submission (added)

Major requirement raised late in the design pass: the curriculum must be exportable for school and governing-authority (MOEHE) submission. These documents are required, not optional, so exports are a Phase 1 concern.

### Confirmed
- **Formats:** PDF and Excel.
- **Scopes:** daily / weekly / unit / subject-for-the-year.
- **Required fields:** lesson/unit names, learning objectives, standards focused on, lesson content with resource links.
- **No bilingual requirement** — English only.
- **No signature blocks needed.**
- **Both Live AND Snapshot modes wanted:** Live = generate on demand, Snapshot = save to a Reports library in-app for later re-download.

### Defaults applied (flagged as open questions for review)
- **Any teacher can export** in Phase 1. Could later restrict formal submission exports to lead role only.
- **Excel structure:** one row per lesson, all metadata as columns (audit-friendly, filterable in Excel).
- **No school template required** at this stage — Phase 1 ships with one clean default PDF layout. If MOEHE/school later requires a specific branded template, that's Phase 2 work (likely with Puppeteer).
- **Teacher notes excluded from exports by default** (private by nature; teacher can opt in per export).
- **Reports library retention** — manual delete only, leads only. Revisit if storage grows.

### Phase implications
- **Phase 1** ships PDF export via `@react-pdf/renderer` (client-side, free, consistent across browsers, archival quality). One default template. Live + Snapshot modes. Reports library.
- **Phase 2** ships Excel export (SheetJS) and any custom branded PDF templates if needed.

### New data model additions
- `learning_objectives` field on `MasterCoreLessonEvent` — new field, separate from directions and notes.
- `SavedExport` entity — one row per snapshot export, holds metadata + Supabase Storage file URL.

### Why this matters (technical note)
The existing "Print preview" feature in §5.9 uses `window.print()`, which is fine for ad-hoc printing (substitute folders, scratch use) but inconsistent across browsers and not archival-quality. For documents that get submitted to MOEHE, you need a real PDF library with a controlled template. These are two different pipelines that should not be conflated:
- **Ad-hoc print** = `window.print()`, used from the Print Preview pages.
- **Archival export** = `@react-pdf/renderer` (Phase 1) → optionally Puppeteer in a Cloudflare Worker (Phase 2), used from the Export Center.

## 21. Admin Section — Plan Now, Build Later (added)

Future requirement raised: at some point an admin (principal, IT coordinator, curriculum coordinator) should have a section in the app to see what's happening across grades and teachers. Decision: **plan the data and roles in Phase 1; build the UI in Phase 3+.**

### Two-tier admin model
- **`SchoolAdmin`** — school-wide, read-only by default across all grades; can manage `GradeLevel`, `TeacherGradeAssignment`, `SchoolYear`. May or may not be a teacher (nullable `teacher_id`).
- **`grade_admin`** — new value in the existing `TeacherGradeAssignment.role` enum. Superset of `lead`: lead privileges + audit-log read + assignment management for that grade.

### What ships in Phase 1 (no UI, just infrastructure)
- `SchoolAdmin` table.
- `grade_admin` enum value.
- `AuditLog` table — append-only, written from every mutation via a security-definer RPC.
- `auditLog()` helper in app code so every mutation already logs.
- `lib/admin/queries.ts` convention established (even if empty) for any future cross-grade or admin-style queries.
- RLS policies that recognize all three role tiers.

### What does NOT ship in Phase 1
- Any admin UI at all. No menu item, no screens, no roster, no activity timeline.
- No admin role is granted at launch. The plumbing exists but is dormant.

### Why this matters
- **Retroactive audit data is impossible to fabricate.** If you start logging in Phase 3, the admin only sees data from then forward. By starting in Phase 1, the admin gets months of history the moment the UI ships.
- **Retrofitting roles is expensive.** Every RLS policy and permission check would have to be revisited if `admin` were added later. Doing it once, upfront, costs hours instead of weeks.
- **No design or product cost.** No screens to design, no flows to spec — just a few database tables and a helper function.

### Phase 3+ admin UI (placeholder)
When this phase begins, the admin section will likely include:
- Activity timeline (filtered audit-log view).
- Roster + role management.
- Grade-level management.
- School-year management.
- System metrics dashboard.
- Anonymized aggregate completion view.
- Export audit.

These are notional and will be designed when the phase begins, against months of accumulated audit data.

## 22. In-App Communication — Comments and Day Shoutbox (added)

Late requirement raised: chat capability so the team can communicate throughout the day. After discussion, decided NOT to build general-purpose chat (WhatsApp already serves that need) and instead build two context-anchored alternatives that do what chat can't.

### Two complementary surfaces

**Option A — Anchored Comments (the bigger feature):**
- Threaded discussion attached to specific lessons, units, or resources.
- One level of reply nesting (top-level + replies, no deeper).
- Visible to the entire grade team. No DMs.
- Author can edit and delete own; `lead` and `grade_admin` can delete any.
- Soft delete preserves thread structure ("[Comment removed]").
- Inline display: appears at the bottom of lesson detail (Weekly expanded card, Daily right pane, Unit page).

**Option B — Day Shoutbox (the lighter feature):**
- A single flat team thread per date, displayed in the Daily view below the Daily Notes banner.
- No threading, just a posts list.
- Auto-hides from inline view after 7 days. Still findable in the Comment Browser.

### Comment Browser (the third piece)
A slide-out (with full-screen option) that mirrors the to-do panel's UX. Solves: "find all comments by author X", "what's been discussed on this unit", "what comments do I have unread".

Filters:
- Scope chips: Today / Past 7 days / This week / This month / All
- Anchor-type chips: All / Lesson / Unit / Resource / Day shoutbox
- Subject, Unit, Author multi-select
- Has-replies-from-me toggle
- Unread-only toggle
- Quick filters: Unread, Replies-to-me, Comments I started, Today's lesson comments

Result rows include the anchor label ("📘 Math · Unit 3 · Lesson 5 · Mon Wk 12") and click-to-jump to the parent view with the thread scrolled in.

### Why anchored vs. flat-stream chat
General chat scrolls past and loses context. Anchored comments stick to the lesson, unit, or resource they're about. A teacher returning from leave can read the thread on Monday's lesson and instantly know "Sarah extended this, Lisa skipped it." That's something WhatsApp can't do.

Comments also become part of the curriculum knowledge base, not ephemeral noise. Phase 2 will include an opt-in checkbox in the Export Center to include comment threads in PDF exports.

### Data model
- `Comment` (id, author_id, grade_level_id, anchor_type, anchor_id, parent_comment_id, body, is_edited, deleted_at, timestamps)
- `CommentRead` (per-teacher per-comment read state)
- New audit-log actions: `comment_posted`, `comment_edited`, `comment_deleted`
- RLS: read scoped to grade; write scoped to author + role-based delete

### Phasing
- **Phase 1:** core comments, day shoutbox, Comment Browser, unread badge (no real-time push, badge updates on app load and view-switch).
- **Phase 2:** real-time badge updates via Supabase realtime; opt-in daily email digest; "Include comments" checkbox in Export Center.
- **Phase 3:** comment activity feed visible inside the Admin section.

### Defaults baked in (and accepted)
1. Threading depth = 1 (no reply-to-reply chains).
2. Day shoutbox auto-archives at 7 days (still in Comment Browser).
3. Phase 1 = no real-time push, just pull-on-open.
4. Author-or-lead delete (not anyone-can-delete).
5. All comments grade-team-visible; no DMs.

### Why this matters strategically
Adding comments shifts the planner's mental model. The app is structured around *time* (weeks, days, units); comments add a second axis: *conversation history*. A new teacher joining mid-year, or one returning from leave, can use the Comment Browser to learn what's been discussed about Math Unit 3 across six weeks. That's a genuine knowledge-management capability the rest of the app doesn't offer.

## 23. Multi-Framework Standards, Custom Events, and Admin Reporting Foundation (added)

Three substantial additions raised in the same round, all related to making the planner usable beyond the original Grade 5 launch context.

### Multi-framework standards
- The school uses CCSS *and* EE standards simultaneously.
- Future schools may use IB, NZ Curriculum, British curriculum, Cambridge, or custom internal frameworks.
- A single lesson, extra lesson event, or day event can be tagged with standards from multiple frameworks at once.

**Schema additions:**
- `StandardsFramework` — id, name, short_code, jurisdiction, provenance (`catalog | school_uploaded`), owner_school_id (nullable), color/icon, max_depth, supports_languages, is_active.
- `GradeFrameworkAssignment` — junction linking grades to frameworks they use.
- `Standard` — rewritten to be framework-aware (references `framework_id`), hierarchy-capable (`parent_standard_id`), and translation-ready (`description_translations`).

**CSV upload:** school admins upload custom frameworks via CSV (`framework_short_code, code, description, parent_code, grade_level, language`). Basic in Phase 1; polish in Phase 2; in-app editor in Phase 3+.

**Catalog vs. school-uploaded:**
- Catalog frameworks vetted by the planner team; visible to all schools. CCSS Grade 5 ships in the catalog for launch.
- School-uploaded frameworks private to the owning school. EE standards will be uploaded by the school admin during onboarding.

### Three distinct "thing on a day" types — unified authoring
Decided NOT to merge Lesson and DayEvent into one entity. Instead, keep three distinct types with **symmetric authoring shape** (title, learning objective, standards, resources — only title required) but **different downstream behavior**:

1. **Core Lesson Event** — master/personal forking, unit/week binding, completion/catch-up tracking. The thing the planner is built around. Implemented by the `CoreLessonEvent` / `MasterCoreLessonEvent` / `PersonalCoreLessonEventCopy` entities.
2. **Extra lesson event** (new entity: `ExtraLessonEvent`) — one-off teaching activities outside the master curriculum (closing circle, enrichment moment). Same fields as a lesson, but no unit/week, no forking. Personal or team scope.
3. **Day event** — non-curriculum entries (assembly, drill, guest speaker, celebration). Same fields as a lesson, optional time, completion checkbox.

**"+ Add to day" chooser:** universal authoring entry point. From Daily/Schedule/Weekly views, a small picker asks which of the three types to add, then opens a form with the symmetric fields. The user doesn't need to understand the schema distinction — they just pick what they're adding.

**Phase 2+ extras:**
- `ExtraLessonTemplate` — teachers save reusable templates ("my Closing Circle template") and drop them into any future day with one click.
- Type conversion — "Convert this Day Event to an Extra Lesson Event" action, preserving data.

### Naming convention locked in (Option A — full schema rename)
The three types are user-facing AND in the schema as **Core Lesson Event / Extra Lesson Event / Day Event** — a symmetric set, top to bottom:
- Core Lesson Event = `CoreLessonEvent` / `MasterCoreLessonEvent` / `PersonalCoreLessonEventCopy` entities.
- Extra Lesson Event = `ExtraLessonEvent` entity.
- Day Event = `DayEvent` entity.

Reorder records are named `EventDayOrderOverride` (since they now order all three event types in a day, not just core lessons). Foreign keys are `core_lesson_event_id`, `extra_lesson_event_id`, `day_event_id` accordingly.

One name everywhere — UI labels, database tables, foreign keys, audit log entries, export PDFs — so there's no translation cost between code, design, and conversations with the team. Slightly more lines to write at build time; zero ambiguity forever.

### Admin reporting foundation (data only in Phase 1; UI in Phase 3+)
Locked in the strategic point: this app should help administrators understand teaching and learning, not just help teachers plan. Reports baked into the spec but explicitly NOT built in Phase 1:

- Standards coverage dashboard (matrix + heatmap)
- Coverage gaps report
- Distribution over time (which weeks each standard appears in)
- Repetition / depth view
- Cross-subject integration (especially for SEL / 21st-century-skills frameworks)
- Teacher variance (master vs. each teacher's personal copies)
- Comment intensity heatmap
- Year-over-year comparison
- Unit deep-read

**Locked design principles for admin reports:**
- **Aggregate by default; per-teacher drill-in requires explicit permission.** Data shouldn't feel like surveillance.
- **Reports are read-only.** Admins observe; they don't edit teacher data from these screens.
- **Reports run against pre-computed `CoverageSnapshot` data**, not live joins. Fast loads, historical accuracy.

**Phase 1 data infrastructure (no UI):**
- `CoverageSnapshot` table.
- Nightly scheduled job computing coverage stats per (school_year, grade, framework, subject).
- `lib/admin/queries.ts` convention established with the snapshot function as its only entry.
- Empty admin UI surface — no menu items, no screens. The data accumulates silently.

### Why all of this matters strategically
This round shifts the planner's identity in three meaningful ways:
1. **From single-school to multi-school ready.** Any school using any framework can adopt it.
2. **From "what we plan" to "what we plan AND what we actually do."** Extra lesson events and day events capture the lived reality of a classroom, not just the master curriculum.
3. **From a teacher tool to an instructional leadership system.** The reporting foundation, even unbuilt, means by the time an admin section ships in Phase 3+, it has months of real data to surface.

### Estimated Phase 1 cost increase
- Multi-framework schema + CSV upload + standards picker rewrite: ~3-4 days.
- Three distinct event types + "+ Add to day" chooser: ~2 days.
- `CoverageSnapshot` table + nightly job + `lib/admin/queries.ts` convention: ~1 day.
- Total: ~6-7 additional days on top of existing Phase 1 scope.

## 25. Claude Design Considerations Not Yet Captured Elsewhere (added)

The earlier rounds focused mostly on *what* the planner does (entities, views, flows). The following are *how it should feel and behave under real-world conditions* — items Claude Design needs but that the spec hasn't called out explicitly yet. Treat these as design-brief additions for the prototyping phase.

### 25.1 Empty states for every view
Every view will spend time in an "empty" condition — a new school year before lessons are entered, a brand-new teacher's first login, a filter that returns nothing, a unit with no lessons yet, a Comment Browser with no matches. Empty states should not be apologetic blank screens; they should be **action-oriented prompts** ("This week is empty. Add a lesson, or copy from a previous week.") with the relevant action available right there. Specify an empty state for:
- Weekly view (week with zero lessons)
- Daily view (day with no academic blocks)
- Subject view (filtered subject with no matches)
- Unit view (unit with no lessons assigned yet)
- Comment Browser (no comments matching filters)
- To-Do panel (no items in active scope)
- Reports library (no saved exports yet)
- Standards picker (no standards in a uploaded framework yet)
- A teacher's very first login (onboarding moment — see 25.10).

### 25.2 Loading, error, and offline states
- **Loading.** Skeleton placeholders that match the eventual layout (not spinners) — keeps the visual rhythm stable while data arrives. Specify skeletons for the weekly grid, the lesson card, the right pane, the to-do slide-out, and the Comment Browser.
- **Error.** Inline error messages anchored to the failed action ("Couldn't save this lesson — check connection") with a retry affordance. No global toast popups except for the offline indicator.
- **Offline.** A subtle persistent banner ("You're offline — changes will sync when you reconnect"). Optimistic UI for the most-used actions (completion checkbox, note edits) so a flaky school Wi-Fi doesn't break flow.

### 25.3 Accessibility — non-negotiable baseline
Public schools in Qatar (and most jurisdictions) have students and staff who need accessibility support. Bake in from prototype day, not retrofit:
- **Keyboard navigation** — every action reachable without a mouse. Arrow keys to move across the weekly grid, Tab through the right pane, Enter to expand a card, Esc to collapse. The right-click context menu must be reachable via keyboard menu key or a "⋯" focus affordance (already specified — confirm in prototype).
- **ARIA labels and semantic landmarks** on the top bar, side panels, main canvas, modals. Screen reader users should be able to navigate the six views without sighted help.
- **Color contrast** — the subject palette uses color heavily. WCAG AA at minimum for any text on a subject-colored background. The "urgent" pulsing red must remain readable for users who set "reduce motion" (drop the pulse, keep the color).
- **Color-blind safety check** — the eight-subject palette needs to be distinguishable for deuteranopia and protanopia (common forms of red-green color blindness). Test with a color-blindness simulator before locking the palette. If two subjects collapse visually, adjust hue or add a secondary differentiator (subtle pattern, short text code).
- **Focus indicators** — visible 2px outline on whatever is keyboard-focused. Not optional.

### 25.4 iPad and tablet ergonomics
Teachers were called out specifically as moving between laptop and iPad. Designing for the iPad isn't an afterthought:
- **Touch targets ≥ 44px square** (Apple HIG minimum). Completion checkboxes, expand chevrons, the "⋯" menu affordance, all reordering grab handles — all 44px.
- **Drag-and-drop on touch** needs an explicit grab handle (a small drag-bars icon on the left of a lesson card), not "long-press anywhere on the card" which conflicts with click-to-expand. Specify the touch interaction explicitly: tap the handle, then drag.
- **Right-click equivalent on touch** — the "⋯" affordance is the iPad's only path to the context menu. Make it always visible on iPad (not hover-revealed), at least as a subtle dot at the top-right of every card.
- **Pinch-to-zoom not needed** — but two-finger scroll should work normally inside expanded cards.
- **Soft keyboard handling** — when a quick-add input is focused on iPad, the keyboard takes ~40% of the screen. Inputs must scroll into view; don't trap them behind the keyboard.

### 25.5 Motion and animation
The spec has one explicit animation (urgent daily note pulsing red). Beyond that, the design tone is "clean, flat, calm — not playful." Specify a small set of allowed motions and forbid the rest:
- **Allowed:** card expand/collapse (200ms, ease-out), drag ghosting (visual fade of original position), slide-out panel transitions (250ms), focus indicators (instant, no fade-in).
- **Forbidden:** bounce, parallax, page-flip transitions, decorative confetti, anything implying delight at the expense of speed. Teachers using this tool for 6 hours a day will hate exuberant motion within a week.
- **`prefers-reduced-motion` respected** — when the OS setting is on, drop expand/collapse animation and the urgent-pulse. The information remains; the motion goes.

### 25.6 Density modes (defer to Phase 2, but design with it in mind)
The spec calls for high information density. Teachers prepping at home on a 27" monitor want one density; iPad users in the classroom want another. Don't build a density toggle in Phase 1, but design the lesson card and weekly grid such that a future "compact mode" can be retrofitted by adjusting padding tokens, not redrawing components. Concretely: lesson card padding should be a CSS variable, not a hardcoded value.

### 25.7 Iconography system
Many small icons appear across the spec: completion checkboxes, resource type icons (PDF/image/doc/YouTube/slides), event type icons (assembly/drill/guest speaker), the "⋯" menu, the 💬 comments badge, the urgent/important/fyi priority indicators, framework badges, drag handles, the personal-fork dot. Specify:
- **A single icon family** (e.g. Lucide, Heroicons, Phosphor) for consistency. Don't mix line and filled icons.
- **An emoji policy.** The spec uses emojis for the chooser (📘 ✨ 📅) and the comment icon (💬). Decide: emojis only in the chooser and badges, never in lesson content. Otherwise emoji creep makes the tool feel inconsistent.

### 25.8 Dark mode — decide now, build later
Teachers often plan at home in the evening. Dark mode is genuinely useful for this audience. Don't build it in Phase 1, but **design every component in Phase 1 with dark-mode-compatible tokens** (use CSS variables for colors, not hex literals). Retrofitting dark mode later costs hours instead of weeks if the tokens are in place from day one. Open question for §10: build dark mode in Phase 2 or Phase 3?

### 25.9 Design tokens file (handoff bridge between Design and Code)
A small file in the repo — e.g. `tokens.json` or `tokens.css` — that holds every spacing value, radius, elevation, color, and font size used in the design system. Both Claude Design (when prototyping) and Claude Code (when building) read from the same tokens. This prevents the slow drift where the prototype shows 12px padding and the production app ends up with 16px because someone eyeballed it. Specify:
- Spacing scale: 4, 8, 12, 16, 24, 32, 48, 64 (px).
- Radius scale: 4, 6, 8, 12 (px).
- Elevation scale: 0, 1, 2, 3 (shadows for cards, slide-out panels, modals).
- Color tokens for every semantic role (background, surface, surface-elevated, text-primary, text-secondary, border, focus, danger, success, etc.) plus subject palette and tag palette.
- Type scale: 11, 12, 13, 14, 16, 18, 22 (px) with weights 400 and 500.

### 25.10 Onboarding and first-run experience
The teacher's first 30 seconds in the app set the entire perception. Specify:
- **First login.** A short welcome modal: "Welcome to [App]. This is your Grade 5 planner." Three steps: confirm name, pick default view, pick completion privacy. Skippable.
- **Empty year.** If a school year exists but no lessons yet (the school's first day on the tool), the weekly view shows a guided empty state: "Let's add your first lesson, or import last year's plan from a CSV." Imports are out of Phase 1 scope — but the empty-state prompt is the trigger to discuss whether bulk import should join Phase 1.
- **First lesson created.** A tooltip pointing at the completion checkbox: "Tap when you finish teaching. Your team won't see this unless you choose to share."
- **First time switching to master mode.** A one-shot tooltip: "You're now viewing the master plan. Click 'Now editing master' to make changes."

### 25.11 Print preview as a designed surface (not an afterthought)
The Print Preview pages (§5.9) and the Export Center PDF template (§5.12) both render curriculum to paper or PDF. Both deserve real design attention:
- **Print stylesheet** should be designed alongside the weekly view, not bolted on at the end. Margins, page breaks between weeks, header repetition (subject + week number on every page), font sizing for legibility on paper.
- **Export PDF template** is the document a school principal or MOEHE inspector will read. It is the planner's external face. Header with school name and grade, generation date, a clean typographic hierarchy, no UI chrome, no color codes that depend on color (use shape/label for standards type, since some printers are B&W).

### 25.12 Holiday and Ramadan visual markers
The data model supports holidays and a Ramadan range. The visual design needs to specify what they look like:
- **Holidays** — weeks containing a holiday show a subtle ribbon or backdrop tint (not a screaming red banner). The holiday name appears small at the top of the day cell. Multi-day holidays span their days.
- **Ramadan range** — the weekly view shows a small "🌙 Ramadan timetable active" indicator at the top of the week when Ramadan mode is on. The Schedule view shows shorter time blocks. The Daily view shows a small indicator on the day's date.
- **Locked / non-school days** — weekends (Friday and Saturday in Qatar) are visually distinct from school days; not just hidden. A small weekend column or end-of-row marker makes it clear that Sun–Thu is the school week, not the full calendar week.

### 25.13 Search results as a designed surface
Top-bar search is mentioned but never specified. Search should query lessons, units, standards, comments, and to-dos — and show results grouped by type with clear anchors and jump-to behavior. This is its own component (a dropdown or full-screen results panel) that deserves a prototype.

### 25.14 Notification surface (Phase 2, but design now)
Master-change notifications (Phase 2) and real-time comment updates (Phase 2) both need a notification surface. Decide: badge-only with a notification panel reachable from the top bar, or inline banner at the top of relevant views? Spec'ing this in Phase 1 (without building) ensures the top bar has a slot for the notification icon from day one, instead of being awkwardly squeezed in later.

### 25.15 Strategic note
Every one of these items is small individually. Together they're the difference between a planner that feels like a draft and one that feels like a product. The right time to specify them is now, before Claude Design starts prototyping — even the ones being deferred to later phases. A token file and an accessibility checklist exist or don't exist; specifying them in week one costs hours, in month six costs weeks.

## 26. Theme System, Density, and AI-Assisted Curriculum Import (added)

Three open questions from §25 got sharpened into real decisions this round.

### Live theme templates (not just dark mode)
The original "dark mode — Phase 2 or 3?" question was too narrow. The actual need is **a small set of complete themes** teachers can switch between live. Five themes targeted for Phase 2:

- **Default** — clean white surfaces, full subject color.
- **Calm** — softer warm-white background, muted subject colors, still light.
- **Dark** — dark background, full subject color, light text.
- **Focus / low-color** — neutral grays everywhere, subject color reduced to a thin stripe only. For teachers who find the rainbow grid visually noisy.
- **High Contrast** — pure white/black, AA+ contrast everywhere. Also serves accessibility.

Architecturally identical to dark mode (CSS variable override blocks per theme) but much more useful — teachers have wildly different preferences over a 6-hour workday.

**Phasing:**
- Phase 1: ships with the Default theme only. Token architecture supports themes even though no picker exists yet.
- Phase 2: Appearance settings panel with theme picker.

### Density folded into Appearance settings
Rather than a standalone density toggle, density becomes one of several controls in a unified **Appearance** settings panel (Phase 2):

- **Theme:** Default / Calm / Dark / Focus / High Contrast
- **Density:** Comfortable / Compact
- **Color intensity:** Full / Muted
- **Reduce motion:** On / Off (also respects OS `prefers-reduced-motion`)

All four combine freely. Settings persist in `TeacherUIState` and sync across devices.

### AI-assisted curriculum import (Phase 1)
The original "should we build CSV import in Phase 1?" question opened a deeper one: **schools' existing curriculum docs look wildly different.** A rigid CSV importer would fail on real-world data and force hours of manual cleanup — exactly the friction barrier the import is supposed to remove. The answer is to let Claude do the schema-mapping work.

**Three layers:**

1. **Strict CSV template** (Layer 1, cheap fallback). A downloadable canonical template with fixed columns. Teachers whose data is already clean paste it in and upload. Validates and inserts directly.

2. **AI-assisted import** (Layer 2, the workhorse). Teacher uploads any format (.csv, .xlsx, .xls, .txt, .md). A Cloudflare Worker calls the Claude API with the file's plaintext + the canonical schema + few-shot examples. The API returns normalized lesson objects with confidence levels (high/medium/low). A preview screen shows source-row → mapped-fields side-by-side. Teacher reviews the "Needs review" tab for low-confidence rows, edits or skips, and confirms.

3. **Developer-assisted seed** (Layer 3, service offering). For data too chaotic even for Layer 2 (40 Word docs, scanned PDFs), a paid migration. The developer uses the same Layer 2 tooling with manual cleanup. Not a feature — a service line item mentioned in onboarding docs.

**Estimated cost:** ~$1-3 in Claude API per school import (one-time at onboarding). Build cost ~5 days.

**Why AI rather than a smarter rigid form:** every school's spreadsheet is different. Column names vary, merged cells exist, prose mixes with structure, multiple sheets per year, free-form notes. Asking the teacher to clean the data into a fixed shape before upload is exactly the friction this is trying to remove. The Claude API absorbs the variability.

**Why this matters strategically:**
- Without this, every new school faces a 2-week manual data entry barrier before the tool delivers value. That's a high adoption risk — momentum dies before the first real benefit.
- "Bring your existing curriculum in any format, we'll figure it out" is a notable feature in its own right and a real differentiator versus tools demanding a clean CSV.

### Open questions retired / sharpened
- Q25 (dark mode timing) → answered: theme system Phase 2, includes Dark.
- Q26 (density toggle) → answered: folded into Appearance settings Phase 2.
- Q27 (bulk import) → sharpened into specific 3-layer architecture with confirmed Phase 1 scope.

### Phase 1 cost increase from this round
- Theme/density token architecture (no themes built, just tokens-everywhere discipline): ~0.5 days.
- AI-assisted import (upload form + Cloudflare Worker + Claude API call + preview UI + conflict resolution): ~5 days.
- Total: ~5-6 additional days on Phase 1.

## 27. Color Refinement and Custom Backgrounds (added)

The early Claude Design prototypes came back looking muted — gray-on-white with thin colored stripes — which clashed with the actual intent ("color must be used aggressively for instant subject/unit recognition" and Padlet-style vividness). Root cause: the original §6.1 tone language ("clean, flat, calm — not playful") directed designers toward restraint everywhere, including in the meaning-carrying color.

### The contradiction in the original spec
- §6.1 said: "calm, not playful."
- §5 said: "color must be used aggressively."
- Padlet was repeatedly cited as a reference point.

These can't all be true at once. The resolution: **vibrant where color carries meaning; calm in supporting chrome.** Subject stripes, unit tints, status colors, priority indicators — all full-saturation. Top bar, panels, navigation chrome, body text — restrained. Padlet vibrancy in the curriculum content; SaaS restraint in the framing.

### Specific changes
- §6.1 tone rewritten to drop the "calm, not playful" framing and replace with the vibrant-content / restrained-chrome principle.
- §6.2 color system extended with explicit guidance:
  - Each subject color now has three named steps: **stripe** (full saturation, ~500-600 weight), **tint** (~50-100 weight), **deep** (~700-800 weight).
  - Lesson cards use **subject tint as background fill**, not pure white. The full-saturation stripe sits on the left edge.
  - Unit rows use light tints of the subject color, not gray.
  - Reference points called out (Padlet, Notion gallery, Linear). Anti-pattern called out (Asana's gray-on-white density).
  - Anti-muted-output guardrail explicit: prototypes that come back gray or with color only as a 2px stripe should be rejected.

### Custom backgrounds (Padlet-style backdrops)
The teacher specifically asked for custom backgrounds — patterns, photos, etc., the visual personality piece Padlet does well. Three decisions locked in:

**Scope of the background:** **Behind main canvas only.** Top bar, side panels, cards, modals all remain opaque. This protects readability — a teacher reading directions never sees text on a busy pattern. Backgrounds fill negative space around the grid.

**Library scope:** **Phase 2 curated library, Phase 3 upload-your-own.** Curated set covers solid colors, subtle textures, patterns, and photos/illustrations (including seasonal sets). Each background has light and dark variants.

**Upload-your-own (Phase 3):**
- Personal-only — never team-wide. Sidesteps moderation entirely.
- Automatic contrast overlay applied at upload time.
- WCAG AA enforcement: contrast check runs against surface tokens layered on the proposed background. Block save until passing.
- School-admin opt-in (default off). Schools may have policies against arbitrary uploads.
- ≤5 images per teacher, oldest auto-pruned.

### Phase 1 architecture (no UI built)
- `background_url` token in the token system, default = solid neutral.
- `appearance_settings` JSON blob on `TeacherUIState` (theme, density, color_intensity, background, reduce_motion). Phase 1 writes defaults only.
- Two new audit log actions: `background_uploaded`, `background_deleted` (will only fire in Phase 3).

### Why this matters strategically
Vibrant content + decorative backdrop + restrained chrome is a specific point in design space:
- **Padlet** gets this right (vibrant, decorative, but a bit chaotic at scale).
- **Notion** gets it half-right (decorative covers, but content is often gray-on-white).
- **Most SaaS planners** get this wrong (clinical gray everywhere, color used as a hint only).

The planner sits between Padlet's warmth and Notion's structure. The color refinement is the difference between teachers feeling they're using a database and teachers feeling they're using their classroom. That's not cosmetic — it's the difference between adoption and abandonment.

### Open questions retired / added
- Q25 expanded to cover custom backgrounds in addition to themes.
- Q26 updated to include the background picker in the Appearance panel.
- Two new questions added:
  - Q28: Upload-your-own backgrounds — school-admin opt-in or default-on? **Answer: opt-in, off by default.**
  - Q29: Background contrast enforcement — block save below AA, or warn and allow override? **Answer: block save. Readability is not negotiable.**

### Phase 1 cost increase from this round
- Token-level architecture for backgrounds + `appearance_settings` blob: negligible (~0.25 days extra on top of the existing token work).
- The real cost is design effort in Phase 1 to nail the vibrant default theme, which doesn't add days but does add iteration cycles on the prototypes.

## 28. Parallel Themes — Quiet and Vivid (added)

A late refinement after the color round. Rather than committing to either the muted aesthetic (existing prototypes) or the vivid one (last round's refinement), build both and decide after seeing them side by side. Both may ship long-term as user-selectable options if both work.

### The two themes
- **Quiet** — the existing muted calm aesthetic. Continue building remaining components in this style. Near-white cards, thin subject stripes, subtle grayscale unit shading.
- **Vivid** — the parallel new build. Subject-tinted card backgrounds, full-saturation stripes, light-tint unit row backgrounds. Padlet-grade color recognition without sacrificing readability. Anti-pattern: gray-on-white density.

### Decision points the parallel build resolves
- Does subject tint help recognition, or distract from reading content?
- Does the vivid grid feel chaotic at scale (7 subjects × 5 days × 40 weeks)?
- Does Quiet feel professional or sterile?
- Does Vivid feel warm or overwhelming?

None of these can be answered by spec alone. They need rendering. Building both in parallel gets to the answer 4-6 weeks earlier than building one, shipping, gathering feedback, and pivoting.

### Architecture (already in place from the last round)
- Token-driven design system already specified — every color/spacing/radius reference is a token.
- A theme is a CSS variable override block. Switching is an instant class swap on `<html>`.
- `TeacherUIState.appearance_settings` already holds theme selection.

So the cost of "build both" is not "build twice." It's: design components once, define two token sets, and validate that the components render correctly under each set.

### Naming
"Muted calm" and "vibrant where color carries meaning" are descriptions, not product names. Locked in user-facing labels:
- **Quiet**
- **Vivid**

Short, descriptive, neutral about which is better. Used in the Settings theme toggle, the user docs, and the design files.

### Phase 1 ships with a minimal theme toggle
Just the two options. The fuller Appearance settings panel (density, color intensity, backgrounds, reduce-motion) still arrives in Phase 2.

**Default at first login:** Quiet. Lower-risk choice for teachers seeing the tool for the first time. They can switch with one click if they prefer Vivid.

### Design implication
Every Claude Design artifact from here forward is built in both themes. The token architecture makes this cheaper than it sounds — designing the component once is the bulk of the work; theming it twice is mostly a token-override pass.

### Why this matters
The original spec contradicted itself ("calm, not playful" vs. "color must be used aggressively"). The muted prototypes were the natural compromise — and they're a perfectly defensible aesthetic. Vivid is a hypothesis about what would serve the same teachers better. The honest answer is: nobody knows yet. Parallel build is how you find out without betting the project on one guess.

### Phase 1 cost increase from this round
- Designing both themes in parallel: ~1-2 days extra design iteration (not coding — coding is the same components).
- Building a minimal theme toggle in Settings: ~0.25 days.
- Total: ~1-2 days. The real cost is iteration cycles in Claude Design, not engineering time.

## 29. UX Improvements and Daily Notes / Day Shoutbox Split (added)

Late-round pass triggered by Claude Design feedback on the early prototypes. Several discoverability, accessibility, and conceptual-overlap issues surfaced. All resolved in this round.

### Daily Notes vs. Day Shoutbox — clean split locked in
The earlier spec had Daily Notes as "shared OR personal" plus a Day Shoutbox for team-by-date chat. Two surfaces that did substantially similar things. Resolution:
- **Daily Notes = personal only.** Private reminders for the author.
- **Day Shoutbox = team only.** Team-visible conversation tied to the date.
- Same date, two surfaces, two purposes, no overlap.

The original spec's 9px lock icon was too small and too binary. New system:

|---|---|

| **Personally moved** (day/order change) | Solid stripe + move-arrow icon (↔ or ⤴) in top-right |

The first three rounds defined what the planner *does*. This round defines what it *feels like to use* — the small surfaces where teachers either feel competent or confused. The cumulative effect is the difference between a planner that ships features and a planner that ships a workflow.

## 32. Personal subjects — full lifecycle (added)

A refinement that fills in real gaps in how personal subjects work end-to-end. The earlier round added the concept; this round defines the lifecycle from creation through promotion through year rollover.

### The time-slot-vs-subject distinction
When a teacher adds a time block in Schedule view setup (e.g., "Morning Meeting" at 8:15am), they're asked: is this a **time slot** (labeled time with optional notes/resources, no subject row in Weekly view) or a **personal subject** (full subject row with color, units, events, completion tracking)?

Default is time slot — most non-academic blocks (lunch, prep) don't deserve a subject row. But teachers who run a structured Morning Meeting curriculum should be able to upgrade. Conversion between the two is supported later via context menu:
- Time slot → Personal subject: opens the personal-subject sub-form; existing notes become an opening Day Event; resources attach.
- Personal subject → Time slot: warning shown, existing events archive to Catch-up, time slot keeps only notes and resources.

### Personal subjects work like team subjects for their owner
A personal subject is not a watered-down version. The owner can:
- Create units within it (e.g., "Morning Meeting Unit 1: Building Community").
- Place Core Lesson Events with learning objectives, directions, standards tags, resources.
- Place Extra Lesson Events (one-off or recurring).
- Tag standards using the per-subject default frameworks.
- Print, export, mark complete, drag-and-drop.

The only difference from a team subject: it appears only in the owner's views.

### Promotion to team-shared
Settings → My personal subjects → "Share with team" sets `promotion_status = pending` and creates an entry in the lead's approval queue. **Lead approves, with delegation to `grade_admin` available** (the lead can hand off to a designated admin via Settings).

On approval:
- `scope` becomes `team`, `owner_id` clears.
- The subject appears as a row in every team member's Weekly view from that moment.
- **Owner's existing events stay personal-only.** Promotion shares the subject *definition* (name, color, default frameworks), not the event history. This protects against accidentally exposing a teacher's notes/drafts when they just wanted to add the row to the team grid.
- Other teachers can add their own events on the new team subject going forward.

Rejected requests can be re-requested after editing.

### Year-over-year persistence
Personal subjects belong to the teacher, not the year. At year rollover:
- Subject definition (name, color, display order, default frameworks) carries forward.
- The curriculum structure clones — units, core lesson events with all their authored content (titles, learning objectives, directions, standards tags, resource links, display order), and recurrence patterns.
- Completion records do NOT clone — they archive with the previous year. The new year starts with empty completion.
- Day Events do not clone (date-specific, not curriculum).

This means a teacher who built out a year of Morning Meeting curriculum walks into next year with the same skeleton ready to teach again. Without this, every teacher would have to rebuild their personal curriculum every August — a non-starter.

### Audit log
Eight new action types added: `personal_subject_created`, `personal_subject_promotion_requested`, `personal_subject_promotion_approved`, `personal_subject_promotion_rejected`, `personal_subject_promotion_cancelled`, `time_block_converted_to_subject`, `subject_converted_to_time_block`, `promotion_approval_delegated`.

### Why this matters
Without this lifecycle, personal subjects would be either too rigid (can't share useful inventions with the team) or too leaky (anyone can add team subject rows). Lead-approval-with-delegation threads the needle: teachers can invent, but the team's visual taxonomy stays curated.

Year-over-year persistence is the part that makes personal subjects worth building. Without it, a teacher who builds a year of Morning Meeting curriculum loses it at year end, and the feature becomes "one-off events with a subject row." That's not enough motivation to use the feature seriously.

### Phase 1 cost
- Subject entity refinement (promotion fields): ~0.5 day.
- Time block creation flow (the time-slot-vs-subject question + sub-form): ~1 day.
- Conversion actions (time slot ↔ personal subject): ~0.5 day.
- Promotion request/approval/rejection flow + approval queue UI: ~1.5 days.
- Year rollover logic for personal subjects: ~1 day (Phase 2 actually, since year rollover ships in Phase 2 — but the data-model considerations are Phase 1).
- **Total Phase 1: ~3-3.5 days.** Year rollover sits in Phase 2.

## 33. Opt-in rollover for personal subjects (added; refined)

Refinement on §32 — making year rollover of personal subjects explicit instead of automatic. After a follow-up clarification, the semantics are: **every teacher with personal subjects is asked at rollover; the per-subject preference set at creation only pre-selects the radio.**

### What changed
The §32 spec had personal subjects auto-rolling over each year. That's wrong for a meaningful slice of cases: a teacher who experimented with a Morning Meeting curriculum for a year and didn't love it shouldn't be forced to drag it into next year. New behavior:

- Every personal subject has a `rollover_preference` field: `ask_me` | `yes` | `no`.
- Default on creation: `ask_me`.
- Teacher can change anytime via Settings.

At rollover, **every teacher with at least one personal subject sees the Year-end Rollover Decision screen.** The preference value pre-selects the row's radio but does not skip the prompt:

- `yes` → row appears with Roll over pre-selected; teacher can override per-row or accept all with one click.
- `no` → row appears with Archive pre-selected; teacher can override.
- `ask_me` → row appears with nothing pre-selected; teacher must actively choose.

### Why always-ask
Earlier draft had `yes`/`no` skip the prompt and only `ask_me` trigger it. The teacher caught this — a subject set to `yes` at creation and forgotten could quietly resurrect next year, which is exactly the surprise the round was trying to prevent. Better: the preference saves keystrokes, but the moment of confirmation is universal.

This also makes bulk actions more useful. A teacher who set most subjects to `yes` lands on the screen, sees them all pre-selected as Roll over, and clicks Confirm — one click. A teacher with mixed preferences sees a mixed pre-selection. A teacher with `ask_me` everywhere does the per-row work.

### The Rollover Decision screen
When the lead initiates rollover, every teacher with at least one personal subject lands on a dedicated screen on next sign-in:
- Per-row radios (Roll over / Archive) on every subject, pre-selected per preference.
- Year-end stats per row (units, events authored, completed, not-done).
- Preference badge per row ("You chose: Roll over" / "You chose: Archive" / "No preference set").
- Bulk actions: Roll over all / Archive all / Reset to my preferences.
- "Skip for now" option that defers the decision — Weekly view shows a banner until resolved.
- Submission is atomic; each decision writes a `personal_subject_rollover_decided` audit entry with metadata on whether the teacher accepted or overrode the pre-selection.

Teachers without any personal subjects skip the screen entirely.

### Knock-on decisions
- **Pending promotions on archived subjects auto-cancel.** You can't promote a subject that no longer exists in the new year. Lead is notified.
- **Recurring events follow their parent subject.** If a subject rolls over, its recurrences roll over with adjusted `school_year_id` and `start_date`. If a subject archives, its recurrences archive with it.
- **Archived subjects are restorable.** Settings → Archived subjects → "Restore" clones the archived definition + curriculum structure into the active year.

### Audit log additions
Three new actions: `personal_subject_rollover_decided`, `personal_subject_archived`, `personal_subject_restored`. The decision entry's metadata captures whether the teacher accepted the pre-selection (signaling "set-and-forget worked for me this year") or overrode it (signaling "I changed my mind on this one").

### Why this matters
Silent automatic rollover treats every personal subject as equally precious. They're not. Teachers iterate on rituals — some land, some don't. An always-ask flow with pre-selection respects both autonomy (the explicit moment of confirmation) and efficiency (the saved keystrokes).

The audit metadata about pre-selection vs. override is also useful product feedback later: if many teachers consistently override their `yes` preferences to Archive, that's a signal that the creation-time choice isn't a good predictor of year-end intent.

### Phase 1 vs Phase 2 split
- Phase 1: `rollover_preference` field on Subject; Settings UI for setting it per-subject (the three-option control); Archived subjects subsection in Settings (visible but empty until first rollover).
- Phase 2: the actual rollover logic + the Year-end Rollover Decision screen + the Weekly-view pending banner.

This matches the existing split where Year Rollover itself ships in Phase 2. The data-model considerations land in Phase 1 so the column exists from day one.

### Phase 1 cost change
- `rollover_preference` field + Settings UI: ~0.5 days.
- Net change vs prior round: negligible — the rollover screen itself is Phase 2 work.

## 38. Resource attachments moved to Phase 1B; Cloudflare R2 chosen as storage backend (added)

The teacher asked whether hosting PDFs, Word docs, and resource files would be too expensive, and whether to use Supabase Storage or Cloudflare R2.

### Cost reality check
At the beta scale (4 teachers, ~800 MB realistic max), both Supabase Storage (100 GB Pro quota + 250 GB egress) and Cloudflare R2 (10 GB free tier + zero egress at any volume) are essentially free. Even at 50 schools / 5,000 teachers, both stay well within free tiers. Storage cost was never the blocker — engineering effort was.

### Why Cloudflare R2 over Supabase Storage
Three reasons:
1. **Zero egress fees, ever.** Supabase charges $0.09/GB beyond the included egress; R2 has no egress fees at any volume forever. If the planner ever serves resources to students/parents (Phase 4), this matters a lot.
2. **Already on Cloudflare.** Pages hosting + future Workers for AI features (Phase 3) + R2 = one account, one billing surface, one set of credentials.
3. **R2 free tier covers the realistic growth path indefinitely.** 10 GB storage + 1M Class A ops + 10M Class B ops, with zero egress, covers the beta and most plausible schools for years.

The one tradeoff: Supabase Storage integrates natively with Supabase Auth RLS. With R2, file access requires a Cloudflare Worker that verifies the requesting teacher's Supabase session before issuing a short-TTL signed URL. ~1 day of additional work upfront; Claude Code can write it.

### What lands in Phase 1B
Resource attachments — a substantial feature that was originally Phase 3 — moves up to Phase 1B because:
- Storage cost is no longer a blocker (R2 free tier).
- The team's existing Padlets need a replacement and waiting until late 2027 is too long.
- Annotation (the harder half of the Phase 3 work) stays in Phase 3; only hosting + viewing + drag-and-drop attachment ship in 1B.

**The resource model (§4.5a + §5.21):**
- **Hosted files:** PDF, DOCX, RTF, single image, image stack. 25 MB per file. Max 10 hosted files per lesson (soft warn at 8, hard block at 10). Image stack counts as ONE file slot regardless of image count.
- **Unlimited links:** external URLs (with Open Graph preview banners), YouTube (with thumbnail), Google Drive (with file-type icon).
- **No videos hosted, ever** — video files are too large and create unbounded storage growth. YouTube links only.
- **No Drive uploads** — Drive links only; Drive's own permissions govern who can open.

**Resource viewer (§5.21):**
- Two-pane layout: main canvas for the focused resource + persistent ~30% right sidebar with other resources, scrollable lesson notes, and standards chips.
- Lesson card displays type-indicator icons (📄🖼️🔗📺📁📝) showing what's attached.

- Image stacks render as slideshow with thumbnail strip.

**Drag-and-drop attachment:**
- Files from desktop → drop on lesson card → uploads + attaches.

### Three product decisions from the research

**1. Parent-facing share view / class website → Phase 2.** Originally Phase 4. Tokenized URL, no parent login, teacher controls which sections + date range + resource visibility. ~1 week of engineering.

**2. Standards coverage report (teacher-facing) → Phase 2.** Already had `CoverageSnapshot` infrastructure in Phase 1A; the teacher-facing report uses existing data with no new infrastructure. Lists every standard with lesson counts and the lesson list, including an "Untagged" section. Closes the most-requested admin question without needing the full Phase 3 admin section. Note: the *admin-facing* aggregate coverage dashboard (cross-teacher gap/redundancy analytics) stays in Phase 3.

**3. Cross-school curriculum marketplace → skipped.** The research recommended this for Phase 4; I pushed back because marketplaces are platform-scale plays needing content moderation infrastructure, payment processing, IP/licensing legal work, and quality curation. Doing it badly is worse than not doing it. The teacher agreed — keep focus on the planner.

### Planbook feature absorption

- Student accounts (explicit scope cut)
- Built-in gradebook (would explode scope)
- Attendance (same)
- Preloaded instructional strategies database (Kagan, Marzano, etc. — US-centric; multi-framework standards approach is more flexible)
- Embed-on-website HTML (covered by class-website Phase 2 spec)
- SAML 2.0 / Apple SSO (Phase 4 if ever; Google + Microsoft SSO covers 95%)
- Aggregate student reporting (Phase 4 territory)

### Mobile strategy resolved
The teacher asked whether they could "just port the website" to native iOS/Android. The honest answer is **yes, three ways with very different costs**:

1. **PWA (Progressive Web App)** — manifest + service worker, install from browser to home screen. 2-3 days of work. ~85% of what teachers actually want from "an app."
2. **Capacitor wrapper** — wraps the existing React app in a thin native shell for real App Store / Play Store listings. 1-2 weeks plus app store overhead.
3. **True native rewrite** — months of work; overkill for this product.

**Decisions:**
- **Phase 1B: PWA.** Includes a first-mobile-login install prompt walking teachers through Add to Home Screen on iOS Safari (without it, ~80% of teachers never discover the install flow). Android push notifications work; iOS push deferred.
- **Phase 4: Capacitor wrapper only if institutional sales motion requires it.** PWA covers ~85% of the "feels like an app" need; App Store listings are mainly a school-admin / parent credibility signal that becomes relevant when GTM motion shifts from teachers to schools.
- **Skip true native rewrite entirely** unless a category-redefining native interaction emerges that the WebView can't deliver.

This is meaningfully better than the original "Phase 4 thing we won't get to for years" framing. Teachers get an app icon on their iPads at Phase 1B for ~3 days of engineering.

### Other research-driven additions
- **Substitute mode → Phase 3** (from Phase 4). Tokenized URL, hides personal content, exposes full lesson plans + Substitute Notes, prints a clean sub-binder PDF. Planbook has Substitute Views but no competitor surfaces a complete sub mode; cheap differentiator.
- **Curriculum-grounded AI → Phase 4** (Brisk's "Curriculum Intelligence" model). When AI generation runs in Phase 3, the AI uses the school's actual Master curriculum + standards + recent lessons as context, not a generic prompt. Differentiator vs. generic AI tools because *we are the curriculum source of truth they're trying to ground themselves in*.
- **Open-curriculum import from Oak National Academy → Phase 4.** UK market wedge: Oak publishes UK National Curriculum lessons under open license with an MCP server. Import as forkable Masters. ~1 week for substantial UK GTM benefit.
- **LMS integration (LTI 1.3) → Phase 4** (with explicit "this is for institutional sales").
- **SIS rostering (PowerSchool, Veracross, Compass, Sentral) → Phase 4** (same reasoning).
- **"Curriculum diff" view → Phase 4.** Natural extension of the Git-style fork architecture; show line-level diff between Master and queued update. Nobody in K-12 has this.
- **Curriculum versioning & branch history → Phase 4.** Full Git-like history; revert to prior version. Free given the fork architecture.
- **AI catch-up suggestions → Phase 4.** Combines unique Catch-up surface with AI capability competitors race on.
- **Wellbeing / cognitive-load nudges → Phase 4.** Extend Teaching Reminders to personalized "you've stacked 3 high-load lessons this week."
- **Multi-language UI → Phase 4** (Spanish, French, German, Simplified Chinese after English-only beta lands).

### What didn't make it (and why)

- **Native iOS/Android in Phase 2** (research recommended) — PWA in Phase 1B does this better at 1/10th the cost.
- **AI generation features in Phase 2** (research recommended) — teacher explicitly moved AI to Phase 3+ in earlier rounds. The research argued "Generate" buttons are table-stakes by 6 months post-launch; the teacher's instinct is right that AI isn't urgent for beta and the field changes every 6 months.
- **Multi-language UI in Phase 3** (research recommended) — meaningful engineering investment; English covers US/UK/IB schools at launch.
- **Marketplace** — declined.

### Phase 1B reality check
Phase 1B was already ~9-11 weeks before this round; now adds:

- PWA + install prompt: ~3 days
- Net: Phase 1B now ~10-12 weeks.

Phase 2 adds:
- Class website / parent share: ~1 week

- Net: Phase 2 grows from ~4-6 weeks to ~7-9 weeks.

Phase 1A is unaffected — the late-August beta gate doesn't shift. Phase 1B wraps mid-December instead of late November; Phase 2 wraps late May 2027 instead of mid-April.

### Honest takeaway
This round substantially strengthened the competitive position without compromising Phase 1A beta delivery. The biggest single win is **PWA in Phase 1B** — for ~3 days of engineering, the team gets the "app icon on their iPad" experience that closes the most visible competitive gap with Chalk and Toddle. The class website + standards report + Reminders Panel additions to Phase 2 give school-admin buyers concrete things to point at when comparing to Planbook and Common Curriculum.

## 41. Deeper Planbook absorption — search, status tags, lesson bank, cross-teacher copy, admin reports (added)

A focused round that did a more careful re-read of the Planbook features document and surfaced eleven features genuinely worth adding. The teacher asked "did you see anything from the Planbook document I should include?" — answer was yes, and on honest review of my prior scan I had missed several.

### What I missed on first pass

The earlier round added six Planbook features (Bump cascade, Extend Lesson, Lock to Date, Reminders Panel, Repeating to-do, Digital stickers). On second read, with fresh eyes, I found six more genuinely worth absorbing — plus five clearly worth skipping documented for the conversation record.

**The biggest miss: full-text search.** No competitor planner ships without search; my Phase 1A/1B specs had none. Teachers search their own history constantly ("what did I do for fractions last year?"). Without it the tool feels broken. Postgres full-text search with trigram is ~3 days of work — landing this in Phase 1B was the right call.

### Phase 1B addition
- **Full-text lesson search** — top-bar search box, `/` shortcut, queries titles + directions + notes + learning objectives + resource labels across active year and (toggle) archived years. Phase 2 expansion adds comments, to-dos, daily notes, lesson personal notes, resources.

### Phase 2 additions
- **Lesson Status Tags** (`draft | ready | approved | taught`) — lifecycle metadata distinct from completion tracking. Default `draft`; completion auto-sets to `taught`; only lead/grade_admin can set `approved`.
- **Status-based class website visibility** — pairs with status tags; class website's primary filter becomes "Show only Ready or higher" instead of date-range-only. Teachers can plan in draft without parents seeing half-baked plans.
- **Default view setting** — Settings option to lock the boot-up view (Daily / Weekly / Subject / Year / Last view I had open). More explicit than the current "remember last view" behavior.
- **View context note on share URLs** — optional teacher-authored header note rendered above the public class page ("Hi parents! Here's the week of plans...").
- **Lesson-specific personal notes** — new private text field on each lesson, visible only to the author, NOT part of Master/Personal forking. Fills the gap between Daily Notes (date-anchored) and lesson `notes` (forks normally). Use case: "remember to slow down here," "Sarah always struggles with this."

### Phase 3 additions
- **Cross-teacher curriculum copy** — package subjects/units/lessons into a `CurriculumBundle` and send to a colleague (in-app inbox) or export as portable file for cross-school sharing. Admin variant: bulk-transfer a departing teacher's entire year to a replacement in one operation. Critical for new-hire onboarding and departing-teacher continuity.
- **Team Lesson Bank** — calendar-independent department library of lessons. Save any lesson to the bank with name + tags; team members drag bank lessons onto future days to schedule. Fills the gap between scheduled lessons and archived years — a place for enrichment activities, snow-day backups, Q3 review lessons.
- **Teacher Plan Submission Report + Due Dates** — admin sets weekly plan deadline ("plans for next week due Friday 4pm"); system tracks Complete/Unsubmitted per teacher; Today dashboard surfaces reminders for behind-schedule teachers; admin can bulk-message unsubmitted. Off by default; most schools won't use it. Useful for districts with pacing-compliance requirements.

### Phase 4 documented-but-skipped
- **Section-level forking** — Planbook offers sharing of specific lesson sections while keeping other sections private. The spec is intentionally atomic at the lesson level for simplicity. Section-level forking would require sub-entity-per-section with its own fork status — significant complexity for a niche workflow. **Not planned.** Documented in Phase 4 so future readers know it was considered.

### Features deliberately skipped (with reasoning)

Worth recording for the conversation record since the teacher could revisit:

- **Gradebook / attendance / student rosters** — explicit scope cut. Toddle and Planbook have them; you're not them.
- **Visual seating charts** — niche feature requiring student rosters.
- **Preloaded instructional strategies databases** (Kagan, Marzano, etc.) — US-centric pedagogical taxonomies; your multi-framework standards approach is more flexible internationally.
- **Equal-size lesson view UI toggle** — cosmetic; defer indefinitely.
- **SAML 2.0 / Apple SSO** — Google + Microsoft SSO covers 95% of schools; SAML matters only for enterprise district sales (Phase 4 if ever).
- **Aggregate student reporting** — student-level feature; Phase 4 territory if student tracking ever ships.
- **Email comment alerts (auto)** — your opt-in daily digest (Phase 2) is the politer version; auto-email would be noise.

Phase 2: +3-4 days (status tags + status visibility + default view + view context note + lesson personal notes + search expansion). Now ~8-10 weeks.

## 42. Shared Modifications Browser — bidirectional Master/Personal flow (added)

The teacher asked for a "modified lessons/personalized lesson curriculum bank" so team members can see and copy each other's modifications. The request had three concrete asks:

1. Master lessons should signal when teammates have modified them
2. Modifications should be browsable with actions to replace the Master, add to the curriculum, or pull into one's own Personal
3. The collection of modifications should function as a bank teammates can browse

### What this connects to in the existing spec

This isn't entirely new — it overlaps significantly with three existing pieces:
- Master/Personal forking infrastructure (Phase 1A) — the modifications already exist as data
- Phase 2 bulk master-change approval (already designed for downward flow)
- Phase 3 Team Lesson Bank (just added last round) — the calendar-independent shared library

The actual addition is **making intra-team modifications discoverable and bidirectional**, plus reusing the Team Lesson Bank as the destination for "add to curriculum without replacing Master."

### Four product questions resolved

**1. Visibility default: opt-in with low-friction share** — teachers' modifications stay private unless explicitly shared, or unless they flip the global "share by default" setting. Respects work-in-progress privacy as the default; one-click share for trusted teams. Per-school admin master toggle to disable sharing school-wide.

**2. Promote-to-Master permission: self-service for `can_edit_master`** — consistent with the existing model where any teacher on a subject's team with editing permission can edit the Master directly. No special approval gate; the master-mode-entry banner sequence provides the safety beat.

**3. "Add to CORE" interpretation: add to the Team Lesson Bank, not as a Master variant** — variant systems add real complexity (sub-entities per section, version picker UI). The Team Lesson Bank already exists for calendar-independent shared lessons; routing "add to curriculum without replacing Master" there is cleaner.

**4. Phase placement: Phase 2** — sits alongside the bidirectional approval flow and Vivid theme launch. Trying to land it in Phase 1B would push beta-readiness; Phase 3 would arrive too late given how immediately useful it is.

### What lands in Phase 2 (§5.24)

**The badge.** Master lesson cards show `✏ N modified` when teammates have shared modifications. Hidden in Simple mode. Click opens the modifications panel.

**The panel.** Right-side slide-out with one row per shared modification: teammate, dates, auto-generated change summary, expand for diff view. Three actions:
- **Pull into my Personal** — copies the modification into the viewer's Personal copy
- **Promote to Master** — replaces the Master (requires `can_edit_master`, triggers master-mode banner, queues through bulk approval)
- **Add to Team Lesson Bank** — sends to the Phase 3 bank with original-author attribution

**The settings.**
- Per-teacher: "Share my modifications with the team by default" toggle (off by default)
- Per-lesson: "Share with team" toggle on every Personal copy with modifications
- Per-school: admin master toggle to disable sharing school-wide

**The data model.** New entity `SharedModification` — just a marker; the modification data lives in the existing Personal copy entities. No data duplication.

**Indexes:** `(personal_event_type, personal_event_id, share_status)` for fast "is this shared?" lookups; derived join on master_event_id for the badge count.

### UX details worth flagging

- **Badge counts "shared" modifications, not "modified" ones** — if 5 teachers have modified but only 2 shared, the badge reads `✏ 2 modified`. This is what makes opt-in actually work; teachers can verify their unshared modifications stay invisible.
- **Diff view uses the Phase 4 curriculum-diff renderer** — this is the first production use of it. Phase 4 keeps it for the standalone "what changed between Master versions" feature, but Phase 2 needs it for the modifications panel.
- **Original-author attribution preserved** — if Sarah pulls Tom's modification into her Personal and then shares it, the audit log preserves "Tom originally modified; Sarah is now sharing." The badge attribution is Sarah's; the lineage is Tom's.

### Why this matters competitively

The research turned up zero competitors that surface intra-team modifications back to the team:
- **Toddle/Atlas/ManageBac+** — institutional curriculum only; no personal-overlay visibility
- **Planbook** — copies that diverge forever; no flow back to Master
- **Common Curriculum/Chalk** — co-editing only; no fork pattern at all

Your spec already had the Master/Personal fork as a primary differentiator. Adding bidirectional flow turns it from "Git-style version control for curriculum" into something more powerful: **the team's best ideas get noticed and adopted instead of disappearing into individual planbooks.** That's a teaching story worth telling — and one no competitor can match without rebuilding their data model.

### Phase 2 cost impact

Phase 2 was at ~8-10 weeks after the Planbook absorption round. Adding the Shared Modifications Browser is ~1 week of focused engineering:
- Badge query + render: ~1 day
- Modifications panel UI + slide-out: ~1.5 days
- Three actions (Pull / Promote / Bank): ~1.5 days
- Settings (per-teacher + per-school): ~0.5 days
- Realtime invalidation: ~0.5 days
- Diff view (first use of Phase 4 renderer; counts as half here since it'll be reused): ~1 day
- Acceptance testing + edge cases: ~1 day

Phase 2 grows to ~9-11 weeks. Still wraps before summer 2027.

### Honest take

This is **quietly one of the most important Phase 2 features**. The Master/Personal fork was the headline differentiator from the beginning; this adds the bidirectional flow that makes the fork actually *productive* for the team rather than just safe. Without this, modifications stay in silos and the team learns slowly. With it, the planner becomes a quiet vehicle for spreading good ideas across the team.

The opt-in default is the right privacy posture. The self-service Promote-to-Master is the right trust model — schools that don't want teachers self-promoting can manage that through the existing `SubjectTeamMembership` permissions (don't give people `can_edit_master`). The Team Lesson Bank routing for "add to curriculum without replacing" is the right architectural answer — no variant system needed.
