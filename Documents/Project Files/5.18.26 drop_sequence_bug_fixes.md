# Drop-Sequence Bug Fixes — Collapse-on-Drag Pattern

**Status:** Bug report + fix instructions for Claude Code
**Context:** The drag and collapse-to-chip behavior is working correctly. The bugs are all in the **release/drop sequence** — between when the user releases the mouse and when the grid settles back into its expanded resting state.

---

## Bug 1: Cards Stay in Chip Mode Too Long After Drop (~1 second delay)

### Symptom
After the user releases the mouse, all peer cards remain visually compact for ~1 second before any re-expansion starts.

### Likely Cause
The `DragState` machine is stuck in either `dragging` or `dropping` phase. The `density` prop derived from it (`density = phase === 'idle' ? 'full' : 'compact'`) isn't flipping to `'full'` promptly.

Three common culprits:

1. **`setTimeout` delay is too long.** The original spec said:
   ```ts
   setDragState({ phase: 'dropping', activeId });
   setTimeout(() => setDragState({ phase: 'idle' }), 280);
   ```
   If the timeout is set to something like 1000ms (or accidentally chained with another delay), the density swap waits that whole time.

2. **Awaiting the network response before clearing dragState.** If `commitMove()` is `await`ed before `setDragState({ phase: 'idle' })`, then the visible un-collapse waits for the server roundtrip. This is the most common mistake when wiring optimistic updates.

3. **`dnd-kit` `dropAnimation` config is too long.** The `dropAnimation.duration` on `<DragOverlay>` defaults to 250ms but can be set higher; the active state lingers until that animation completes.

### Fix
```ts
onDragEnd={({ active, over }) => {
  // 1. Optimistically commit the move to local state IMMEDIATELY — synchronously
  if (over) commitMoveLocally(active.id, over.id);

  // 2. Fire the network call but DO NOT await it before clearing dragState
  if (over) persistMoveToServer(active.id, over.id).catch(revertMove);

  // 3. Clear dragState right away — re-expansion should start within one frame
  setDragState({ phase: 'idle' });
}}
```

**Eliminate the `'dropping'` phase entirely** unless there's a specific intermediate animation that needs it. Going straight from `'dragging'` → `'idle'` is cleaner. The drop animation of the floating overlay itself is handled by dnd-kit's `<DragOverlay dropAnimation={...}>` and runs in parallel with the re-expansion — they don't need to be sequenced.

Set `dropAnimation` to ~220ms max:
```tsx
<DragOverlay dropAnimation={{ duration: 220, easing: 'cubic-bezier(0.2, 0, 0, 1)' }}>
```

## Bug 2: All Cells Go Blank for ~400ms Mid-Transition

### Symptom
Between chip mode disappearing and full-mode cards rendering, every cell appears empty — showing the "Drag a lesson here or click +" empty-cell placeholder. Pagination indicators ("1 of 2", "1 of 3") are visible as floating chrome with no content beneath them. Lasts roughly 12 frames at 30fps (~400ms).

This means the cell components are momentarily seeing **empty `lessons` arrays**, not just transitioning size.

### Likely Cause
The state update on drop is going through an intermediate "all cells empty" state. Three possible patterns that produce this:

1. **Clear-then-repopulate update pattern.** Code that does:
   ```ts
   setLessonsByCell({});             // wipes everything
   setTimeout(() => {
     setLessonsByCell(reorderedData); // repopulates
   }, 0);
   ```
   This is a common antipattern when developers want to "force a re-render."

2. **Key change on the SortableContext or CardGrid.** If the React `key` on the container changes during the drop (e.g., a timestamp-based key, or a key derived from the dragState), React unmounts and remounts every child, which briefly renders the empty state before the new data flows in.

3. **`SortableContext items` array briefly empty.** If `items={isDragging ? [] : sortedIds}` or similar conditional is in play, dropping toggles `isDragging` and the array goes empty for one render cycle.

### Fix

**Audit the drop handler for these patterns:**

```ts
// ❌ WRONG — clears state then re-sets
onDragEnd={() => {
  setLessonsByCell({});                              // BUG: wipes all
  setLessonsByCell(reorderArray(prev, source, dest)); // never run synchronously after this
}}

// ❌ WRONG — key change
<SortableContext key={`grid-${dragState.phase}`} items={...}> // BUG: key changes on phase

// ❌ WRONG — items conditional
<SortableContext items={dragState.phase === 'idle' ? ids : []}> // BUG: empty array

// ✅ CORRECT — single atomic update, stable key, stable items array
onDragEnd={({ active, over }) => {
  if (!over || active.id === over.id) {
    setDragState({ phase: 'idle' });
    return;
  }
  setLessonsByCell(prev => moveLesson(prev, active.id, over.id)); // single atomic swap
  setDragState({ phase: 'idle' });
}}

<SortableContext items={allLessonIds}>  {/* stable across drag states */}
```

**Make the move function pure and atomic:**
```ts
function moveLesson(
  cells: Record<string, Lesson[]>,
  lessonId: string,
  targetCellId: string,
): Record<string, Lesson[]> {
  const sourceCellId = findCellContaining(cells, lessonId);
  if (!sourceCellId) return cells;
  const lesson = cells[sourceCellId].find(l => l.id === lessonId);
  if (!lesson) return cells;

  return {
    ...cells,
    [sourceCellId]: cells[sourceCellId].filter(l => l.id !== lessonId),
    [targetCellId]: [...(cells[targetCellId] ?? []), lesson],
  };
}
```

One call, one state update, no intermediate empty state.

## Bug 3: Text Gets Massively Stretched During Re-Expansion (~3× vertical)

### Symptom
For a brief window during the chip → full re-expansion, every card's text content appears at roughly 3–4× its normal vertical size — distorted, tall, narrow letters. Lasts ~5 frames (~165ms). Clearly visible in the video on words like "Equivalent fractions warm-up" and "Theme mapping."

### Root Cause (very likely)
**`framer-motion`'s `layout` prop uses the FLIP animation technique with `transform: scale()`.** When an element's height changes from 28px (chip) to 88px (full card) — a 3.1× change — framer-motion snapshots the old size, applies the new size to the DOM, then animates `transform: scale(1, 3.1)` → `scale(1, 1)` over the transition duration. The TEXT inside is also visually scaled by that transform, which is what produces the stretched appearance.

This is *correct behavior* for framer-motion's `layout` when used on a container whose internal content is supposed to scale proportionally. It's *wrong* for cards where the content should reflow at the new size, not scale.

### Fix — Option A (recommended): Replace `layout` with explicit height animation

Don't use framer-motion's `layout` prop for the density transition. Animate `height` directly via `AnimatePresence` + variants, which causes a natural layout reflow rather than a transform-based scale.

```tsx
function Card({ id, density, data, floating }) {
  return (
    <motion.div
      // ❌ REMOVE: layout
      // ❌ REMOVE: layoutId={`card-${id}`}
      className={cn(
        'rounded-lg border bg-white overflow-hidden',
        floating && 'shadow-xl ring-2 ring-purple-300 scale-[1.02] opacity-95',
      )}
    >
      {/* Header — always rendered */}
      <HeaderRow density={density} data={data} />

      {/* Rich content — height animates naturally because we animate height, not scale */}
      <AnimatePresence initial={false}>
        {density === 'full' && (
          <motion.div
            key="rich-content"
            initial={{ height: 0, opacity: 0 }}
            animate={{ height: 'auto', opacity: 1 }}
            exit={{ height: 0, opacity: 0 }}
            transition={{
              height: { duration: 0.2, ease: 'easeOut' },
              opacity: { duration: 0.15, ease: 'easeOut' },
            }}
            style={{ overflow: 'hidden' }}  // critical — prevents content bleed during animate
          >
            <RichContent {...data} />
          </motion.div>
        )}
      </AnimatePresence>
    </motion.div>
  );
}
```

Why this works: animating `height: 0 → auto` causes the browser to reflow content at full size on each frame. Text is rendered at its target font-size from frame one — only the container's height grows. No `transform: scale()` involved, no stretching.

### Fix — Option B: Keep `layout` but constrain it to position-only

If you want to keep `layout` for position animations (e.g., when a card moves to a different cell), use `layout="position"` instead of `layout` or `layout="size"`:

```tsx
<motion.div layout="position" transition={{ duration: 0.2, ease: 'easeOut' }}>
```

This tells framer-motion to animate only x/y translation, never size. Combine with Option A's AnimatePresence pattern for the height change inside.

### Fix — Option C: Force text to its target size

If for some reason you need to keep `layout` as-is, you can counter-scale the text inside:

```tsx
<motion.div layout>
  <motion.div style={{ transform: `scale(1, ${1 / heightRatio})` }}>
    {textContent}
  </motion.div>
</motion.div>
```

**Don't do this.** It's brittle, and the math has to be re-derived on every render. Use Option A.

## Bug 4 (minor): Dropped Card Retains Floating Styling Briefly

### Symptom
After release, the dropped card keeps its purple ring + shadow + slight scale-up ("floating chip" style) for an extra moment before settling.

### Cause
The `<DragOverlay>` portal-mounted clone unmounts on drop, but if the underlying card (the real one in the grid) inherits the `floating` style from a stale prop, it briefly looks ghosted too.

### Fix
Make sure the `floating` prop on `<Card>` is **only ever true inside `<DragOverlay>`**, never on the in-grid card. The real card in the grid should never receive `floating={true}`:

```tsx
<Card id={lesson.id} density={density} data={lesson} />  {/* never floating */}

<DragOverlay>
  {dragState.phase === 'dragging' && (
    <Card id={dragState.activeId} density="compact" data={getLesson(dragState.activeId)} floating />
  )}
</DragOverlay>
```

If you're using `useSortable` and applying an `isDragging` style to the in-grid card during drag, instead of `floating` use a separate "ghost" style (~40% opacity, no ring, no shadow) so the original cell shows the placeholder without looking like the cursor's chip.

```tsx
const { isDragging } = useSortable({ id });
return (
  <motion.div
    className={cn(
      isDragging && 'opacity-40',  // ghost, not floating
    )}
  >
```

## Summary of Changes — Hand This to Claude Code

In priority order:

1. **(Bug 3, highest impact)** Replace `motion.div layout` on Card with the AnimatePresence + height-auto pattern from Fix 3A. This eliminates the text stretching.

2. **(Bug 2, second priority)** Audit `onDragEnd` for the three antipatterns in Fix 2 (clear-then-repopulate, key changes, conditional items). Make the state update single, atomic, and synchronous using the `moveLesson` pure function pattern.

3. **(Bug 1, third priority)** Remove `await` from any persistence call in `onDragEnd` before `setDragState({ phase: 'idle' })`. The optimistic local update + dragState clear must be synchronous. Network calls happen in parallel, not before.

4. **(Bug 4, cleanup)** Verify that `floating={true}` is only ever passed to the Card inside `<DragOverlay>`, never to the in-grid card. Use an opacity-based ghost style for the in-grid placeholder if needed.

**Verification after fixes:**
- Drop → first re-expansion frame: should be **< 30ms** (one render cycle).
- Total drop → settled state: should be **< 300ms**.
- No frame should show all-empty cells.
- No frame should show stretched/scaled text.
- Open DevTools → Performance tab → record a drop. The transition should show one synchronous state update followed by a single animation, not a sequence of state updates with gaps.
