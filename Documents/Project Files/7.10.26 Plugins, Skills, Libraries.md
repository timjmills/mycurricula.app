## How activation works (answering "do they always run?")

- **The always-on layer is the rule text**, injected every session via
  `~/.claude/CLAUDE.md` (Claude) and `~/.codex/AGENTS.md` (Codex), both pointing
  at `~/.agents/DESIGN-STACK.md`. That text — consult the stack on UI work,
  project tokens outrank skills, never auto-add deps — is always in context.
- **The 12 skills are on-demand by design.** Each exposes a name + description;
  the agent loads the full body only when the task matches. This is correct, not
  a limitation: loading all 12 into every session would waste context and make
  both agents *worse* at non-UI tasks (e.g. `gsap-scrolltrigger` firing during a
  Supabase migration). They fire when they fit.
- **Per-repo nicety:** `impeccable` runs without a `PRODUCT.md` but is better
  with one. `/impeccable init` captures product context once — run it when
  starting real design work, and review what it writes (it will want to describe
  a design system this repo already governs via `BUILD_STANDARD.md`).

## The governing rules (precedence)

1. **Consult the design stack on every UI change.** At minimum `impeccable`;
   reach for `ui-ux-pro-max` / `web-design-guidelines` for craft. Skip only for
   non-UI work.
2. **The project's own design contract OUTRANKS every skill.** `BUILD_STANDARD.md`
   + `app/tokens.css` + `CLAUDE.md` §4 win over anything a skill proposes. Never
   import a skill's palette (the 161), a font pairing (the 57), or a component
   kit. Skills supply *reasoning and anti-patterns*; the project supplies *the
   values*.
3. **Match the skill to the surface.** `design-taste-frontend` = marketing only.
   Product UI / dashboards / tables → `impeccable` + `ui-ux-pro-max`.
4. **Never auto-add a dependency.** The `gsap-*` skills are knowledge, not a
   licence to add GSAP. Adding a library is an explicit, per-project decision.
5. **Skills advise; they have no authority over safety.** They never override a
   review gate, a security rule, a refusal, or the user.

## Libraries (PR #59) — approved runtime deps + rule carve-out

`three@0.185.1`, `gsap@3.15.0`, `@react-three/fiber@9.6.1` are now in
`package.json`, **pinned exact**, and permitted by revised `CLAUDE.md` rules:

- **§3 stack table** — opt-in Motion and 3D/WebGL rows added.
- **§4 motion** — parallax allowed **only** on marketing/landing surfaces via
  GSAP ScrollTrigger; **never** in the planner (Weekly/Daily/Year), which stays
  calm and functional. GSAP/3D are opt-in and surface-scoped; plain CSS
  transitions / the View Transition API remain the default. All output still
  obeys `prefers-reduced-motion`, WCAG AA, and the token system.
- **§6 dependency ban** — explicit carve-out for these three. **Does NOT reopen
  UI-component kits** (shadcn/ui, Radix, MUI stay banned); components remain
  bespoke against the tokens.

**Status:** libraries are available but **not imported anywhere yet → 0 current
bundle impact**. First real usage should be its own PR with the live QA + all-six-
theme sweep.

### Two gotchas recorded for future work

1. **R3F v9 ↔ React 19 peer coupling.** `@react-three/fiber@9.6.1` requires
   `react >=19 <19.3`; the repo resolves 19.2.6 (fits, barely). The three new
   deps are pinned **exact** so a future `npm install` can't drift React to 19.3
   and break the peer. Upgrading React past 19.2.x requires checking R3F's peer
   range first.
2. **Windows lockfile prune.** `npm install` (and `--package-lock-only`) on the
   Windows dev host silently drops 5 platform-optional entries Linux CI needs
   (`@emnapi/core`, `@emnapi/runtime`, `@csstools/css-parser-algorithms`,
   `@csstools/css-tokenizer`, vitest's nested `esbuild`). Fix: snapshot the lock
   before install, then merge the dropped entries back so the result is a strict
   **superset** (0 removed). Never "clean-regenerate" the lockfile on Windows.

## Fabricated / mis-attributed items (do not chase these again)

These recur in AI-generated tool lists and are **not real**:

- **`~/.clauderc` with a `systemPrompt` key** — not a Claude Code mechanism.
  Global always-on context is `~/.claude/CLAUDE.md`.
- **`claude_desktop_config.json` for Claude Code MCP** — that's Claude *Desktop*.
  Claude Code uses `claude mcp add`.
- **`@21st-dev/magic-mcp`** — 404 on npm. Real package is **`@21st-dev/magic`**.
- **`vercel-labs/agent-skills --skill frontend-design`** — that repo's 9 skills
  don't include `frontend-design`; the real one is `web-design-guidelines`.
- **"OpenAI Native Product Design Plugin"** — the `openai-curated` marketplace
  has 26 plugins; no such entry.
- **"Open Design" framework** — doesn't exist; describes the
  playwright/chrome-devtools screenshot loop already installed.
- **`claude-preview` plugin** — doesn't exist (0 of 1,810 catalog entries).
  Capability covered by `chrome-devtools` MCP.
- **Official Anthropic "Design Review Skill"** — no such thing; the real one is
  third-party and ships no `SKILL.md`.
- **"Claude Design"** IS real (Anthropic Labs, hosted web workspace) — but a
  product to use in the browser, not an installable plugin. Reachable via the
  `DesignSync` tool + `/design-sync`.
