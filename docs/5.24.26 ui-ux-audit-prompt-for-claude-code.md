# UI/UX audit — mycurricula.app

A two-phase audit of the deployed product against both the source code
and the live runtime. The goal is a single document that captures
**what's broken** (Phase 1) and **what could be meaningfully better**
(Phase 2), prioritized so I can triage in one sitting.

## Context

mycurricula.app is a curriculum planner for K–12 teachers. The
audience is **busy, often interrupted, often on mediocre school
wifi and aging laptops**. Cognitive load, speed, and never losing
work matter more than visual polish. Many teachers print views.
Many teach multiple classes / sections / preps. Time-of-day, day-of-
week, and week-of-year are load-bearing concepts in every view.

This is the v1 of the product. We're past "does it work" and into "is
it actually usable in a Tuesday-morning chaos scenario."

## Methodology

Run the audit in two distinct phases. Don't blur them — defects and
improvements are different conversations with different urgency.

**Phase 1 — Defects.** Find things that are broken, missing, or wrong.
Tag each with a severity. These are bugs in the strict and the loose
sense: literal JS errors *and* the kind of "this should have a confirm
dialog but doesn't" omissions.

**Phase 2 — Improvements.** Find things that work but could be
materially better. Tag each with an impact-vs-effort score. These are
not bugs; they're opportunities. Be opinionated. If a workflow is
fundamentally clunky, say so — don't just polish around the edges.

For both phases, use **two sources in parallel**:

1. **Source code** — semantic HTML, ARIA, focus order, Tailwind drift,
   component reuse, error boundaries, loading state coverage, JSX
   that hints at missing empty/error states.
2. **Live runtime via the Claude bypass** — actual rendered HTML on
   the production deploy, with the real Supabase data behind it. Some
   issues only surface when a real lesson has a 280-character title,
   when "this week" straddles a school holiday, when the to-do list
   is empty, when the network is slow. The bypass lets you see those.

## How to authenticate against the live site

The Claude bypass is wired in production. Three ways to use it — pick
based on the tool you're reaching for:

**1. Bearer header (best for curl, scripted fetches):**

```bash
TOKEN="$(grep '^CLAUDE_BYPASS_TOKEN=' .env.local | cut -d= -f2-)"
curl -L \
  -H "Authorization: Bearer $TOKEN" \
  -H "User-Agent: claude-code-audit/1.0" \
  https://mycurricula.app/weekly
```

Read the token from `.env.local` (gitignored). Don't paste it inline
in commands you'll log or commit, and don't echo it to a markdown
file.

**2. URL query parameter (best for WebFetch, which doesn't pass
custom headers):**

```
https://mycurricula.app/weekly?claude=<token>
```

The worker strips `?claude=` from the URL after minting the session,
so the rendered page URL won't echo the token back.

**3. Cookie-redirect endpoint (best for any flow where you want a
real Supabase session cookie set on a browser, then crawl):**

```
https://mycurricula.app/auth/claude-login?token=<token>&next=/weekly
```

For this audit, mostly use #1 — you have Bash and curl. Use #2 only
if WebFetch is genuinely cleaner for a specific check.

### Mobile UA spoofing

For mobile-responsive checks, add a mobile user-agent so the server
delivers the mobile variant (where applicable):

```bash
curl -H "Authorization: Bearer $TOKEN" \
  -H "User-Agent: Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 like Mac OS X) AppleWebKit/605.1.15 (KHTML, like Gecko) Version/17.0 Mobile/15E148 Safari/604.1" \
  https://mycurricula.app/weekly > /tmp/weekly-mobile.html
```

Diff the mobile and desktop HTML to spot responsive differences in
delivered markup.

## Routes to audit

Priority order, depth-first. If you run out of time, finish a route
before starting the next.

| Priority | Route | Notes |
|---|---|---|
| 1 | `/weekly` | Primary view. Most user time. Audit deeply. |
| 2 | `/daily` | Single-day focus. Teachers use this morning-of. |
| 3 | `/schedule` | Schedule editor. Mistakes here ripple everywhere. |
| 4 | `/catch-up` | "What I haven't covered yet." Failure-state heavy. |
| 5 | `/curriculum` | Long-form planning. Long page; scroll/anchor matters. |
| 6 | `/yearly` | High-level zoom. Print-friendliness matters. |
| 7 | `/auth/claude-login` | Bypass entrypoint. Just verify 307 + cookies. |
| 8 | `/` and `/login` | Unauthenticated SSO landing. Check wrong-domain errors. |

For each, also probe edge cases:

- **Empty state** — a panel with zero items. Pick a future week if
  the current week has data.
- **Overflow state** — the longest titles, longest descriptions, the
  most items in a panel.
- **Loading state** — throttle the network in DevTools mentally; if
  the source has no Suspense fallback or skeleton, flag it.
- **Error state** — what does a 500 from Supabase look like? Look at
  the source for try/catch and error boundaries.
- **Holiday / off-week** — a week straddling a holiday or summer
  break. Date math is a common bug source.

# Phase 1 — Defects

For each category below, list every defect you find. The goal is
comprehensive coverage, not brevity. If a category has zero findings,
say so explicitly.

## 1.1 Functional bugs

Things that don't work or work wrong.

- Buttons that do nothing
- Links to non-existent routes / 404s
- Save-then-reload data loss
- Off-by-one in dates, weeks, periods, sections
- Wrong data displayed (mixed up classes, swapped subjects, etc.)
- State that survives across users/sessions when it shouldn't
- Optimistic-update inconsistencies
- Race conditions when typing-then-clicking-too-fast

## 1.2 Accessibility blockers

WCAG 2.2 AA as the floor. Anything that fails it is a defect.

- Heading hierarchy broken (skipped levels, multiple h1s, no h1)
- Missing landmark regions (`<header>`, `<nav>`, `<main>`)
- Icon-only buttons without `aria-label`
- Form controls without associated `<label>` (label-by-placeholder
  doesn't count)
- Focus order doesn't match visual order
- Focus rings removed by `outline: none` with no replacement
- Color contrast below 4.5:1 for body, 3:1 for large text (inferable
  from Tailwind class combinations)
- Color used as the only signal (red/green status with no text/icon)
- `<div onclick>` instead of `<button>`
- Modals that don't trap focus or restore focus on close
- `<img>` without alt (decorative needs `alt=""` explicitly)
- ARIA live regions missing for async status updates (saved/error)
- Keyboard-only flows that break (drag-and-drop with no keyboard
  alternative, tabbable elements hidden under overlays, etc.)

## 1.3 Visual / layout bugs

- Text clipped, overflowing containers, or wrapping awkwardly
- Z-index conflicts (tooltips behind modals, dropdowns clipped by
  overflow:hidden)
- Misaligned baselines, inconsistent padding, snap-to-grid violations
- Broken sticky/fixed positioning on scroll
- Print stylesheet missing or wrong (teachers print)
- Dark-mode bugs if dark mode exists

## 1.4 Missing states

For every async data fetch and every form, there should be states
for: idle, loading, success, empty, error. Missing any is a defect.

- Loading states (skeleton vs spinner vs nothing — anything is better
  than nothing)
- Empty states (zero-item lists, fresh accounts, new weeks)
- Error states (network error, 5xx, validation error)
- Success confirmations (especially destructive operations and saves)
- "Last saved" / "saving…" feedback for autosave

## 1.5 Form bugs

- Required fields not marked as required
- Validation errors not announced to screen readers
- Validation errors that disappear on next keystroke before being read
- Submit button enabled while invalid
- Submit button disabled in a way that obscures *why*
- Lost data on validation failure (the form clears or scrolls away
  from the user's place)
- No autofocus on the obvious first field
- No `enter`-to-submit on single-line forms
- No keyboard escape from modal forms

## 1.6 Navigation bugs

- Back button breaks state
- Deep links don't work (loading directly to `/weekly?week=2026-W12`
  should put you on that exact view)
- Breadcrumbs missing on deep routes
- "Today" button doesn't actually take you to today
- Active-nav state wrong or missing
- Dead-end pages (no way out except back button)

## 1.7 Data integrity

- Stale data after navigating away and back
- Time zone mistakes (server in UTC, user in EST, lesson at 8:10am
  showing as 1:10pm)
- DST transitions
- Week-of-year edge cases (Jan 1 falling mid-week, ISO week vs US week)
- Off-by-one period numbering
- Standards (CCSS codes etc.) mismatched to lessons

## 1.8 Performance defects

Not optimization opportunities — actual bugs.

- N+1 fetches visible in the source
- Bundle splitting failures (loading every route's JS upfront)
- Images served at 4× the displayed size
- Blocking sync scripts in `<head>`
- Layout shift (CLS) from late-loading content
- Large lists rendered without virtualization
- Memory leaks from event listeners not cleaned up in useEffect

## 1.9 Mobile / responsive bugs

- Horizontal scroll at 375px (iPhone SE) width
- Touch targets smaller than 44×44px
- Hover-only interactions with no tap alternative
- Modal dialogs that don't fit on mobile
- Tables that become unreadable below 768px with no fallback

## 1.10 Browser / device compatibility

- Safari-only bugs (different `<input type="date">` rendering, sticky
  positioning quirks, backdrop-filter support)
- Firefox-only bugs (CSS subgrid, container queries)
- iOS-specific (rubber-band scroll, virtual keyboard pushing content)
- Old Chrome on school Chromebooks (often 2 versions behind)

## 1.11 Copy bugs

- Typos
- Inconsistent terminology ("class" vs "section" vs "period")
- Microcopy that lies ("Saved!" when it failed, "1 item" pluralized
  wrong)
- TODO / FIXME / placeholder text shipped to production
- Tooltips that just repeat the button label
- Error messages written for developers, not teachers
- Hardcoded strings that should be translatable

## 1.12 Auth-flow bugs

- Session expiry mid-edit loses unsaved work
- Wrong-domain SSO login: cryptic Google error vs friendly explanation
- Logged-out users hitting a deep link: lose the destination after
  login (no `?next=` preservation)
- Multi-tab session conflicts

## 1.13 Console / network errors

In source: search for `console.error`, `console.warn` in production
code paths. Note any that fire on normal flows.

On live: harder to check from curl, but inspect the inline `<script>`
blobs in delivered HTML for embedded error/warning strings.

# Phase 2 — Improvements

Things that work, but could be better. Use an impact × effort matrix:

- **Impact**: high (changes the daily teacher experience) / medium
  (noticeable improvement) / low (nice-to-have)
- **Effort**: small (< 1 day) / medium (1–3 days) / large (> 3 days
  or a structural change)

Be opinionated. If a workflow is fundamentally clunky and would
benefit from a redesign, say so — don't just suggest a tooltip on the
existing broken thing.

## 2.1 Ease of use

- Reduce clicks for common workflows (how many clicks to add a
  lesson? to move a lesson? to copy a week?)
- Keyboard shortcuts for power users (Cmd+S, J/K navigation, /
  search, ? help)
- Undo/redo for destructive actions
- Bulk select + bulk operations (apply to all Mondays, copy to all
  weeks)
- Better defaults (new lesson defaulting to the current day/period)
- Smart suggestions (auto-fill from last week, from same day last
  year, from a colleague's template)
- Drag-and-drop with snap-to-grid + visual feedback
- Inline edit (double-click to rename) instead of opening a modal

## 2.2 Functional additions

- Search across all curriculum content
- Filters on long lists (subject, unit, week)
- Saved views / pinned filters
- Calendar export (iCal feed for personal calendar integration)
- Print-optimized stylesheet for weekly view (no nav, no chat, just
  the schedule)
- PDF export
- Lesson templates with placeholders
- Co-editor presence (see who else is viewing)
- Comments / notes on individual lessons
- Standards-coverage report (show me which CCSS I haven't hit yet
  this year)
- Substitute-teacher view (read-only printable "here's what to do
  tomorrow")

## 2.3 Compatibility expansion

- PWA install prompt + offline-first for read-only access
- Service worker caching of the current week's data
- Print stylesheet
- High-contrast mode
- Reduced-motion mode (`prefers-reduced-motion`)
- Dark mode (if not present)
- Touch-optimized variants for tablets in the hands of students /
  subs

## 2.4 Information density + visual design

- Better defaults for views (collapsed vs expanded panels)
- Reduce visual noise on the weekly grid
- Stronger visual hierarchy (what's the most important thing on
  this page?)
- Better use of color to encode status without becoming a unicorn
  vomit (lots of subject colors can drown out the *task* status)
- Sticky column headers on long tables
- Better empty states that teach the next action

## 2.5 Onboarding + help

- First-run experience: 60-second tour of the weekly view
- Contextual tooltips on hover for novel UI patterns
- Sample data for new accounts so they're not staring at an empty grid
- In-app docs / help drawer (no separate site to context-switch to)
- "What's new" changelog for returning users
- Keyboard-shortcut overlay (press ? to see)

## 2.6 Performance optimization

- Prefetch the next likely route (`/weekly` → `/daily`)
- Image optimization (next/image, AVIF/WebP, blur placeholders)
- Code-split per route
- Cache the curriculum/standards lookup tables
- Optimistic UI for moves, edits (don't wait for server roundtrip)
- Background sync for autosave instead of synchronous

## 2.7 Accessibility enhancements

Beyond WCAG minimums.

- Skip-to-main-content link
- Announce route changes to screen readers
- Improve focus visibility (custom focus ring per theme)
- Reduce reliance on color for state (icons + text)
- High-contrast theme
- Adjustable text size (respect `rem` everywhere)

## 2.8 Mobile-first wins

- Bottom sheet patterns for modals on mobile
- Swipe gestures (swipe week, swipe lesson)
- Pull-to-refresh on the daily view
- Larger touch targets without ballooning desktop
- Save data on slow networks (skeleton-first, hydrate later)

## 2.9 Personalization

- Per-user view preferences (start week on Sun vs Mon, period numbering)
- Theme picker
- Default landing page
- Subject color overrides
- Hidden subjects (if I don't teach Latin, don't show it)

## 2.10 Collaboration

- Share-a-week link (read-only)
- Comments on lessons
- Real-time co-editing of a single lesson (operational transform
  or CRDT)
- Per-section presence indicators
- Activity log per week ("Sarah moved Lit Circles to Wednesday")

## 2.11 Workflow + automation

- Recurring lesson templates ("every Monday, period 2, do X")
- Auto-rollover of unfinished lessons to next week
- Smart catch-up scheduling (auto-distribute the 28 uncovered items)
- Calendar conflict detection (period 2 has two lessons assigned)
- Lesson-from-standard generator (pick a standard, get a starter
  lesson)

## 2.12 Analytics + observability for *you*

- Error tracking (Sentry or similar) — flag if not wired
- Real User Monitoring (Core Web Vitals)
- Feature-usage telemetry so you know what's actually used
- In-app feedback widget so teachers can flag issues without leaving

# Constraints

- **Audit-only branch.** Don't commit changes to source files during
  the audit. The deliverable is the report. We'll triage and ticket
  the work separately.
- **Don't write the bypass token anywhere except `.env.local`** —
  not in commit messages, not in markdown, not in code comments, not
  in shell history. Read it from the env var.
- **Rate-limit yourself** to ~30 bypass requests per minute.
- **Skip the known audit-log schema mismatch.** The
  `claude_access_log` table currently has columns `success`/`path`
  while the middleware writes `ok`/`pathname`, so your bypass calls
  silently fail to log. That's a separate fix, not part of this audit.
- **Don't audit the bypass itself.** That's a security-and-auth
  conversation, not a UX one.
- **Be opinionated, but back it.** A finding like "this view is
  cluttered" is useless. "This view shows 14 distinct color tokens
  in a 12-card grid, which competes with the task-status colors and
  buries the most important signal (which lessons are not done) — see
  WeeklyGrid.tsx:84-120" is useful.
- **Time-box yourself**: 3 hours for Phase 1, 2 hours for Phase 2.
  If a route is taking longer, summarize and move on.
