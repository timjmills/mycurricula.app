## 1. Findings I confirm

### Codex #1 — App chrome doing too much
**Confirmed.** Codex names this directly; I touched it lightly in my §2 #10 ("Information density at 1280px") but only as a density observation, not a recommendation to move route-specific actions OUT of the top bar. Codex's framing — "make the top bar answer only global questions: Where am I? Personal or Master? Where to next?" — is sharper. **I missed the prescription**, not the symptom.

### Codex #2 — Weekly right rail unreachable below 1280px
**Confirmed but partially mitigated by recent work.** Codex's finding is real — `WeeklyShell.module.css` hides the rail at ≤1280 with comments pointing at a "future affordance." What Codex couldn't see (lane work shipped same day): Lane CC + Lane FA shipped a `GlobalRail` + `RightIconRail` that surfaces equivalent toggles (Resources/Comments/To-do) on every planner route including phone. **However**, the right-rail *content* (the actual Resources panel, etc) still mounts at /daily only — so the icon is reachable but the panel isn't on /weekly tablet/phone. My audit missed this entirely. Codex is right that an alternate path is needed; the Lane CC icon is half the fix, not the whole fix.

### Codex #3 — Daily panes can resize to 40px
**Confirmed and embarrassing — I missed this.** The Daily splitter accepts any width down to 40px, which lets a teacher create a layout where the lesson list is unreadable. My §2 #4 covered the *splitter-disappears-at-≤720px* problem but not the *pane-shrinks-to-broken-at-1280px* problem. Codex's two-layer solution (meaningful readable minimum + separate collapsed state) is correct. Add to my fix list.

### Codex #4 — Curriculum nested buttons (a11y bug)
**Confirmed; I missed entirely.** `<button>` inside `<button>` in `components/subject/SubjectView.tsx` is invalid HTML and breaks keyboard + screen-reader behavior. The Curriculum view has the same pattern uncorrected. This is a **real accessibility blocker**, not a UX nit. Add to fix list above my §2 #1.

### Codex #5 — Yearly disabled "coming soon" actions
**Confirmed with caveat.** My §3-G1 noted the SOON badge needs contrast check. Codex makes the stronger argument: hide them or move to a roadmap area. Lane D explicitly disabled-with-tooltip'd these for "onboarding-visible scope" — that was a deliberate choice. Both positions have merit; see §4 (real disagreements).

### Codex #6 — Placeholder controls everywhere
**Confirmed; I missed the systematic scope.** Lesson Detail menu/full-screen stubs, Lesson Flow's "Edit lesson flow / template," Section Resources' "Show more," Resource Composer's Search tile, Schedule Pane's disabled options — Codex enumerated 6 surfaces. My audit was hover-coverage-focused; I never stepped back to ask "do these visible controls actually work?" This is a meaningful gap in my method.

### Codex #7 — Catch-up bulk actions don't work
**Confirmed and serious.** The handler for "Add all to to-do" only clears the selection — it doesn't create to-dos. Carry-over marks rows but records no destination. **These are bugs, not UX nits.** My §3-A2 talked about *teaching* the Catch-up concept; Codex caught that the actions inside Catch-up are broken. Add to fix list.

### Codex #8 — School week hardcoded Sun-Thu in Schedule

### Codex #13 — Mobile Curriculum subject switching

### Codex #14 — Master/Core/Personal terminology inconsistent

See §4 — there's a real principle disagreement here about which pair wins.

## 2. Findings I dispute

### Codex #5 — "Hide disabled actions or move to a roadmap area"
**Disputed in principle.** Codex's prescription ("hide unless it explains a critical upcoming workflow") conflicts with Tim's explicit direction earlier in this session: visible disabled controls + tooltip explanation were a *deliberate* onboarding choice (Lane D commit `be87ec7`). Tim wanted teachers to see the planned scope. The risk Codex names (user clicks a disabled control, frustration) is mitigated by the existing `aria-disabled` + opacity-0.5 + onboarding-voice tooltip pattern. **My position:** the current treatment is correct; if anything, the missing piece is the SOON badge contrast (my §3-G1), not hiding the controls. This is a real principle disagreement (see §4).

### Codex Rule #1 — "Every interactive control needs a tooltip is too broad"
**Disputed.** Codex's argument: "Save" → "Save" is noise, trains users to ignore tooltips. **My counter:** the CLAUDE.md §4 rule explicitly mandates *onboarding voice* — "Save your edits to this lesson — the team's Master and your Personal stay independent until you toggle Master mode" is NOT noise. It teaches the consequence. The risk Codex names is real *only if* the tooltip text is lazy. The rule isn't "label every button with its own name"; it's "explain what each control accomplishes in context." If the rule is honored in voice (which CLAUDE.md §4 explicitly says), the noise concern dissolves. **The fix is voice discipline, not rule scope.** That said, Codex's "ambiguous, icon-only, disabled, destructive, high-consequence, or non-obvious" carve-out is a useful concretization. See §4.

### Codex Rule #2 — "Match /weekly exactly"
**Partially disputed.** Codex says "exactly" is too restrictive. Actually CLAUDE.md and BUILD_STANDARD.md don't say "match exactly" verbatim — they designate /weekly as the *canonical reference* for the design vocabulary, which is exactly what Codex proposes ("Use Weekly as the canonical design vocabulary, but adapt layout to the surface's job"). **My position:** the rule as actually written is fine; Codex is fighting a stricter version of the rule than exists. Worth verifying word-for-word before "softening."

### Codex Rule #4 — "Color carries information, never decoration"
**Partially disputed.** Codex softens the rule to allow neutral tinting + brand accents + hover states. **My position:** the rule's *intent* is that *informational color* (subjects, status) doesn't compete with *decorative color*. Neutral tinting in CLAUDE.md's ink scale (--ink-50 through --ink-900) is *already* decorative-neutral and not flagged as a violation anywhere. Codex is softening a rule that wasn't blocking anything. The current implementation already does what Codex proposes. Verify before changing the rule.

### Codex #2 weekly-rail-fallback claim — "no alternate path"
**Partially disputed (timing).** Codex audited before Lane FA/CC shipped GlobalRail + RightIconRail. The Comments/Resources/To-do icons ARE now reachable on every route. What's still missing is the *panel content* on non-/daily routes — but the *trigger* is no longer hidden. Codex's framing implies the rail is fully unreachable; it's the panel mount that's the gap, not the trigger.

## 4. Real disagreements worth surfacing to Tim

These are principle disagreements, not emphasis or scope differences. Both positions are defensible; Tim picks.

### Disagreement 1 — Tooltip-everywhere rule scope

**Codex:** require tooltips only where the visible label *doesn't* explain the consequence/state/destination. "Save" should not need a tooltip.

**Claude:** the rule already mandates *onboarding voice* (CLAUDE.md §4) — "Save your edits to this lesson; the team's Master copy is unaffected" *is* the tooltip on Save. The rule isn't "label every button"; it's "explain consequence." The problem Codex names exists only if voice is lazy. Fix the voice, keep the rule.

**Both sides fairly:** Codex protects against tooltip fatigue. Claude protects against agents skipping the explanation because the button label "looks self-evident." First-time teachers benefit from explanation; experienced teachers will skip-read regardless.

### Disagreement 2 — Master / Core / Personal vocabulary

**Codex:** pick "Personal" + "Team Curriculum" everywhere. "Master" is git-jargon, opaque to teachers.

**Claude (implicit in my audit):** keep "Master / Personal" — it's the existing convention, anchored in CLAUDE.md §2's "git-style forking" model, and changing it requires editing the planning_document, every dialog, every context menu, and every code comment.

**Both sides fairly:** Codex prioritizes first-time-teacher comprehension. Claude prioritizes spec / codebase consistency. "Master" carries genuine ambiguity (master-class vs master-version); "Team Curriculum" is unambiguous. Counter: "Team Curriculum" reads as a noun (the thing) where "Master" reads as a mode (the act). The TOGGLE is a mode-switch, not a noun-switch — "Mine" / "Team" reads naturally for the toggle but breaks "Master" terminology used in CLAUDE.md §2's forking model.

### Disagreement 3 — Pre-edit gate model for Master mode

**Codex Rule #5:** keep no-confirm-dialog on toggle, but add strong inline affordance before/during the first shared edit (presumably: cursor changes, field shading, edit-control treatment shifts).

**Claude §3-A1:** tutorial popover on first flip to Master (educates the teacher one beat before they start editing); MasterBanner persists thereafter.

**Both sides fairly:** Codex's inline cues live at the edit point — the highest-risk moment. Claude's popover teaches the *concept* before any edit. They're complementary, not exclusive. A hybrid (popover on first flip + persistent banner + inline field treatment in Master mode) is strictly better than either alone — but adds visual noise. Tim picks where the budget lives.

### Disagreement 4 — Disabled "coming soon" controls: visible vs hidden

**Codex #5:** hide them, or move to a roadmap area. Disabled controls compete visually with live ones, look broken on click.

**Claude (implicit — accepting Lane D's commit `be87ec7`):** keep them visible with tooltip + opacity + aria-disabled. Tim explicitly chose this in Lane D as an onboarding signal: "tells teachers the planned scope."

**Both sides fairly:** Codex protects against the "click does nothing" frustration moment. Claude/Tim protects against the "where did Filter / Export go?" frustration moment (when these were hidden). The disabled-visible pattern works *if* the styling is unambiguous (the opacity drop + cursor:not-allowed make it clear). Codex's argument is that "unambiguous" doesn't survive a glance on a tablet.

### Disagreement 5 — GlobalRail philosophy: real-actions-only vs unified-nav

**Codex #12:** rails should hold *real, reliable actions only*. Future features go to a "labs / roadmap" area.

**Claude (implicit — accepting Lane FA's commit `1c54590`):** rails are the universal nav. All icons coexist; the drag-to-move + Hidden bucket let teachers curate their own rail.

**Both sides fairly:** Codex sees the rail as a *contract* ("everything here works"). Claude sees the rail as a *workspace* ("everything you might want; hide what you don't"). The Hidden bucket is Claude's answer to Codex's concern (teachers can remove "soon" icons themselves), but Codex would counter that *new* teachers haven't seen the Hidden feature yet and will hit the inert icons first.

### Disagreement 6 — `P | M` collapse rule (BUILD_STANDARD.md L261)

**Codex Rule #3:** risky abbreviation; "P / M" are internal shorthand. Use "Mine / Team" or icons with labels.

**Claude (recent BUILD_STANDARD.md edit):** kept `P \| M` literally. RES-CRIT-002 explicitly preserved both labels on the toggle.

**Both sides fairly:** Codex's concern is real for first-time use. Claude's concern is bar-budget at 360px. A middle ground ("My" / "Team" — 2 chars + 4 chars vs 8 chars + 6 chars) fits the budget AND reads. Worth changing — Codex is probably right here.

## 5. Gaps neither audit covered

Things reading Codex's audit made me see that neither doc addresses:

1. **Top-bar search — what does it search?** Both audits mention the search trigger; neither asks what scope it covers (lessons? standards? resources? team comments?). At first launch the search is empty; if a teacher types "fractions" do they get every fractions lesson, or every fractions resource, or both? No documented model.

2. **Shoutbox name collision** — recent work renamed the global Comments panel to "Shoutbox" while a per-day "Day Shoutbox" already exists. Both audits missed that a teacher seeing "Shoutbox" in two places (global rail + Daily) will conflate them. Disambiguation copy is in commit `3483db1` but the surface labels both still say "Shoutbox."

3. **Notification surface** — when another teacher edits Master, does the local teacher get any notification? Neither audit covers async team awareness. Phase 1B (Supabase realtime) presumably handles this but the UX surface (where notifications land, badge counts, dismissal) isn't designed yet.

4. **Print fidelity** — the /weekly/print and /year/print routes exist (Lane O). Neither audit checked whether: page breaks land correctly, ink-heavy chrome wastes paper, the printout fits the school's standard paper (US Letter vs A4 — Qatar uses A4), color survives a black-and-white printer. Likely real gaps for a teacher who actually prints to share with sub coverage.

5. **Keyboard shortcuts discoverability** — `components/shell/shortcuts-overlay.tsx` exists (referenced in earlier audits). Neither current audit covers (a) what shortcuts exist, (b) whether they're documented anywhere outside the overlay, (c) whether the `?` keystroke is itself discoverable.

6. **Color-blind accessibility on the 8-subject palette** — both audits accepted the subject palette as a given. With 8 subjects in `vivid + highlight` mode, deuteranope teachers may struggle to distinguish (say) Math from Reading from Writing in a dense Year roadmap. Worth a deuteranope simulation pass.

7. **RTL / Arabic locale support** — the beta school is in Qatar. Mock data uses English. Neither audit asked whether the layouts work right-to-left, whether dates render in Hijri options, whether weekday labels translate. Probably Phase 1B but worth flagging.

8. **Performance with realistic data volume** — both audits used the 36-lesson mock fixtures. At 8 subjects × 4 lessons/week × 36 weeks = 1,152 lessons. Year view renders all UnitBars in DOM. Catch-up lists all uncovered. Neither audit measured rendering perf at scale.

9. **Data ownership in shared-team mode** — when a teacher leaves the team, what happens to their Personal forks? When a new teacher joins mid-year, do they inherit existing Master, or start from a fresh fork? Phase 1B governance question; UX surface (the "leave team" / "invite teacher" flows) doesn't exist yet.

10. **Undo scope** — Ctrl+Z is wired (top-bar.tsx). Does it undo a Master edit that another teacher just made elsewhere? Does it undo across the Personal/Master boundary? Neither audit asked.

11. **Resource-link rot** — resources are URLs (YouTube, Google Drive). When a URL 404s, what does the planner show? Neither audit covers offline resilience.

12. **Onboarding wizard re-entry data loss** — my §3-B3 proposes "replay the onboarding tour" routing to `/onboarding`. The Plan agent confirmed `lib/onboarding-state.tsx` is resumable, but neither audit actually verified what happens to a teacher who already configured their school week and re-runs the wizard. Does the wizard pre-fill their answers? Does it overwrite on completion? Unverified.

13. **Concurrent editing** — when two teachers are on Master at the same time, both editing the same lesson, what happens? Phase 1B Supabase realtime question; the UX surface (conflict resolution, last-write-wins indicator, "Janet is also editing this" presence) isn't designed yet.
