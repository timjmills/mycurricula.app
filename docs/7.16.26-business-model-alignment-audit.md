# Business-model alignment audit — individual-teacher repositioning

> **Snapshot disclaimer (repo convention, CLAUDE.md §8):** dated audit snapshot,
> 2026-07-16. Findings may be fixed, deferred, or superseded by later work —
> verify against current code before treating anything here as open.
>
> **Trigger:** CLAUDE.md §1 was rewritten 2026-07-16 — the customer is an
> **individual teacher**, not a school. Teams are teacher-assembled via invites;
> packaging (tiers, invite counts) is explicitly undecided and must not be
> encoded; school/site/district accounts are out of phase. This audit swept the
> active v2 branch (`claude/v2-wave3-chrome`) + `master` (prod) for
> contradictions. Report-only; nothing was changed.

## Headline

The codebase is architecturally MORE aligned than the old positioning implied:
the 2026-06-06 provisioning rework already made `schools` a **hidden per-teacher
tenant container** (1:1 with a `teams` row) with a teacher-assembled invite
model, and `school_admins` is **dormant** for every individually-provisioned
teacher. Remaining contradictions concentrate in one marketing page, a handful
of copy strings, and one env default. No solo/individual UX exists yet — every
teacher sees team framing.

## (a) Live contradictions

- **A1 — `app/welcome/page.tsx` ~376–424 (master + v2): hardcoded pricing
  table.** Teacher "Free" / Team **"$6 /teacher · mo"** / **"School — Let's
  talk"** with "admin support", "school-wide rollout". Encodes exact packaging
  AND a school-purchase tier, both now forbidden. Mitigant: `/welcome` is
  **orphaned** (no internal link anywhere). Decide first when packaging lands;
  interim = blank the School tier + price, or drop the page from the build.
- **A2 — domain-mode auth copy assumes a school administers the account:**
  `components/auth/google-sign-in-button.tsx:157` ("contact your school lead"),
  `app/login/page.tsx:70`, `components/auth/magic-link-form.tsx:127,155` ("your
  school email"). Wrong for self-serve individual signup; coupled to B3.
- **A3 — "school admin" surfaces to users (live on prod):**
  `app/settings/standards/actions.ts:35` ("Only a school admin can change the
  school's default frameworks"), `app/api/standards/frameworks/route.ts:9`.
  Degrades fine (per-teacher `teacher_frameworks` is the real scope) — a
  copy/concept contradiction, not a broken flow.

## (b) Latent scaffolding (needs a decision before packaging lands)

- **B1 — school-org schema repurposed, not removed.**
  `20260518102823_initial_schema.sql` defines `schools`, `school_admins`,
  `is_school_admin()`, school-scoped RLS writes. The 6-06 rework
  (`20260606120000_teams_invitations.sql`, `…130000_individual_provisioning.sql`)
  redefined `schools` as the hidden tenant primitive;
  `…160000_workspace_notebook_admin.sql` confirms individual provisioning mints
  NO `school_admins` row (tier dormant). Still load-bearing inside authz RPCs
  (`rename_notebook`/`create_notebook` check `is_school_admin OR is_grade_lead`)
  — don't drop; **re-brand user-facing concept to "workspace admin"** (code
  comments already half-do this in `app/settings/workspace/page.tsx`).
- **B2 — invite count pre-encoded:** `teams.seat_cap` default **5**
  (`teams_invitations.sql:82`, check 1–50) + duplicated `cap: 5` fallback
  literals (`lib/admin/queries.ts:176,183`, `app/settings/workspace/page.tsx:68`).
  The per-team DB column is the right config home; the magic default + scattered
  literals are the problem. When packaging settles: one default, delete the
  literals.
- **B3 — provisioning defaults to the SCHOOL model.**
  `lib/supabase/ensure-teacher.ts:109`: `PROVISIONING_MODE === "individual" ?
  "individual" : "domain"` — default is the domain/allow-list/"school lead"
  path; individual is opt-in (localhost/beta only). For the new positioning,
  **individual should become the default**, domain kept as Qatar-beta opt-in.
  Gates the A2 copy fixes.
- **B4 — no viewer/limited role exists.** `lib/types.ts:27` `TeacherRole =
  "lead" | "teacher"`; invited colleagues get FULL teacher seats
  (`can_edit_master` defaults true). Consistent with "what invited accounts can
  do is open" — just don't assume invited = read-only anywhere yet.
- **B5 — solo/individual UX unbuilt.** No `'solo'` handling anywhere; the v2
  plan's Wave-12 proposes it (hide Personal/Team toggle, pin save target
  personal, suppress pink glow, verify solo can't write core). Today solo
  teachers see full team framing. Decide: build Wave-12 solo mode or consciously
  accept team framing.

## (c) Fine — product mechanics, not business model

"Whole team"/"your school" copy in settings (teaching context, not purchase
claim); grade-scoping; `MAX_BOARDS_PER_TEACHER = 50` (`lib/teach/limits.ts`,
anti-abuse cap — though it's the kind of lever that could later become a tier
limit); resource size caps; mock fixtures' roles.

## Prioritized decide-first list

1. Packaging/pricing decision → reconcile **A1** (interim: strip School tier +
   $6, or pull the orphan page).
2. Flip **B3** default to `individual` + rewrite **A2** copy (coupled pair).
3. Rename user-facing "school admin" → "workspace admin" (**A3**); keep DB
   plumbing (**B1**) as internal authz.
4. When limits settle: single-source `seat_cap` default, delete `cap: 5`
   literals (**B2**); decide invited-seat capability, add a real role if
   limited (**B4**).
5. Decide **B5** solo mode in/out for v2.
