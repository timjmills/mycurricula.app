# Teach → Supabase Go-Live Runbook

> **Status:** PRE-GO-LIVE CHECKLIST — not yet executed. Dated 2026-06-08.
> **Context:** The Teach board/widget Supabase-persistence port is **merged to `master`
> but DORMANT** — `NEXT_PUBLIC_TEACH_USE_SUPABASE` is unset, so prod runs the in-memory
> mock and there is no Teach-on-Supabase behavior in production yet. This runbook is the
> gated procedure for flipping it live. The migration that backs it is
> `supabase/migrations/20260607160000_teach_golive_hardening.sql` (plus the two earlier
> Teach migrations, `20260530090000_teach_view.sql` and `20260531120000_teach_freeform.sql`).
>
> **Verification done so far:** static (tsc/lint/`next build`), unit tests (vitest, pure
> helpers only), 3 Codex audit rounds + multiple independent reviews (all findings fixed),
> and the migration validated on a **stub Postgres 16** (no RLS/auth). **NOT yet done:** a
> behavioral end-to-end run against a REAL Supabase under the flag — that is exactly what
> Phase 3/Phase 5 below are for. Flipping the flag is the first real-world test.
>
> **Golden rule:** the flag is reversible (unset → Teach back to mock, minutes); the
> migration's schema change is **not**. Only the Teach surface is affected — Weekly /
> Planner / Settings are untouched no matter what happens.

---

## Phase 0 — Safety setup

- [ ] Confirm **only your account** is active (no colleagues invited yet) — keeps the blast radius to you.
- [ ] Take a **Supabase backup** before any migration (Dashboard → Database → Backups → manual backup, or `pg_dump`).
- [ ] Memorize the **instant rollback**: remove the `NEXT_PUBLIC_TEACH_USE_SUPABASE` env var → redeploy → Teach back on mock.

## Phase 1 — Apply the Teach migrations to Supabase

_(terminal; needs Supabase access)_

- [ ] `supabase link --project-ref <your-project-ref>` (if not already linked).
- [ ] `supabase migration list` — note which Teach migrations are pending. The three that matter:
      `20260530090000_teach_view.sql`, `20260531120000_teach_freeform.sql`, `20260607160000_teach_golive_hardening.sql`.
- [ ] Review first: `supabase db push --dry-run` (or read the pending files) — confirm only the expected migrations, nothing destructive.
- [ ] Apply: `supabase db push`.
- [ ] **GATE 1 — verify the schema** (SQL editor):
  ```sql
  select data_type from information_schema.columns
    where table_name='widgets' and column_name in ('id','type');     -- both must be 'text'
  select proname from pg_proc
    where proname in ('teach_replace_lesson_set','teach_commit_board_pages',
                      'teach_grade_for_lesson','teach_valid_pages');  -- all 4 present
  select tgname from pg_trigger where tgname='trg_boards_lesson_grade';  -- present
  select proname from pg_proc where proname='teach_replace_team_set';    -- must return NOTHING
  ```
  If any check fails → **STOP**, restore the backup, report.

## Phase 2 — Verify the data prerequisites (the "will it even work" gate)

_(SQL editor)_

- [ ] **Lessons/grades must exist in the DB** — Teach boards hang off real lessons:
  ```sql
  select count(*) from master_core_lesson_events;  -- must be > 0
  select count(*) from grade_levels;               -- must be > 0
  select count(*) from units;                      -- must be > 0
  ```
  **GATE 2:** if these are `0`, **STOP** — Teach-on-Supabase will error (no lessons to reference;
  `gradeIdForLesson` returns nothing). You must first get the planner data (lessons/grades/units)
  into Supabase — i.e. put the Planner on Supabase with your grade-5 data. **This is the single
  most likely blocker — confirm the Planner is on Supabase with real lessons, not mock.**
- [ ] **Your provisioning** — find your auth uid (Dashboard → Authentication → Users → your gmail), then:
  ```sql
  select id from teachers where id = '<your-auth-uid>';   -- you must have a teacher row
  ```
  If missing: sign in to the app once (the ensure-teacher flow provisions on login) and re-check.
  Confirm your teacher is attached to a grade you can read.

## Phase 3 — Dress rehearsal LOCALLY (do this before prod — it IS the real test)

_(terminal + Chrome)_

- [ ] `git checkout master && git pull`.
- [ ] Create `.env.local`:
  ```
  NEXT_PUBLIC_TEACH_USE_SUPABASE=1
  NEXT_PUBLIC_SUPABASE_URL=<your supabase url>
  NEXT_PUBLIC_SUPABASE_ANON_KEY=<your anon key>
  ```
- [ ] `npm install && npm run dev` → `http://localhost:3000` → sign in with your account.
- [ ] Run the **Phase 5 Smoke Test Checklist** locally.
      **GATE 3:** all pass locally against real Supabase → high confidence for prod. Anything errors →
      fix here; **do not flip prod.** Capture the exact error + console output.
- [ ] Delete `.env.local` when done (local goes back to mock).

## Phase 4 — Flip production

- [ ] Set the build-time env var (it is `NEXT_PUBLIC_…` → **build-time**, so a rebuild is required):
      GitHub Actions secret/variable `NEXT_PUBLIC_TEACH_USE_SUPABASE=1`, **or** Cloudflare Pages →
      Settings → Environment variables.
- [ ] Trigger a fresh deploy (re-run "Deploy to Cloudflare" / push / Cloudflare "Retry deployment").
- [ ] Wait for the deploy to go green.

## Phase 5 — Production smoke test (Chrome, your account)

Open `mycurricula.app` → sign in → **/teach**. Each must pass; **any failure → Phase 6 rollback.**

- [ ] 1. A lesson's boards **load** (default set seeds on first open) — no error toast, no blank crash.
- [ ] 2. **Add a board** → appears → **reload** → still there.
- [ ] 3. **Add a widget** (e.g. Timer) → reload → still there.
- [ ] 4. **Move + resize** a widget → reload → position persists.
- [ ] 5. **Add a page**, put a widget on page 2, edit it → reload → persists (page model).
- [ ] 6. **Personal vs Team** toggle behaves (no errors).
- [ ] 7. **Push to team** (confirm flow) → team set updates → reload confirms.
- [ ] 8. **Sandbox:** enter sandbox → add a board + widget → **pin to a lesson** → pinned set shows on that lesson.
- [ ] 9. **Board Library:** open a library board → it **pulls a copy into the current lesson** (doesn't blank the workspace).
- [ ] 10. Browser console: **no uncaught errors**; no "couldn't load/add" toasts.
- [ ] 11. **DB confirm** (SQL editor): `select id,title,scope,owner_id from boards order by created_at desc limit 10;` → your boards are real rows.

## Phase 6 — Rollback (if any smoke-test step fails)

- [ ] Remove `NEXT_PUBLIC_TEACH_USE_SUPABASE` (or set ≠ `1`) → redeploy → Teach back on mock. Rest of app unaffected.
- [ ] The schema changes remain (harmless). To clean test boards:
      `delete from boards where owner_id = '<your-auth-uid>';` (widgets cascade).
- [ ] Save the failing step + console errors → fix as a follow-up.

## Phase 7 — Only AFTER a clean prod smoke test

- [ ] Load the **full real grade-5 data** (the mock → real migration).
- [ ] **Then** invite colleagues (RLS-scoped to the grade).
- [ ] Re-test **push-to-team** and **personal/team forking** with a second real teacher before relying on it.

### Known-deferred items (non-blocking; revisit post-launch)

- Board-cap error message degrades to a generic toast across the Server Action boundary in prod (the no-feedback/rethrow bug is fixed; only the message specificity degrades).
- The board cap is check-then-act (TOCTOU) across _different_ lessons — same accepted pattern as the create path; negligible at single-/few-teacher scale.
- Teach data-source methods rely on the SQL harness + audits rather than vitest (the repo tests pure helpers only).
- The full migration file is not idempotent on a manual _full_ re-apply (standard runners apply each file once; the newest sections are idempotent).
