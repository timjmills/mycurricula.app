## Phase 1 — Apply the Teach migrations to Supabase

_(terminal; needs Supabase access)_

- [ ] `supabase link --project-ref <your-project-ref>` (if not already linked).
- [ ] `supabase migration list` — note which Teach migrations are pending. The three that matter:
      `20260530090000_teach_view.sql`, `20260531120000_teach_freeform.sql`, `20260607160000_teach_golive_hardening.sql`.
- [ ] Review first: `supabase db push --dry-run` (or read the pending files) — confirm only the expected migrations, nothing destructive.
- [ ] Apply: `supabase db push`.
- [ ] **GATE 1 — verify the schema** (SQL editor):
  ```sql
  select data_type from information_schema.columns
    where table_name='widgets' and column_name in ('id','type');     -- both must be 'text'
  select proname from pg_proc
    where proname in ('teach_replace_lesson_set','teach_commit_board_pages',
                      'teach_grade_for_lesson','teach_valid_pages');  -- all 4 present
  select tgname from pg_trigger where tgname='trg_boards_lesson_grade';  -- present
  select proname from pg_proc where proname='teach_replace_team_set';    -- must return NOTHING
  ```
  If any check fails → **STOP**, restore the backup, report.

- [ ] 1. A lesson's boards **load** (default set seeds on first open) — no error toast, no blank crash.

### Known-deferred items (non-blocking; revisit post-launch)

- ✅ **RESOLVED in Wave 6b (PR #47):** board-cap error message no longer degrades to a
  generic toast across the Server Action boundary — `BoardCapError` now travels as a
  discriminated envelope and is rebuilt client-side, so the specific "delete a board to
  make room" prompt survives Next's prod error redaction.
- **DEFERRED → Wave 6d: ephemeral sandbox-board orphans.** A real mock-vs-Supabase
  divergence: sandbox scratch boards persist as `ephemeral=true` rows on Supabase (the
  mock was in-memory, so they vanished on reload), and `listBoardsForLesson(SANDBOX)`
  returns all of them. The fix is destructive-adjacent (deleting scratch boards after a
  successful pin) and must be designed against the **observed live flow** with
  content-preservation proof — so it was deliberately **not shipped blind** at cutover.
  Blast radius is one teacher (Tim) until colleagues are invited. Track as Wave 6d.
- The board cap is check-then-act (TOCTOU) across _different_ lessons — same accepted pattern as the create path; negligible at single-/few-teacher scale.
- Teach data-source methods rely on the SQL harness + audits rather than vitest (the repo tests pure helpers only).
- The full migration file is not idempotent on a manual _full_ re-apply (standard runners apply each file once; the newest sections are idempotent).
