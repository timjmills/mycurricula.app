# Archive & Roll-Forward — Ultraplan

> Build plan for the **Archive & Roll-Forward** feature: archive any slice of
> the curriculum (year / subject / unit / lesson / board) with all its data, and
> copy (roll forward) any slice onto the current year OR a new calendar year,
> resetting completion. Dated 2026-06-01. Branch: `claude/ecstatic-bohr-1ovo3`
> (at deployed master 5301f45). Owner decisions captured in
> `docs/6.1.26 Archive and Roll-Forward Spec Notes.md`; this plan grounds them
> in the real schema + data layer. Companion patterns: the **Planner Data →
> Supabase Ultraplan** (`docs/6.1.26 Planner Data Supabase Ultraplan.md`) and the
> Teach trio (`lib/teach/{supabase-source,actions,client}.ts`). Mock stays the
> default; everything is flag-gated; nothing regresses until the owner flips it.

## 0. Why this plan / grounding facts (verified against code)

- **The year entity exists** (`school_years`, initial schema §3) but is NOT a
  hard FK target on most curriculum rows. The grade-scoped chain is
  `subjects → units(school_year_id) → master_core_lesson_events → completion_status`.
  **`units` is the ONLY curriculum row carrying `school_year_id`** today.
  Lessons inherit their year transitively through `unit_id → units.school_year_id`.
  This matters: roll-forward to a new year is fundamentally "re-parent a fresh
  copy of units (and everything under them) onto a target `school_year_id`."
- **Archivable entities today** — relevant rows and their existing soft-state:
  - `subjects` — has `is_archived boolean` already (used for personal-subject
    rollover, §4.5), NO `archived_at`.
  - `units` — no archive column; carries `school_year_id`.
  - `master_core_lesson_events` — has `deleted_at` (soft-delete), no `archived_at`.
  - `personal_core_lesson_event_copies` — lazy fork rows; no archive column.
  - `personal_authored_lessons` (migration `20260601120000`) — has `deleted_at`,
    no `archived_at`.
  - `completion_status` — keyed `(teacher_id, core_lesson_event_id→master id)`.
    **Never forks; never copied on roll-forward** (locked decision #1).
  - `boards` (migration `20260530090000`) — keyed to a lesson via
    `master_core_lesson_event_id`; `scope` ∈ team|personal; partial-unique on
    `(master_lesson, owner, title)`. No archive column.
  - `widgets` — `board_id` FK, `config`/`state` jsonb (STRUCTURE only, §11.4).
  - `lesson_sections` (migration `20260601120000`) — polymorphic
    `(owner_kind, owner_lesson_id)`; resources live in its `resources` jsonb.
- **No `team_settings` table yet** — academic-year + school-months are
  localStorage (`mycurricula:team:academic-year-{start,end}`,
  `lib/use-academic-year.ts`). "New calendar year" target therefore needs either
  a real `school_years` row (preferred when the planner backend is on) or a
  client-side year stub when mock-default. Plan supports both.
- **Data-layer seam already exists for the planner** (`lib/planner/*`:
  `source.ts` interface, `mock-source.ts`, `supabase-source.ts`, `actions.ts`
  generic dispatch, `client.ts` Proxy facade, `id-bridge.ts`). Teach mirrors it
  (`lib/teach/*`). Archive/roll-forward is a NEW concern spanning BOTH planner
  rows and Teach boards, so it gets its own `lib/archive/` module that mirrors
  the same source/mock/supabase/actions/client five-file shape and **reuses** the
  planner + teach sources for the per-entity reads/writes where possible.
- **Destructive-action UX is in place**: `components/ui/Tooltip.tsx` carries
  `tooltipId` + `required` (always-on); `lib/consequence-toast.tsx` shows the
  team-effect confirmation toast. Archive is destructive-ish and team-scoped →
  `required: true` tooltips + a confirm step.
- **Flag pattern**: `isPlannerSupabaseConfigured()` reads
  `NEXT_PUBLIC_SUPABASE_URL` + `NEXT_PUBLIC_PLANNER_USE_SUPABASE=1`. Archive
  reuses the SAME gate (it operates on planner + teach data) — no new flag; when
  the flag is off, archive/roll-forward runs entirely against the mock sources.

## 1. Schema — additive migration `20260602120000_archive_rollforward.sql`

**Principle: additive only.** New nullable columns + one provenance table + RLS.
Nothing existing changes shape, so the mock/grid paths are untouched.

### 1.1 `archived_at timestamptz` on the archivable entities

Add a nullable `archived_at timestamptz` (default null) + a partial index on each:

| Table | Column added | Rationale |
| --- | --- | --- |
| `subjects` | `archived_at timestamptz` | Subject-level archive. (Distinct from the existing `is_archived` boolean, which is the personal-subject *rollover* flag, §4.5 — keep both; document that `archived_at` is the new general archive timestamp and `is_archived` stays the rollover-decision flag. Optionally backfill `archived_at = now()` where `is_archived` and `archived_at is null` in a later pass — NOT in this migration.) |
| `units` | `archived_at timestamptz` | Unit-level archive. |
| `master_core_lesson_events` | `archived_at timestamptz` | Lesson-level archive. Distinct from `deleted_at` (30-day soft-delete): archive is intentional, indefinite, restorable; delete is the trash window. |
| `personal_authored_lessons` | `archived_at timestamptz` | Teacher-authored lessons archive too. |
| `boards` | `archived_at timestamptz` | Board-level archive (boards carry the forking model, §13.1). |

```sql
alter table subjects                   add column if not exists archived_at timestamptz;
alter table units                      add column if not exists archived_at timestamptz;
alter table master_core_lesson_events  add column if not exists archived_at timestamptz;
alter table personal_authored_lessons  add column if not exists archived_at timestamptz;
alter table boards                     add column if not exists archived_at timestamptz;

create index if not exists idx_units_active        on units (grade_level_id) where archived_at is null;
create index if not exists idx_master_events_active on master_core_lesson_events (unit_id) where archived_at is null;
create index if not exists idx_boards_active        on boards (master_core_lesson_event_id) where archived_at is null;
```

Children DO NOT each get a column where the parent's archive implies theirs:
- `widgets`, `lesson_sections` — archive-state is **derived** from the owning
  board / lesson (a widget is archived iff its board is). No column; the read
  filters by joining to the parent's `archived_at`.
- `completion_status`, `personal_core_lesson_event_copies` — follow their master
  lesson's `archived_at`. Forks are NOT separately archivable (a fork with no
  visible master is meaningless). When a master lesson is archived, its forks +
  completion travel with it as the historical record (locked decision #1).

**Semantics (resolves spec open-Q #1): tag + hide, never destroy.** Archiving
sets `archived_at = now()` and every active view filters `where archived_at is
null` (or the parent's). An "Archived" filter/view surfaces them for browse +
restore (restore = `archived_at = null`). No row is deleted.

### 1.2 Copy provenance table — `curriculum_copies`

Records each roll-forward so copies are auditable + a copy can show "copied from
<source> on <date>". One row per copy *operation* plus child link rows would be
heavy; instead one row per *operation* with a jsonb id-map is sufficient for v1.

```sql
create type copy_target_kind as enum ('current_year', 'new_year');

create table curriculum_copies (
  id                 uuid primary key default gen_random_uuid(),
  grade_level_id     uuid not null references grade_levels(id) on delete cascade,
  copied_by_id       uuid not null references teachers(id) on delete cascade,
  source_school_year_id uuid references school_years(id) on delete set null,
  target_school_year_id uuid references school_years(id) on delete set null,
  target_kind        copy_target_kind not null,
  -- The selection scope: { years:[], subjects:[], units:[], lessons:[], boards:[] }.
  scope              jsonb not null default '{}',
  -- The Advanced data-type toggles actually applied (completion always false).
  data_types         jsonb not null default '{}',
  -- Map of source-row-id → new-row-id per table, for provenance + idempotency.
  id_map             jsonb not null default '{}',
  copied_at          timestamptz not null default now()
);
create index idx_curriculum_copies_grade on curriculum_copies (grade_level_id);
create index idx_curriculum_copies_target_year on curriculum_copies (target_school_year_id);
```

This also satisfies the existing audit enum (`copied_from_archive`,
`school_year_archived`, `school_year_started` already exist in `audit_action`) —
the data layer writes an `audit_log` row in addition to a `curriculum_copies` row.

### 1.3 The copy walk (server-side, transactional)

A copy of a selected subtree under a `target_school_year_id`, with **fresh uuids**
and **no completion**. The walk, top-down, only over rows matching the selection
scope AND the enabled data-types:

```
for each selected subject (or all subjects of the source year):
  insert subjects' (new id)  -- only if copying to a NEW subject set; on
                             -- current-year copy, reuse the existing subject id
  for each selected unit under it (units.school_year_id = source year):
    insert units' with school_year_id = target, new id          [always]
    for each non-archived master_core_lesson_events under the unit:
      insert master_core_lesson_events' new id, unit_id = units'.id
        - title/directions/learning_objectives  ← if data_types.lessonContent
        - resources (jsonb mirror)               ← if data_types.resources
        - standards (uuid[])                     ← if data_types.standards
        - notes                                  ← if data_types.notes
        - week_number: remap or verbatim (open-Q #1, §4)
        - NO completion_status rows              [locked: never]
        - NO personal_core_lesson_event_copies   (master content only, open-Q #4)
      for each lesson_sections (owner_kind='master') of the source lesson:
        insert lesson_sections' new id            ← if data_types.lessonContent
                                                    (resources gated by data_types.resources)
    for each board on the source lessons (scope per open-Q #3):
      insert boards' new id, master_core_lesson_event_id = new lesson id
                                                  ← if data_types.boardsWidgets
      for each widget on the board:
        insert widgets' new id, board_id = boards'.id (config/state jsonb verbatim)
  write curriculum_copies row (id_map) + audit_log row(s)
```

Implemented as a **Postgres `security definer` function**
`copy_curriculum_subtree(p_scope jsonb, p_data_types jsonb, p_target_year uuid,
p_target_kind copy_target_kind, p_actor uuid) returns jsonb` (returns the
id_map). Reasons it lives in SQL not TS:
1. It must be **transactional** — a partial copy on failure is corruption.
2. It does many inserts; a round-trip-per-row from TS is an N+1 storm.
3. RLS: as `security definer` it runs as table owner, but it MUST re-check that
   `p_actor` `can_read_grade(grade_level_id)` for the source AND
   `can_edit_subject_master` for any team-scoped target write, and that
   `p_target_year` belongs to the actor's school — these checks are explicit at
   the top of the function (mirrors the §10 helper doctrine).

Archive is simpler — a guarded `update ... set archived_at = now()` per selected
row (+ cascade to children via the parent-derived read filter, so no child
writes needed). It runs as a normal RLS-scoped statement from the source (no
`security definer` needed because it's owner/grade-scoped writes only).

### 1.4 RLS implications

- New `archived_at` columns inherit each table's existing policies — no new
  policy needed for the column itself. The **active-view filter is application
  logic** (`where archived_at is null`), NOT RLS; RLS still governs *who* sees
  the row. Archived rows remain readable by the same teachers (so they can
  browse/restore).
- `curriculum_copies`: enable RLS; read = `can_read_grade(grade_level_id)`,
  insert = same (the copy function inserts as definer but we still gate reads).
- `copy_curriculum_subtree` is `security definer set search_path = public` with
  explicit actor authorization checks at the top (raise on failure). It must NOT
  copy another grade's or school's data — assert `target_year.school_id =
  auth_teacher_school_id()` and source grade ∈ `auth_teacher_grade_ids()`.
- Team vs personal on copy (open-Q #4): master/team content copies under team
  scope (gated by `can_edit_subject_master` for the target subject); personal
  forks + personal boards are NOT copied (per-teacher, per-year). Boards copy as
  team boards by default (open-Q #3).

## 2. Data layer — `lib/archive/`

Mirror the established five-file source pattern (planner/teach). New module
because the concern spans planner rows + teach boards and is its own operation
set; it composes the existing sources rather than duplicating their mappers.

```
lib/archive/
  source.ts          ArchiveDataSource interface + isArchiveConfigured() (reuses planner flag)
  mock-source.ts     in-memory: archive flags lib/mock rows; copy clones with fresh ids
  supabase-source.ts server-only: archive = guarded updates; copy = rpc('copy_curriculum_subtree')
  actions.ts         "use server"; generic archiveDispatch(method, args) (mirrors plannerDispatch)
  client.ts          Proxy facade; flag picks mock vs server (mirrors plannerClient)
  types.ts           ArchiveScope, DataTypeToggles, CopyTarget, ArchiveNode tree types
```

### 2.1 Shared types (`lib/archive/types.ts`)

```ts
/** A selection across the curriculum tree. Empty arrays = nothing of that kind. */
export interface ArchiveScope {
  schoolYearIds: string[];   // whole-year selections
  subjectIds: string[];
  unitIds: string[];
  lessonIds: string[];       // master lesson ids (the stable identity)
  boardIds: string[];
}

/** Advanced per-data-type toggles. Completion is intentionally absent (never
 *  copied, never a toggle — locked decision #1). All default true. */
export interface DataTypeToggles {
  lessonContent: boolean;    // title, objective, directions
  resources: boolean;        // attached links/files
  standards: boolean;        // standards tags
  boardsWidgets: boolean;    // boards & widgets
  notes: boolean;            // teacher notes (master/team notes; open-Q #4)
}
export const ALL_DATA_TYPES_ON: DataTypeToggles = {
  lessonContent: true, resources: true, standards: true,
  boardsWidgets: true, notes: true,
};

export type CopyTargetKind = "current_year" | "new_year";
export interface CopyTarget {
  kind: CopyTargetKind;
  /** For new_year: an existing school_year id, or a {label,start,end} to create. */
  schoolYearId?: string;
  newYear?: { label: string; startDate: string; endDate: string };
  /** open-Q #1 default: remap week numbers to the target year's week 1 origin. */
  remapWeeks: boolean;
}

/** One node in the scope-picker tree (year→subject→unit→lesson→board). */
export interface ArchiveNode {
  id: string;
  kind: "year" | "subject" | "unit" | "lesson" | "board";
  label: string;
  archivedAt: string | null;
  children: ArchiveNode[];
}

export interface CopyResult {
  copyId: string;
  idMap: Record<string, Record<string, string>>; // table → {sourceId: newId}
  counts: Record<string, number>;                 // table → rows copied
}
```

### 2.2 Interface (`lib/archive/source.ts`)

```ts
export interface ArchiveDataSource {
  // ── Tree for the scope picker (active + archived, grade-scoped) ───────────
  getCurriculumTree(
    gradeLevelId: string,
    opts?: { includeArchived?: boolean },
  ): Promise<ArchiveNode[]>;

  // ── Archive / restore (tag + hide; never destroys) ───────────────────────
  archive(scope: ArchiveScope, actorId: string): Promise<{ archivedCount: number }>;
  restore(scope: ArchiveScope, actorId: string): Promise<{ restoredCount: number }>;

  // ── Roll-forward / copy ──────────────────────────────────────────────────
  /** Deep-copy the selected subtree under the target year with fresh ids,
   *  honoring the data-type toggles, resetting completion (always). */
  copyForward(
    scope: ArchiveScope,
    dataTypes: DataTypeToggles,
    target: CopyTarget,
    actorId: string,
  ): Promise<CopyResult>;
}

/** Reuses the planner gate — archive operates on planner + teach data, so it
 *  persists exactly when the planner backend is on. No separate flag. */
export { isPlannerSupabaseConfigured as isArchiveConfigured } from "@/lib/planner/source";
```

- `supabase-source.ts`: `archive`/`restore` run guarded `update` statements per
  table under RLS (filter to the scope ids ∧ grade); `copyForward` calls
  `supabase.rpc('copy_curriculum_subtree', {...})` and maps the returned jsonb to
  `CopyResult`. `getCurriculumTree` is a batched read (subjects→units→lessons→
  boards) reusing the planner/teach mappers; never an N+1.
- `mock-source.ts`: archive toggles an `archivedAt` field on cloned `lib/mock`
  records held in module state; `copyForward` clones the selected mock rows with
  fresh ids (uuid v5 off the source id + target year, mirroring the importer's
  deterministic scheme) and zeroes completion. Byte-identical default behavior
  preserved (no mock row is mutated until the user acts).
- `client.ts` + `actions.ts`: exact Proxy/generic-dispatch copies of the planner
  pair, renamed `archiveClient` / `archiveDispatch`.

### 2.3 Store integration

- The planner store (`lib/planner-store.tsx`) gains awareness only of the
  **active-view filter**: hydrate already excludes soft-deletes; extend the
  source's `listLessons`/`listUnits` to also exclude `archived_at is not null`.
  Archived browse is a SEPARATE read (`getCurriculumTree({includeArchived:true})`),
  not part of the live document — so the reducer is untouched.
- After a `copyForward` into the **current** year, the store re-hydrates the
  affected grade (the new rows appear). Into a **new** year, no live-view change
  (the active year is unchanged) — a toast confirms + offers "switch to <year>".

## 3. UI — Settings → "Archive & Roll-forward" (Advanced)

The spec says "advanced settings." The Settings hub
(`app/settings/layout.tsx` `TABS` registry) is the home. Add a **team-scoped**
tab.

### 3.1 New route + tab

- New page: `app/settings/archive/page.tsx` (+ `page.module.css`).
- Register in `TABS` (and the `VALID_SLUGS` set in `app/settings/page.tsx`):
  `{ slug: "archive", label: "Archive & roll-forward", href: "/settings/archive",
  scope: "team" }`. Team scope → the layout renders the team chip + the
  `title=` "changes affect every teacher on your team" affordance already wired.

### 3.2 Page composition (three regions + confirm)

1. **Scope picker** — a collapsible tree (`ArchiveNode[]` from
   `getCurriculumTree`): Year ▸ Subject ▸ Unit ▸ Lesson ▸ Board, each row a
   tri-state checkbox (checking a parent selects all descendants; partial =
   indeterminate). A segmented filter toggles **Active / Archived** so the same
   tree powers both archive (from Active) and restore (from Archived). Reuses
   the subject-color stripe vocabulary (`.cp-subj.<subject>`) and the existing
   tree/disclosure primitives; no new color tokens.
2. **Action selector** — two primary actions on the current selection:
   **Archive** (or **Restore** when viewing Archived) and **Roll forward / copy**.
3. **Advanced panel** (collapsed by default — "general it will do it all") — the
   five `DataTypeToggles` switches, all ON. Completion is shown as a **disabled,
   explanatory** row ("Completion is never copied — a new year starts fresh")
   with a `required` tooltip explaining *why* it's disabled (per §4 disabled-
   control rule). Each toggle is a switch with a `required: false` onboarding
   tooltip (these are not high-consequence individually).
4. **Target picker** (only for roll-forward) — radio: **Current year** vs **New
   calendar year**. New-year reveals a label + start/end date stub (writes a
   `school_years` row when the backend is on, else a client year stub via
   `lib/use-academic-year.ts`). A **"Remap week numbers"** switch (default ON,
   open-Q #1).
5. **Confirm step** — Archive and Roll-forward are consequential + team-scoped:
   - The action `<Button>` carries a `required: true` tooltip (always-on,
     ignores the global off-switch + per-id dismissal), voiced to the
     consequence ("Archive these 12 lessons and everything attached — they leave
     the active plan for the whole team; you can restore them later").
   - On click, a **summary confirm** (not a blocking modal chain — one confirm
     consistent with "no confirm dialogs for *mode*" but archive IS a mutation):
     lists exactly what will be archived/copied (counts per kind) + the target,
     then a single "Archive N items" / "Copy N items to <year>" button. After
     success, `useConsequenceToast()` fires the team-effect toast.

### 3.3 Discoverability + entry points

- Primary entry: Settings → Archive & roll-forward.
- Optional deep affordances (Phase-later, NOT this wave): a per-unit "⋯ →
  Archive" / "Copy forward" on the `/year` roadmap and `/subject` unit pages that
  deep-link to `/settings/archive?preselect=<unitId>`. Wave scope keeps it in
  Settings only; the preselect query param is designed-in but the call sites are
  deferred.

## 4. Open questions — recommended defaults (owner approves/adjusts)

| # | Open question | **Recommended default** | Why |
| --- | --- | --- | --- |
| 1 | New-year copy: remap week numbers or copy verbatim? | **Remap to the target year's calendar (week 1 = target start), with a "Remap week numbers" switch defaulting ON.** Copy verbatim when off. | A new year usually has a different number of weeks/holiday layout; verbatim week 34 may not exist. Remap = shift so the subtree's earliest source week becomes the target's week 1, preserving relative spacing. A teacher who wants identical numbering flips the switch off. |
| 2 | Conflict if target slot already occupied | **Duplicate-alongside (never overwrite, never silently skip).** The copy always inserts new rows; if a slot already has a lesson, both coexist (ordered by `display_order_within_day`), flagged in the post-copy summary ("3 slots already had lessons — copies added alongside"). | Overwrite risks destroying the team's current plan with no undo; skip silently loses content. Duplicate-alongside is non-destructive and visible; the teacher reconciles manually. Copying onto an *empty* new year has no conflicts anyway (the common case). |
| 3 | Boards roll forward as team / personal / mirror source? | **Copy as TEAM boards by default** (the curriculum a team shares forward is the team set). Personal boards/annotations are NOT copied (per-teacher, per-year, like personal forks). The Advanced "Boards & widgets" toggle gates whether boards copy at all. | Roll-forward is a team-curriculum operation; team boards are the shared asset. Personal boards + ink annotations are private working state that doesn't belong in next year's master. Consistent with open-Q #4. |
| 4 | What carries on a lesson copy: notes? personal forks or master only? | **Master/team content only.** Copy master lesson fields (gated by the toggles); do NOT copy `personal_core_lesson_event_copies` rows or per-teacher completion. "Notes" toggle copies the lesson's **master/team notes**; personal notes are not copied. | Personal forks are per-teacher, per-year by definition; copying them would silently seed every teacher's next-year personal layer from this year's. Master-only keeps the new year a clean shared baseline each teacher re-forks as they go (the §2 forking model). |

(Plus the resolved spec open-Q on archive semantics: **tag + hide via
`archived_at`, with an Archived view to browse/restore** — §1.1.)
