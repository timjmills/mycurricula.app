# 5.27.26 UI UX Review (Codex)

## Scope

This review covers the current mycurricula.app UI/UX direction, primary planner surfaces, and selected project rules from a product usability perspective.

The audit was based on repo documentation and implementation review across the planner shell, Weekly, Daily, Yearly, Curriculum, Catch-up/settings patterns, and shared UI primitives.

I could not complete a browser viewport pass because the local `/weekly` route returned `500`, and `npm run build` failed with a missing `.next/build-manifest.json`. The findings below are therefore code-grounded and should be validated visually once local rendering is healthy.

## Prototype Caveat

This product is still partially implemented. Some static buttons and disabled controls are intentional stand-ins: they show future placement, communicate planned scope, and help the team track what has not been wired yet.

That is acceptable for a prototype. The UX concern is not "all placeholders are bad." The concern is whether a stand-in looks too much like a finished working control, especially inside a teacher's daily workflow. For prototype review, placeholders should be clearly distinguishable from live actions. For production readiness, they should either be wired, hidden, disabled with clear language, or moved into a roadmap/coming-soon area.

## Highest-Impact UX Issues

### 1. App Chrome Is Doing Too Much

The persistent app chrome includes the top bar, rails, global panels, and always-present navigation/actions. Right now the top bar carries a lot of responsibilities: app navigation, Personal/Master mode, filter toggle, view mode, search, catch-up, to-dos, comments, profile, sign-out, save state, and overflow handling.

This creates two UX problems.

First, teachers have to visually parse too many controls before they even get to the planner content.

Second, the CSS has to aggressively hide or clip controls at smaller widths. A rule like `overflow-x: clip` prevents horizontal scroll, but it can also hide the symptom instead of solving it: controls may stop causing overflow while becoming hard to discover or unreachable.

Recommendation: make the top bar answer only global questions: "Where am I?", "Am I editing Personal or Master?", and "Where do I go next?" Route-specific actions should live inside the route header, local toolbar, drawer, or panel.

Relevant reference:

- `components/shell/top-bar.module.css`
- `components/shell/top-bar.tsx`

### 2. Weekly Right Rail Is Unreachable Below 1280px

The Weekly view has a right rail for supporting content such as resources, to-dos, and comments. Below `1280px`, the CSS hides that rail. The code comments indicate access will come through a future affordance, but that means the current tablet and phone experience loses access to useful content.

This matters because hiding content on smaller screens is only good when there is an alternate path. If resources are important on desktop, they should still be reachable on tablet and phone through a drawer, bottom sheet, toolbar button, or route-level panel.

The issue is not that the rail hides. Rails are often poor mobile UI. The issue is hiding the rail without replacing it.

Recommendation: ship a tablet/phone drawer for Weekly resources/to-dos/comments, or explicitly demote that rail content from "primary" until the drawer exists.

Relevant reference:

- `components/weekly/WeeklyShell.module.css`
- `components/weekly/WeeklyShell.tsx`

### 3. Daily Panes Can Resize To Unusable Widths

The Daily view lets teachers resize panes, and the current minimum width is `40px`. That is technically flexible, but it allows states where a pane becomes too narrow to read or operate. A user can accidentally create a layout that looks broken.

A better pane system usually has two layers:

- A meaningful readable minimum, such as `280px` for a lesson list or rail.
- A separate collapsed state, such as an icon, drawer, or hidden panel.

That gives teachers layout control without making them manage broken intermediate states.

Recommendation: replace the `40px` floor with usable minimum widths per pane, then add explicit collapse behavior for cases where a pane should get out of the way.

Relevant reference:

- `components/daily/DailyView.tsx`
- `components/daily/DailyView.module.css`

### 4. Curriculum Rows Have Nested Buttons

In the Curriculum view, a lesson row is rendered as a clickable button, and inside that row there is another button for completion. HTML does not allow interactive controls nested inside other interactive controls.

This can cause real accessibility and interaction bugs. Keyboard focus may behave strangely, screen readers may announce the row incorrectly, and clicking the inner completion button can conflict with the outer row click.

Recommendation: make the row a non-button container, then put separate controls inside it. For example, use one button for expand/open and one button for completion.

Relevant reference:

- `components/subject/SubjectView.tsx`
- `components/subject/SubjectView.module.css`

### 5. Yearly View Shows Disabled "Coming Soon" Actions

The Yearly header includes disabled controls like Filters and Export. They are visible but unavailable.

This is usually frustrating in production UI because it makes the page look more capable than it currently is, then blocks the user. Disabled future actions also compete visually with actions that actually work, such as Add unit, Today, and Print.

Recommendation: if an action is not available now, hide it unless it explains a critical upcoming workflow. If it must be shown, place it in a secondary menu, roadmap area, or release-note surface rather than the primary action cluster.

Relevant reference:

- `components/year/YearView.tsx`
- `components/year/YearView.module.css`

## Additional Findings From Second Pass

### 6. Placeholder Controls Need Clear Prototype Treatment

Several high-value teacher workflows include visible controls that either do nothing, open only a status line, or are explicitly marked as future work in comments. In a half-built prototype, these can be useful stand-ins because they show intended placement and future scope.

The UX risk is when a stand-in looks like a finished control. Individually, each one may be small. Together, they can teach users that the app has affordances they cannot trust.

Examples:

- Lesson detail has menu and full-screen buttons described as stubs.
- Lesson detail has an "Add status" button for a future status submenu.
- Lesson flow has "Edit lesson flow / template" with coming-later copy.
- Section resources has a "Show more" button that is a non-functional placeholder.
- Resource composer has a Search tile that only shows "Search the resource board - coming soon."
- Schedule pane has disabled options, date-picker, and Add time block controls.

Recommendation: keep placeholders where they help explain planned product shape, but style and label them consistently as prototype/future controls. If a feature is not usable yet, either hide it, place it in a clearly labeled "coming later" area, or make the whole unavailable feature visibly disabled before the user tries to use it.

Relevant reference:

- `components/daily/LessonDetail.tsx`
- `components/lesson-flow/lesson-flow.tsx`
- `components/lesson-flow/section-resources.tsx`
- `components/daily/ResourceComposer.tsx`
- `components/schedule/ScheduleDayPane.tsx`

### 7. Catch-up Bulk Actions Need Clearer Prototype State

The Catch-up bulk action bar includes "Add all to to-do", but the handler only clears the selection. That is a serious trust issue because the user thinks they successfully created reminders, but nothing was added.

Carry-over is also incomplete: it marks lessons as carried but records no target destination. The row visually changes, but the teacher has not actually answered the practical question: "When am I teaching this now?"

Recommendation: if these actions are stand-ins, mark them as such in the UI. For production, remove or disable "Add all to to-do" until it writes real to-dos. For Carry over, require a target week/day before applying the action, or label the action as "Mark as needs carry-over" if it is only a triage state.

Relevant reference:

- `components/catchup/CatchupScreen.tsx`
- `components/catchup/BulkActionBar.tsx`

### 8. Configurable School Week Is Not Consistently Reflected

The product rules say the school week is configurable, but the Schedule route and Schedule drawer currently hard-code a five-day Sun-Thu strip. That will feel wrong for schools using Mon-Fri, six-day weeks, or custom schedules.

This is especially important because Schedule is a primary shipped surface. A teacher who already configured a school week should never see a different week model inside the schedule UI.

Recommendation: derive Schedule day strips from `useSchoolWeek()` or the same school-week source used by Weekly/Daily. Keep mock Sun-Thu only in fixture data, not in route-level UI.

Relevant reference:

- `app/(planner)/schedule/page.tsx`
- `components/schedule/SchedulePanel.tsx`
- `components/schedule/ScheduleDayPane.tsx`

### 9. Grade 5 Still Leaks Into User-Facing UI

The product is multi-grade by design, but several user-facing surfaces still say Grade 5 directly. This appears in app metadata, Subject header metadata, Catch-up metadata, and print output.

That matters because the teacher's confidence in the app depends on the app reflecting their actual team. If a Year 7 or K-12 teacher sees "Grade 5", the product immediately feels like a demo instead of their planner.

Recommendation: replace hard-coded Grade 5 copy with the configured curriculum label everywhere. If no label exists, omit the suffix rather than falling back to a grade-specific value.

Relevant reference:

- `app/layout.tsx`
- `components/subject/SubjectView.tsx`
- `components/catchup/CatchupScreen.tsx`
- `app/(planner)/year/print/page.tsx`
- `app/(planner)/weekly/print/page.tsx`

### 10. Resource Discovery Is Too Passive For A Curriculum Tool

The Curriculum resource list shows resources, but rows are non-interactive, capped to 20, and pagination is deferred. The resource type symbols are also abstract, so the list is harder to scan than it needs to be.

Teachers are likely to come to Curriculum asking, "Where is the slide deck / handout / video for this unit?" A passive table that does not open the resource or jump to the lesson is less useful than it looks.

Recommendation: make each resource row actionable. At minimum, rows should open the resource or jump to the attached lesson. Replace abstract glyphs with consistent file-type icons and add sorting/search before pagination becomes necessary.

Relevant reference:

- `components/subject/ResourcesSort.tsx`
- `components/subject/ResourcesSort.module.css`

### 11. Editing Is Too Dependent On Hidden Or Desktop-Centric Affordances

Lesson title and objective editing rely on double-click or keyboard Enter/F2 on text that looks mostly like content. This can work for power users on desktop, but it is weak for teachers on tablets or touch devices.

The app already uses tooltips to explain the behavior, but tooltips are not a strong primary discovery model on touch. If a teacher cannot tell what is editable by sight, editing feels like a hidden feature.

Recommendation: expose edit affordances more directly on editable fields. A small pencil button, "Edit" control on focus/hover, or always-visible edit affordance in dense detail views would make the interaction clearer without adding a modal.

Relevant reference:

- `components/daily/LessonDetail.tsx`
- `components/daily/lesson-detail.module.css`

### 12. Global Rails Mix Navigation With Future Feature Markers

The global rail contains real actions, inert "soon" icons, and route-limited icons in the same vertical strip. On phone, the rail hides entirely and the user is expected to rely on top-bar overflow/profile paths.

This creates two issues. First, teachers may interpret the rail as a stable app navigation system, but several icons cannot be used. Second, behavior changes sharply on phone, which increases the amount a teacher has to relearn between desktop and mobile.

Recommendation: during prototype work, the rail can show future placement, but future icons should be visually distinct from live navigation. For production readiness, keep the rail to real, reliable actions only or move future features into a secondary "labs" or roadmap area. On phone, provide a single persistent navigation/menu affordance that clearly replaces the rail.

Relevant reference:

- `components/shell/rail-icons.tsx`
- `components/shell/GlobalRail.module.css`
- `components/shell/top-bar-more-menu.tsx`

### 13. Mobile Curriculum Subject Switching Can Hide Context

On mobile, the Curriculum subject sidebar becomes a horizontal scrolling row. At `<=480px`, the "Subjects" label is hidden and subject buttons shrink to truncated labels.

Horizontal scrolling can be acceptable for chips, but subject switching is a primary navigation action inside Curriculum. If users do not realize the row scrolls, later subjects may feel missing. Truncated labels also make similar subjects harder to distinguish.

Recommendation: consider a subject dropdown, segmented menu, or bottom-sheet subject picker on phone. If the horizontal strip stays, add a clear scroll affordance and preserve enough label text to distinguish every subject.

Relevant reference:

- `components/subject/SubjectView.module.css`
- `components/subject/SubjectView.tsx`

### 14. Master/Core/Personal Terminology Is Inconsistent

User-facing copy mixes "Master", "Core Curriculum", "Core", "Personal", and "Personalized Curriculum". The underlying model may need all of those concepts internally, but teachers should not have to learn multiple labels for the same ownership distinction.

This is especially risky because the distinction controls whether changes affect only the teacher or the whole team.

Recommendation: pick one user-facing pair and use it everywhere. For example, "Personal" and "Team Curriculum" may be clearer than "Master/Core" for teachers. Keep implementation terms in code, not visible copy.

Relevant reference:

- `components/shell/top-bar.tsx`
- `components/weekly/save-target-dialog.tsx`
- `components/lesson-card/context-menu.tsx`
- `components/lesson-card/lesson-card.tsx`

## Project Rules To Revisit

### 1. "Every Interactive Control Needs A Tooltip"

This rule is well-intentioned, but too broad.

Tooltips are useful for icon-only buttons, ambiguous controls, disabled controls, destructive actions, and complex workflows. But forcing every button to have a tooltip creates noise and encourages long hover text.

For example, a simple "Save" button usually does not need a tooltip. If every simple control speaks, users stop treating tooltips as useful.

Suggested revision: require tooltips where the visible label alone does not explain the consequence, state, or destination of the control.

### 2. "Match `/weekly` Exactly"

Weekly is a useful visual reference, but "exactly" can be too restrictive.

A weekly grid, daily detail page, yearly roadmap, and settings form have different jobs. They should share tokens, spacing, typography, subject-color logic, and component vocabulary, but they should not necessarily share the same density, card structure, or interaction model.

Suggested revision: "Use Weekly as the canonical design vocabulary, but adapt layout to the surface's job."

### 3. Mobile `P | M` Labels

The docs allow Personal/Master to collapse to `P | M` on very small widths. That is risky because "P" and "M" are internal shorthand. New teachers may not know what they mean, and Master mode is high-consequence.

If space is tight, it is better to keep meaningful labels, use shorter but clear copy like "Mine / Team", or use icons with visible accessible labels.

Suggested revision: avoid `P | M` unless the control also includes a clear visible or persistent explanation nearby.

### 4. "Color Carries Information, Never Decoration"

This is a strong principle, especially because subject colors matter. The risk is treating it as absolute.

Some visual design needs neutral tinting, brand accents, hover states, and hierarchy. Those are not random decoration; they help users read and operate the interface.

Suggested revision: subject and status colors must always mean something. Decorative color should be restrained, neutral, and never conflict with subject or status meanings.

### 5. "No Confirm Dialogs For Master Mode"

Avoiding a modal just to enter Master mode makes sense. Confirm dialogs get annoying quickly.

The dangerous moment is not toggling into Master mode. The dangerous moment is changing shared team content. The persistent banner helps, but editable fields could also carry stronger inline cues while in Master mode.

Suggested revision: no confirmation on entering Master mode, but add strong inline affordance before or during the first shared edit.

## Action Plan

This plan turns the review findings into implementation waves. The sequence is intentional: first make the app auditable, then fix structural/accessibility risks, then restore trust in prototype controls, then polish discoverability and terminology.

### Wave 0: Unblock Visual Verification

Goal: make local rendering reliable enough to validate the rest of the work at real breakpoints.

Actions:

1. Investigate the `/weekly` local `500` and the missing `.next/build-manifest.json` build failure.
2. Re-run the affected route locally after the fix and capture the actual failure cause in the handoff.
3. Establish a repeatable viewport check for `/weekly`, `/daily`, `/year`, `/subject`, `/schedule`, `/catch-up`, and `/settings/*` at 360 or 400px, 768px, and 1280px.

Acceptance criteria:

- `/weekly` renders locally.
- `npm run build` either passes or has one documented, actionable blocker.
- The first visual pass records whether each primary route has document-level horizontal scroll, unreachable controls, or crowded sticky chrome.

Primary references:

- `app/(planner)/weekly/page.tsx`
- `components/weekly/WeeklyShell.tsx`
- `components/shell/top-bar.tsx`

### Wave 1: Fix Hard UX And Accessibility Breaks

Goal: remove issues that can directly block teachers, keyboard users, or touch users.

Actions:

1. Fix the Curriculum nested-button structure. Make lesson rows non-interactive containers with separate explicit controls for open/expand and completion.
2. Replace Daily pane `40px` minimums with readable pane minimums, then add explicit collapse states for panes that need to get out of the way.
3. Add a tablet/phone access path for Weekly right-rail content before hiding the rail below `1280px`. Use a drawer, bottom sheet, or route-level panel so resources, to-dos, and comments remain reachable.
4. Validate that the top-bar overflow menu preserves access to primary navigation and high-value actions on phone.

Acceptance criteria:

- No interactive control is nested inside another interactive control in Curriculum lesson rows.
- Daily panes cannot be resized into unreadable intermediate states.
- Weekly rail content has an alternate path below `1280px`, or the rail is explicitly demoted from primary workflow status.
- At 360 or 400px, every primary action has a visible path or documented More-menu fallback.

### Wave 2: Reduce Chrome Load And Make Prototype State Honest

2. Audit visible no-op, stub, disabled, or "coming soon" controls in Lesson Detail, Lesson Flow, Resource Composer, Schedule, Year, Global Rail, and Catch-up.
3. For each prototype control, choose one treatment: wire it, hide it, disable it with clear explanatory copy, or move it to a clearly labeled future/roadmap area.
4. Fix Catch-up bulk actions so "Add all to to-do" writes a real to-do or is disabled/renamed. Require a destination for Carry over, or rename it to a triage-only state.
5. Revisit disabled Yearly header actions. Keep only live primary actions in the main header; move future actions out of the primary action cluster.

Acceptance criteria:

- No visible primary button implies a completed workflow unless it performs that workflow.
- Catch-up does not silently discard selections or imply that to-dos/destinations were created when they were not.
- Future controls are visually and textually distinct from live controls.
- The top bar has fewer always-visible route-specific responsibilities than the current implementation.

### Docs And Standards Follow-Up

These are documentation changes, not implementation blockers. Make them after the first implementation wave proves the better language in product.

1. Rewrite the tooltip mandate as a progressive-help rule: every ambiguous, icon-only, disabled, destructive, high-consequence, or non-obvious control needs an onboarding explanation; obvious text buttons should not require redundant tooltip copy unless consequence or destination is unclear.
2. Adjust "Match `/weekly` exactly" to "Use Weekly as the canonical design vocabulary, adapting layout to the surface's job."
3. Clarify that subject and status colors must always carry meaning, while restrained neutral tinting may support hierarchy and interaction states.
4. Keep the no-confirm-dialog rule for entering Master/team mode, but add stronger inline cues before or during the first shared edit.

Primary references:

- `CLAUDE.md`
- `BUILD_STANDARD.md`

### Verification Gate For Each Wave

Before closing any wave:

1. Run `npm run lint`.
2. Run `npm run format:check`.
3. For UI changes, inspect affected routes at 360 or 400px, 768px, and 1280px.
4. Confirm no document-level horizontal scroll, all primary controls reachable, phone/tablet touch targets at least 44 x 44px, and sticky chrome does not crowd phone content.
5. Record the result with the required line:

```text
Verified: 360 OK / 768 OK / 1280 OK
```

If a tier is not OK, replace `OK` with the actual remaining issue.
