## 3. Workspace / account model ‚Äî the one real reconciliation

The handoffs specify a **simple** model: `profile.role ‚àà {team, solo}`, a flat
`workspaces{ active, personal[], teams[] }`, one `team{ name, school, members[] }`, a flat
`curriculums{ active, list[] }`, and (7.21) a per-unit **share link** (a copy-link, not an
ACL). `config.jsx` is byte-identical across the whole stack.

But **richer product decisions were locked (7-17) and are partly shipped**: solo/team
**derived per workspace**, per-course sharing (personal = invisible to team; creator +
admin control), and **full multi-workspace + a v2 workspace-first wizard**. W12b-1 (the
course-sharing RPC seam) and W12c (workspace-first onboarding) already landed.

**Resolution:** the app-model intentionally **exceeds** the handoff's design truth here.
Treat the handoff `config.jsx` as the **visual/section template** for Setup, but wire it
to the **shipped richer model** ‚Äî not the flat prototype model. This is the crux the
Settings/Config wave must get right (¬ß5.1).

*(v1.1)* **The per-unit share link (and the workspace header's Share ‚Üó button) is
DEFERRED**, consistent with the user's Wave-9b decision (forgeable base64 token ‚Äî ¬ß6).
It is *not* part of the model to wire; the Share ‚Üó button ships hidden/disabled until a
signed share-link design exists. This resolves an internal inconsistency (v1.0 ¬ß3 implied
wiring it while ¬ß6 deferred it).
