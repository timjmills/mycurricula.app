## 0. Decisions locked (owner, 2026-06-06)

1. **Full pivot to individual + invite teams** as the live model; teacher is the
   root entity, no org wrapper required.
2. **Keep the school-domain gating infrastructure intact but dormant** for a future
   "sell to schools" phase — nothing deleted.
3. **Auth = Google + magic link** (passwordless email). *Supersedes the scope doc's
   email/password* — simpler (no reset/confirm-password flows) and reuses plumbing
   that already exists (§4).
4. **Architecture = low-risk reuse (Strategy A).** A "team" maps **1:1 to a hidden
   workspace** (an existing `schools` row, invisible in the UI). Members of a team
   **share that workspace**, so the existing ~40 RLS policies keep working
   **unchanged**. Defer multi-team-per-teacher and multi-grade-per-team.

Carried forward from the scope doc unchanged: **solo = Master in your own space**
(the linchpin — no content migration on solo→team); **seat model** 1 owner + 4
invited = **5 cap**, pending invites hold a seat, any member may invite or
**rescind**; **invitees fork-only by default**; **backfill the beta Grade-5 school
into one team**; **`schools`/`school_admins`/domain-gating kept dormant**.

## 1. Current-state facts (verified by recon)

- **`teachers.id` = `auth.users.id`** (`initial_schema.sql:218`) — every RLS policy
  keys off `auth.uid()`.
- **The whole authz model funnels through two helpers** backed by
  `teacher_grade_assignments` (TGA):
  - `can_read_grade(grade_level_id)` — caller has a TGA to that grade (or is a
    school admin). Grade-scoped tables (units, master/personal events, subjects,
    sections, day events, Teach boards, todos, tags, comments…) gate on this.
  - `auth_teacher_school_id()` — `teachers.school_id` for `auth.uid()`. School-scoped
    tables (schools, **teachers (teammate list)**, school_years, frameworks,
    resources, teaching_reminders) gate on this.
  - **Decisive consequence:** because team members will **share one `school_id`**
    (the team's hidden workspace), both helpers already return the right answer for
    teammates. **No rewrite of the ~40 policies.** This is the single biggest lever.
- **Auth paths today** (both converge on a Supabase session, both passwordless):
  - **Google** via GSI `signInWithIdToken` (`components/auth/google-sign-in-button.tsx:136`).
  - **Claude bypass** mints a session with `admin.generateLink({type:'magiclink'})`
    → `verifyOtp({token_hash})` (`lib/claude-bypass.ts:304,337`) — **this is the
    magic-link redemption plumbing we reuse**; only *sending* the email + a confirm
    route are missing.
  - **Real end-user magic link is NOT wired** (`signInWithOtp`/`resend` absent).
- **Provisioning** (`lib/supabase/ensure-teacher.ts`): called by callback + bypass
  with the service-role admin client; fails **closed** behind
  `ALLOWED_TEACHER_EMAIL_DOMAINS`; resolves "oldest school → first active grade."
  Creates a `teachers` row + a `teacher_grade_assignments` row.
- **Forking model:** `master_core_lesson_events` (one Master, grade-scoped) ←
  `personal_core_lesson_event_copies` (lazy per-teacher fork, owner-only RLS).
  Master-edit is **per-subject** via `subject_team_memberships.can_edit_master`
  (defaults **true** — a redeem hazard, §5). Today "team" == the grade; there is no
  `teams`/`invitations` table.
- **Master vs branch (verified, conflict-free):** master already has the security
  wave + Supabase backend + a **newer server-side GSI provisioning commit
  (`fa68392`)** + the scope doc. The branch adds only the **bundle fix** (master's
  deploy still needs it) + the review-gate docs. The two sides touch **zero
  overlapping files** → clean merge. *(Note: the recon read the branch; reconcile to
  master first so we build on `fa68392`'s auth — Wave 1 verifies against it.)*

## 2. Architecture (Strategy A — the hidden workspace)

```
auth (Google | magic link)
   └─ converged provisioning hook  (middleware / auth trigger; runs for ALL paths)
        ├─ fresh signup  → ensureIndividualWorkspace()  → new hidden school =
        │                   team-of-1: grade + 8 subjects + school_year + teachers
        │                   (school_id=that school) + TGA(lead) + self STM(edit) +
        │                   teams(owner, seat_cap=5) + team_membership(owner)
        └─ invite accept → redeem_invitation(token) RPC → JOIN the inviter's team:
                            teachers(school_id = team's school) + TGA(grade,role) +
                            team_membership + STM only for collaborator/lead
teams (1:1 schools)  ·  team_memberships (seat ledger)  ·  invitations (lifecycle)
   (control plane that WRITES teachers.school_id + TGA; the rest of the app is unchanged)
```

**Why this is low-risk:** a team is a hidden `schools` row; every member has
`teachers.school_id` = that school. So `auth_teacher_school_id()` and
`can_read_grade()` already isolate correctly — **the existing 40 policies are
untouched.** The scope doc's "must-fix #1" (re-scope same-school reads to
co-membership) is **not needed**, because teammates literally share a school. We
keep only the two invite-specific fixes (§5: redeem-perms + TGA-write-RPC-only).

**What teachers see:** sign up → "My Curriculum" (their own Master) → invite up to 4
→ invitees read that Master and fork personal copies. No "school" anywhere in the UI.

**What stays dormant:** `school_admins`, `is_school_admin()`, the domain allow-list,
and the school-as-product surface (admin console, buying seats). The `schools` *table*
is reused as the invisible per-team container; the school *product* is later-phase.

**Deferred (documented, not built):** one teacher in multiple teams at once;
multi-grade teams; billing/payment (we enforce the seat **count** of 5 only); an
existing solo user joining a *different* team (v1 = create your own, or join via a
fresh-signup invite).

## 3. Data model — additive only (no structural change to existing tables)

```sql
-- teams: 1:1 with a hidden schools row; the seat/naming control plane
create table teams (
  id uuid primary key default gen_random_uuid(),
  school_id uuid not null unique references schools(id) on delete cascade,
  name text not null,
  owner_teacher_id uuid not null references teachers(id),
  seat_cap int not null default 5 check (seat_cap between 1 and 50),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
-- team_memberships: the "active" side of the seat ledger
create table team_memberships (
  id uuid primary key default gen_random_uuid(),
  team_id uuid not null references teams(id) on delete cascade,
  teacher_id uuid not null references teachers(id) on delete cascade,
  role grade_role not null default 'teacher',     -- reuse existing enum
  created_at timestamptz not null default now(),
  unique (team_id, teacher_id)
);
-- invitations: lifecycle; token stored ONLY as a hash (raw shown to inviter once)
create table invitations (
  id uuid primary key default gen_random_uuid(),
  team_id uuid not null references teams(id) on delete cascade,
  target_grade_level_id uuid not null references grade_levels(id),
  role grade_role not null default 'teacher',
  token_hash text not null unique,                -- sha-256(raw)
  invitee_email citext null,                      -- null = link invite; set = email-bound
  status text not null default 'pending'
    check (status in ('pending','accepted','revoked','expired')),
  inviter_teacher_id uuid not null references teachers(id),
  expires_at timestamptz not null,
  accepted_by uuid null references teachers(id),
  accepted_at timestamptz null,
  created_at timestamptz not null default now()
);
create unique index invitations_one_pending_per_email
  on invitations (team_id, lower(invitee_email)) where status='pending' and invitee_email is not null;
create index invitations_pending_by_team on invitations (team_id) where status='pending';
```

- **Existing tables: no structural change** under Strategy A. (We do **not** make
  `school_id` nullable or add owner columns — that was the higher-risk scope-doc path.)
- **Seat accounting** = `active members + pending invites ≤ teams.seat_cap`,
  enforced inside SECURITY DEFINER RPCs (cross-row aggregate; not raw RLS).
- Add `audit_action` enum values `invite_created/accepted/revoked/expired/resent`.

## 4. Auth & provisioning

- **Magic link (the real gap):** add client `supabase.auth.signInWithOtp({ email })`
  + a small `app/auth/confirm` route that redeems the email `token_hash` via
  `verifyOtp` — symmetric to `claude-bypass.ts`. SMTP: configure a **production
  provider** (Resend/Postmark/SES) in Supabase Auth (built-in is rate-limited) — an
  **ops dependency**, not just code (§8 R5).
- **Email verification gates space creation** (scope decision #2): `signUp` creates
  only the `auth.users` row; `ensureIndividualWorkspace` runs on first **verified**
  entry. Google arrives pre-verified.
- **Provisioning convergence:** centralize so **all** entry points (Google GSI,
  magic link, OAuth code, Claude bypass) run one hook — middleware post-auth or a
  Supabase `auth.users` insert trigger. Fixes the latent gap where GSI sign-ups are
  provisioned only lazily on the next request, and ensures the bypass user still gets
  a workspace under individual mode (§8 R9). **Invite-accept takes precedence** over
  auto-workspace-creation (a user mid-redeem must not also get a solo workspace).
- **Retire the domain gate behind a flag:** `PROVISIONING_MODE = individual | domain`
  (default `individual`). `domain` = today's `ensure-teacher.ts` behavior verbatim;
  `individual` = `ensureIndividualWorkspace`. The allow-list + school-resolution code
  stays intact, just unconsulted in `individual` mode.
- **Open-signup hardening:** Supabase email-confirm ON; CAPTCHA on signup; Auth rate
  limits; generic "if an account exists…" responses (no enumeration); reuse
  `safeRelativePath` for every redirect; copy the origin/CSRF guard from
  `app/auth/gsi/route.ts` (master `fa68392`).

## 5. Invites, seats & the two RLS fixes that remain

**Lifecycle (all via SECURITY DEFINER RPCs with `auth.uid()` membership checks):**
```
create  → invitations(status='pending', token_hash, expires_at)   [seat held]
            RPC asserts: active members + pending invites < seat_cap
accept  → /invite/[token] → sign in/up → redeem_invitation(token):
            validate hash + pending + not expired + email match (email invites)
            + re-check seat_cap ATOMICALLY (FOR UPDATE on the team row);
            insert team_membership + TGA; STM only for collaborator/lead
revoke  → any member sets status='revoked' (decision #4)            [seat freed]
expire  → lazy: expires_at < now() ⇒ status='expired'              [seat freed]
```
**Invite-role mapping:** viewer → `role 'teacher'`, **no STM** (read + fork only);
collaborator → `'teacher'` + STM `can_edit_master=true`; lead → `'lead'` + STM.

**The two RLS fixes we DO keep** (both about the invite path, not the 40 policies):
1. **Redeem must set `can_edit_subject_master` explicitly per role** — the column
   **defaults true**, so a careless redeem would silently grant every invitee
   Master-write. Viewers get **no** STM row.
2. **`teacher_grade_assignments` writes stay admin/RPC-only** — no authenticated
   self-insert policy; `redeem_invitation` (definer) is the only non-admin TGA write.

**RLS on the 3 new tables:** members of a team may `select` their team's
`teams`/`team_memberships`/`invitations`; **all mutations go through RPCs** (mirror
how `audit_log` is locked to `log_audit_event`). `invitations.token_hash` is never
selectable to non-admins beyond existence.

## 7. Wave breakdown (each wave: self-review gate + lint/tsc/vitest/build; flag-gated so master stays safe)

- **Wave 0 — Reconcile + merge to `master`.** Merge the branch's 2 commits (bundle
  fix + review-gate docs) with master's 4 (verified conflict-free). **This is the
  production deploy** that finally ships the security wave + Supabase backend + bundle
  fix. *Gated on explicit owner OK; verify a preview build before the prod push.*
  Then branch fresh from the reconciled master for Waves 1+.
- **Wave 1 — Provisioning convergence + `PROVISIONING_MODE` flag (no behavior
  change).** Default = `domain` (today's behavior byte-identical); converge all auth
  entry points onto one provisioning hook; verify against master's `fa68392` GSI
  route. *Accept:* existing flow unchanged; GSI sign-ups provisioned reliably.
- **Wave 2 — DB migration (additive control-plane tables).** `teams`,
  `team_memberships`, `invitations` + indexes + audit enum values. *Accept:*
  `supabase db reset` + seed green; **zero** existing policies/tables altered.
- **Wave 3 — `ensureIndividualWorkspace` + flip default to `individual` + magic
  link.** New solo Google **and** magic-link users land in a private workspace (8
  subjects, empty Master), seeing nobody else's data; domain mode still available
  behind the flag; SMTP configured. *Accept:* two fresh accounts are fully isolated.
- **Wave 4 — Invite backend (RPCs).** `create/redeem/revoke/expire` + atomic seat
  cap + invite email + the two RLS fixes (§5) + RLS on the 3 new tables. *Accept:*
  invite→accept grants grade access (forking works); pending holds a seat; revoke
  frees it; concurrent redeem can't overshoot; expired/revoked tokens rejected.
- **Wave 5 — Invite/seat UI + solo UX.** Landing magic-link field, accept-invite
  page, Settings→Team, solo toggle/tooltip variant. *Accept:* full create→invite→
  accept→manage loop in-browser at all 3 tiers; required tooltips on destructive actions.
- **Wave 6 — Backfill + tests + abuse limits.** Backfill the beta Grade-5 school into
  one `teams` row + memberships (additive; snapshot first; verify beta login on a
  preview before prod). Two-tenant RLS probe (A can't read B); seat-race test; invite
  lifecycle/expiry tests; rate-limit invite creation; audit entries via `log_audit_event`.
  *Accept:* green CI + documented cross-tenant probe + verified beta continuity.

**Sequencing rationale:** isolation-critical waves (1–4) ship behind flags with the
beta untouched; the user-visible flip is Wave 3's default change + Wave 5 UI. Each
wave is an independently reviewable PR.

## 8. Risks & mitigations

| # | Risk | Mit. |
|---|---|---|
| R1 | Beta backfill detaches the live Grade-5 team | Additive `teams`+memberships over the existing school; snapshot; verify beta login on preview before prod; assert no half-migrated rows. |
| R2 | Invitee silently gets Master-write (STM defaults true) | Redeem sets `can_edit_master` explicitly per role; viewers get no STM (§5 fix 1). |
| R3 | RLS regression = **full** cross-tenant breach | Strategy A leaves the 40 policies untouched (members share a school); add a two-tenant read probe as a release gate; TGA stays RPC-only-write. |
| R4 | One teacher in multiple teams (scalar `school_id`) | **Deferred** for v1 (one team per teacher). Documented limitation; revisit with set-returning helpers later. |
| R5 | Magic-link / invite email deliverability | Production SMTP provider in Supabase Auth; configure redirect/allowed-email domains for prod + localhost; built-in SMTP is dev-only. **Ops task.** |
| R6 | Seat overshoot via concurrent redeem; invite spam | Atomic cap re-check in the redeem tx (`FOR UPDATE`); pending counts vs cap; rate-limit invite creation (DB-backed, not the per-worker limiter). |
| R7 | Logged-out invitee bounced off `/invite/[token]` | Add the route to `PUBLIC_PATHS`; resume accept via guarded `?next=`. |
| R8 | Per-tenant seed cost on every signup | `ensureIndividualWorkspace` replicates the 8-subject graph per workspace (subjects are per-tenant rows; team-wide rule is about colors, not row identity). |
| R9 | Claude bypass under `individual` mode | The convergence hook handles the bypass user too (creates/join its own workspace) so AI surfaces keep data access. |
