## 0. Strategy & sequencing

### 0.1 Migration approach ‚ÄĒ **feature-flagged incremental behind `NEXT_PUBLIC_V2`**

Built on a dedicated long-lived worktree off clean `master`, cut over wave-by-wave. **Not** a
full-replace branch (a months-long red branch on a live prod app, with constant master
re-merges and one terrifying cutover ‚ÄĒ rejected).

**The named risk ‚ÄĒ the appearance engine is global, not per-screen.** `data-frame`/`data-theme`/
`data-tone` live on the app root, and `app/tokens.css` is imported app-wide via `globals.css`.
You cannot run v1 tokens on `/weekly` and v2 tokens on a v2 `/day` in the same document ‚ÄĒ the
`:root` token set is singular. **Mitigation:** Wave 2 is **additive and back-compatible**, not a
swap. v2 token names are almost all *new* (`--canvas`, `--surface`, `--subj-1‚Ä¶15`, `--accent`,
`--r-sm‚Ä¶2xl`); the few semantic collisions (`--brand-*`, `--honey-*`, `--done/progress/idle/warn/
danger`, `--font-*`, `--r-*`) are re-pointed to v2 values v1 tolerates. The `NEXT_PUBLIC_V2` flag
gates the **shell/router** (which chrome + which screen mounts per route), not the tokens; v1
screens keep rendering under a v2-default `paper` frame until each wave rebuilds them, then v1
tokens/components are deleted in Wave 13.

**Shared-worktree discipline** (memory: shared-worktree-session-races, worktree-junction-delete-footgun,
dev-server-foot-guns): dedicated worktree off clean `master` with its **own `npm install`** (never a
junctioned `node_modules`); always `git commit -- <explicit paths>` (a bare commit once swept 91
sibling files into a PR); probe dev servers on **port ‚Č•3010**; never `npm run build` while `next dev`
runs.
