## 0. Where we actually are (verified 2026-06-04)

The seam, schema, and provisioning largely **exist** — but the planner cannot
be flipped on safely yet, and three structural things won't hold at scale.

**Already solid:**
- Multi-tenant schema is real: `schools → grade_levels → {subjects, units,
  master events, …}`, teachers pinned to a school, per-grade roles. RLS enabled
  on every table.
- Provisioning is done + idempotent on BOTH entry paths (`ensureTeacherRecord`
  in the Google SSO callback `app/auth/callback/route.ts:44` and the Claude
  bypass `lib/claude-bypass.ts:293`), keyed on the real `auth.uid()`.
- The data-source seam (interface, mock/supabase sources, generic Proxy facade
  + server-action dispatch) is wired and flag-gated
  (`NEXT_PUBLIC_PLANNER_USE_SUPABASE`).

**Flip blockers (functional):**
1. **Identity is mock.** `lib/planner-store.tsx` passes `ME.id` (`"lh"` slug,
   lines 61/1415/1490/1500/1520) as the owner and the literal `"g5"` as the
   grade. RLS keys on `auth.uid()` + uuid `grade_level_id`. Flip today → blank
   planner, rejected writes. The real uid already lives in
   `app-state.tsx` `currentUser.id` — it just never reaches the store.
2. **Source is stale vs schema.** Migration `20260601120000` added
   `lesson_sections` + `personal_authored_lessons`, but
   `supabase-source.ts` still `throw`s in `createLesson` (:748) and
   synthesizes/flattens sections against flat jsonb (:635/:771) — headings,
   bodies, order never persist.
3. **`softDeleteLesson` deletes for the whole team** — writes `deleted_at` on
   the shared master row ignoring owner (:757) — breaks the forking model.
4. **Most mutators don't persist** — only move/status/edit tee to the backend;
   create, archive, section/resource CRUD, duplicate are reducer-only.
5. **N+1 hydration** — one `getSections` server-action round-trip per lesson
   (`planner-store.tsx:1419`), full-grade lesson scan with no window.

**Scale risks (architecture won't hold):**
- **A. Tenancy-by-join on the hottest table.** `master_core_lesson_events` has
  no `grade_level_id`; RLS resolves tenancy via `unit_id → units.grade_level_id`
  per row (`master_events_read`, M1:1333). The Weekly/Daily grid is the hottest
  read path in the app.
- **B. Cross-tenant write leak.** `recurrence_patterns_write` (M1:1325) lets
  **any** authenticated teacher write **any** `scope='team'` pattern in **any**
  school. `audit_log_insert` (M1:1602) allows forged `actor_teacher_id IS NULL`
  rows with arbitrary grade/school.
- **C. Missing `deleted_at` partial indexes** + an unpartitioned, append-only
  `audit_log`. "Reusable year-over-year" means soft-deleted rows accumulate.
- **D. `can_read_grade` (M1:1139)** runs an admin-join OR-branch for every
  ordinary teacher on every grade-scoped SELECT.

## 1. Design principles for scale

1. **Denormalize the tenant key onto hot rows.** Grade scoping must be a local
   indexed column on the lesson tables, not a join — so RLS and every grade
   query filter `grade_level_id = ANY(auth_teacher_grade_ids())` directly.
2. **RLS that evaluates the membership set once per statement,** not a
   correlated admin-join per row.
3. **Partial indexes track the live-row predicate** (`WHERE deleted_at IS
   NULL`) so archived years never tax current reads.
4. **Tenant-safety is a write-side invariant** — every team write is gated to
   the writer's school/grade; audit rows are stamped server-side, never
   client-trusted.
5. **Batch the hydrate, window the reads, keep mutations optimistic** — one
   round-trip to seed sections, no full-table scans, per-row writes await +
   surface errors.
6. **Everything stays flag-gated and additive.** Flag OFF = byte-identical
   mock. Migrations are additive (add column/index/policy-replace + backfill +
   sync trigger); no destructive rewrites, no in-place table recreation.

## 2. The contract (LEAD owns — frozen before agents start)

`PlannerDataSource` (`lib/planner/source.ts`) gains/changes — the Proxy facade
(`client.ts`) and generic `plannerDispatch` (`actions.ts`) are method-agnostic,
so only `source.ts` + the two source impls change:

- `getActiveGradeLevelId(ownerId): Promise<string | null>` — resolve the
  teacher's grade uuid from their assignment (`teachers.default_grade_level_id`
  → first `teacher_grade_assignments`). Mock returns `"g5"`.
- `getSectionsBatch(lessonIds: string[], ownerId): Promise<Record<string,
  LessonSectionContent[]>>` — one call to seed every lesson's sections (kills
  the N+1). `getSections` stays for single reads.
- `createLesson(input, ownerId, gradeLevelId): Promise<Lesson>` — implement
  against `personal_authored_lessons`.
- `softDeleteLesson(id, ownerId)` — **personal-scoped**: master lesson → write
  the new `personal_core_lesson_event_copies.archived_at`; authored lesson →
  its own `deleted_at`. Never mutate the shared master row.
- `getSections` / `setSections` / section-resource ops → back onto
  `lesson_sections` (full fidelity), personal-fork-resolved.

Owner id and grade id are **always uuids** end-to-end after the flip.
