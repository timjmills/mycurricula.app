## Section 0 — Decisions already made (do not re-litigate)

These were settled during the walkthrough. Claude Code should treat them as fixed inputs, not open questions.

1. **Tooltip model — dismissible first-run, not always-on.** `[tim]` Replace the broad "every control gets a tooltip" mandate (CLAUDE.md §4). New model: every non-obvious control shows its onboarding-voice tooltip **once automatically**, with an inline **"turn this off"** and a **"turn all off,"** plus a **re-enable switch in Settings.** **Exception:** high-consequence and destructive controls (Personal/Team toggle, deletes, team-wide settings) keep their warning tooltip **always on**, regardless of the global off switch. Self-evident text buttons (Save, Cancel) don't need a teaching tooltip at all.

2. **Vocabulary — "Personal" / "Team Curriculum."** `[tim]` Standardize all user-facing copy to this pair. Retire "Master," "Core," "Core Curriculum," "Personalized Curriculum" from the UI. Keep `master` as an **internal code term only**.

3. **Master/Team-mode teaching — popover + inline cues (both).** `[tim]` One-time tutorial popover on first switch to Team mode (teaches the concept), **plus** persistent banner, **plus** inline treatment (ring/shading) on shared editable fields while in Team mode (cues the actual edit moment). No recurring confirm dialog on toggle.

4. **Coming-soon controls — keep visible, mark unmistakably as "future."** `[tim]` Don't hide them; they signal planned scope. But give "future" controls a distinct, consistent treatment (style + label, not just dimmed) so they never read as broken live controls. **At release, hide any beta features still not working.**

5. **Rails — universal nav rail on the LEFT, contextual rail on the RIGHT.** `[tim]` Keep the unified rail; give not-yet-live ("soon") icons a clearly distinct look so first-time teachers don't hit dead icons before discovering the Hidden bucket. Keep "soon" icons for now; hide non-working ones at release.

6. **Schedule — keep its own surface AND make it discoverable near Grid/List.** `[tim]` Retain the `/schedule` route + local chrome, and also surface a discoverable entry point alongside the Grid/List toggle.

7. **Catch-up on/off — per-teacher (user-scoped).** `[tim]` Toggling never affects teammates. Copy/toast says "for you," not team-wide. (This narrows the consequence-toast work — see #29; Catch-up is **excluded** from team-wide toasts.)

8. **Top-bar search — searches everything** (lessons, standards, resources, team comments). `[tim]`

9. **Team awareness — minimal viable now.** `[tim]` Realtime notifications and concurrent editing are in scope from the start, but spec only a **notification surface + a "someone else is editing / last change wins" indicator** now. Defer rich presence (avatars, live cursors) to later.

10. **Help — extend the existing `?` shortcuts overlay**, don't build a parallel surface. `[tim]` Add a visible `?` trigger and route-aware help content to the help system that already exists.

11. **Filter panel — scope to the unit/subject planner.** `[tim]` The filter panel should be visible where filtering applies (the unit/subject planner), not the current "closed on `/subject`, open everywhere else" logic. **`VERIFY`** the exact route logic with Claude Code and align.

### Scope boundaries

- **In scope (active):** everything in Waves 0–4 below.
- **In from the start but minimal:** team realtime notifications + concurrent editing (see #9).
- **Near-term backlog ("soon after"):** print fidelity (page breaks, **A4 for Qatar**, black-and-white survival).
- **Out of scope:** RTL/Arabic locale support; color-blind palette pass. `[tim]`
- **Noted, not prioritized — confirm with Claude Code:** performance at realistic data volume (~1,152 lessons), undo scope across the Personal/Team boundary, resource-link rot, data ownership/governance flows.

## Section 3 — VERIFY against live code (route to Claude Code)

Both cross-reviews flagged findings that may no longer match the code. Resolve each before building the detailed plan.

1. **`VERIFY` Filter-panel route logic** — align to Decision #11 (visible for the unit/subject planner). Current code closes it on mount at `/subject` (`components/subject/SubjectView.tsx:1016-1027`; default `leftPanelOpen=true` in `lib/app-state.tsx`). Confirm intended routes and invert/scope accordingly.
2. **`VERIFY` SOON tabs in More menu** — Codex found no current primary tab is `soon` (Daily/Weekly/Yearly/Curriculum are all live in `components/shell/top-bar.tsx:66-90`). If confirmed, **drop** the "SOON-tab fallback" finding entirely.
3. **`VERIFY` `P | M` collapse labels** — Codex found current code keeps full Personal/Master labels at all widths (`top-bar.tsx:465-471`). If confirmed, this is **docs cleanup only** (BUILD_STANDARD.md), not a UI change. (Decision #2 supersedes the labels anyway → "Personal / Team Curriculum.")
4. **`VERIFY` Weekly right-rail panel mount** — is the panel content still `/daily`-only on non-daily routes? (C3.)
5. **`VERIFY` Subject→Daily "jump to lesson"** — `SubjectView` routes to `/daily?lesson=${id}`, but `DailyView` doesn't read `useSearchParams` and initializes to the first not-done lesson. Likely a real bug: cross-route links may not land on the intended lesson.
6. **`VERIFY` Catch-up flame vs in-grid bar** — confirm the in-grid bar's tooltip is sufficient so B5 narrows to the flame badge only.
7. **`VERIFY` Tooltip/Button `title` fallback on touch** — `Button` mirrors `tooltip`→`title=` as a touch/disabled fallback, but `<Tooltip>` strips child `title` on clone. The claimed native long-press fallback may not exist. Check actual touch behavior.

## Section 4 — General plan (wave-ordered, beta-first)

Waves run sequentially; lanes within a wave are file-disjoint and parallel-safe. Effort is open-ended, but order is by beta-criticality.

**Wave 0 — Make it auditable.** `[codex]` Fix the `/weekly` local `500` and the missing `.next/build-manifest.json` build failure. Establish a repeatable viewport check at 360/768/1280 for `/weekly`, `/daily`, `/year`, `/subject`, `/schedule`, `/catch-up`, `/settings/*`. *Nothing else can be visually verified until this is done — this is why Codex's audit was code-only.*

**Wave 1 — Blockers (Tier A).** A1 nested buttons · A2 Catch-up actions · A3 Daily pane minimums · A4 school-week derivation · A5 Grade 5 → curriculum label · A6 "future" treatment for placeholders. Plus VERIFY-5 (jump-to-lesson) if confirmed a bug.

**Wave 2 — Teachability, vocabulary & scope (Tier B).** B2 vocabulary sweep first (everything downstream uses the new terms) → B3 dismissible tooltip system → B1 Personal/Team teaching → B7 settings scope chips → B8 consequence toasts → concept teaching B4/B5/B6.

**Wave 3 — Discovery, responsive & navigation (Tier C).** C1 chrome slimming + C2 Schedule discoverability · C8 help surface · C7 rail discovery/soon-distinction · C3 weekly-rail tablet/phone fallback · C4 resource rows · C5 edit affordances · C6 mobile subject switching · C9 sticky back · C10 intro subtitles · C11 empty states · C12 responsive cascade audit · C13 reorder teaching.
