## Progress

**✅ Cutover gate built + baselined** — `scripts/probe-v2-hydrate-gate.mjs`.
A flag-ON production build against **real Supabase** that asserts **data rendered**, not
HTTP 200 (a 200-with-an-empty-document is exactly what shipped, so status codes can
never gate this). Aim it at a `wrangler versions upload` preview URL:

```bash
node scripts/probe-v2-hydrate-gate.mjs --base=https://<version>.workers.dev
node scripts/probe-v2-hydrate-gate.mjs            # defaults to prod
```

Per route it asserts the render marker (`[data-planner-item^="lesson:"]`,
`[data-year-lane]`), absence of the exact store string `[planner] hydrate failed`,
console/page health, and records `requestfailed` counts as diagnostic.

**Two findings from building it:**

1. **A `>= 500` health check would have MISSED this incident.** The real fingerprint is
   a **server-action POST (browser → OUR Worker, NOT → Supabase) 404-ing at the edge**
   (the open 21×404). **404 < 500.** The gate flags any non-OK same-origin POST and
   specially marks Next.js server actions (POST + `Next-Action` header).
2. **An aborted navigation FORGES the incident's signature.** A probe that navigates to
   a planner route while a previous load is still hydrating aborts its in-flight fetches
   → `net::ERR_ABORTED` → `TypeError: Failed to fetch` → `[planner] hydrate failed`.
   **Discriminator: the artifact emits `requestfailed` events; the real incident had
   ZERO.** That zero-count is load-bearing evidence, not an incidental detail. Settle on
   `networkidle` after login and between routes.

**v1 prod baseline (trustworthy):** /weekly 37 lessons + clean, /daily 8 lessons +
clean. /year (0 lanes) and /planner (404) fail **as expected on v1** — those markers are
v2-only, confirming the gate discriminates rather than rubber-stamping.

**Confirmed:** `.env.local` carries no `NEXT_PUBLIC_PLANNER_USE_SUPABASE` — the
documented gate hole that let a data-layer failure slip every wave.
