## Addendum — Codex round-3 audit + re-derivation on master (2026-06-05)

A round-3 Codex pass was done against a **stale local branch** (8 commits
behind `origin/master`, missing #14 authorized-master-write). A four-agent
audit found much of it redundant with already-merged work and two items
outright unsafe, so the local edits were **not** landed as-is — the branch was
fast-forwarded to `origin/master` and only the genuinely-new, safe pieces were
re-derived on top of master (verified: lint + tsc + 30 tests + build all green;
forking `saveTarget`/`core` contract preserved end-to-end).

**Kept (re-derived on master):**
- **Server-side GSI provisioning** (`app/auth/gsi/route.ts`, new). The primary
  Google login exchanged the ID token *client-side*, bypassing fail-closed
  provisioning entirely. The button now POSTs the credential to a server route
  that runs `ensureTeacherRecord` and `signOut()`s (expiring `sb-*` cookies) on
  any denial — no session left behind.
- **Shared `lib/safe-path.ts`** — `safeRelativePath` extracted dependency-free so
  the client sign-in button reuses the same redirect validation as the
  server-only bypass path (no more weak `startsWith("/")` duplicate).
- **`claude-bypass` provisioning now fatal** — bypass mints no session on
  provisioning denial (was audit-and-continue); URL bypass secret scrubbed off
  the SSO fall-through bounce.
- **`ensure-teacher` cross-tenant guards** — rejects existing wrong-school teacher
  rows + stale cross-school grade/`default_grade_level_id`; `domain:school_id`
  now mandatory (see owner item #2 above — breaking).
- **Planner**: granular section-mutator persistence (threading `saveTarget`),
  `forkAndPatch` concurrent-first-edit race fix, read-only synthetic-fallback
  hydration (zero extra round-trips, never persisted).

**Consolidated Codex gate (run on the combined diff) — dispositions:**
- **HIGH — SSO callback bypass (FIXED).** `app/auth/callback/route.ts` was still
  fail-OPEN (provisioning failure ignored, session kept), so a wrong-tenant
  Google account could bypass the new `/auth/gsi` gate via Supabase's hosted
  OAuth flow. Now fail-closed identically to `/auth/gsi`: provisioning must
  succeed or it `signOut()`s, expires `sb-*` cookies on the redirect, and sends
  the user to `/login?error=provisioning`.
- **MEDIUM — provisioning atomicity (FIXED).** `ensure-teacher` now re-reads the
  teacher row after the `ignoreDuplicates` upsert and verifies `school_id`
  before attaching a grade assignment, closing the concurrent-first-provision
  race during a domain re-pin.
- **MEDIUM — out-of-order `setSections` / un-persisted `editSection` (DEFERRED,
  Phase 1B).** Both live in the Supabase persistence seam, which is OFF in
  production today (the app renders against `lib/mock/`; persistence is gated on
  `NEXT_PUBLIC_PLANNER_USE_SUPABASE`). They match the existing fire-and-forget
  last-write-wins property of the persist seam; per-`lessonId` write
  serialization and debounced text-edit persistence are tracked for the Phase 1B
  persistence-hardening pass, not this security commit.

**Rejected (NOT landed):**
- The OG-preview **allowlist** swap — kept master's SSRF **blocklist** (og-preview
  is for arbitrary teacher-pasted links; an empty-by-default allowlist would
  silently disable previews). Residual DNS-rebinding gap noted for a later
  IP-pinning pass.
- The **in-place edit of committed migration `…140000…`** + its destructive
  `DELETE`/`UPDATE` "stale-row" cleanup — editing an applied migration silently
  no-ops on migrated envs and breaks checksums, and the delete predicates risked
  removing legitimate units / team `lesson_sections` (`owner_id IS NULL`). Any
  such cleanup must be a separate, dry-run-first data script, reviewed on a
  staging clone — not buried in a hardening migration.
- The stale `setSections` rewrite that dropped `saveTarget` (would have forced
  Team-mode edits into personal forks — a forking-model violation).
