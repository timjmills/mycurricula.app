# Collapse-on-Drag Interaction Pattern

**Status:** Design spec for implementation
**Scope:** Weekly view (primary), Schedule view (deferred), Daily view section reorder (new)
**Reference video:** Padlet-style board interaction (see project context)

---

## 1. The Pattern in One Sentence

When the user picks up a card/section for dragging, **every peer element across the surface collapses simultaneously to a single-line chip**; the user reorders against this dense layout; on drop, **every element re-expands** to its full content form. The dragged item itself rides the cursor in its collapsed (chip) form.

### Why it works

A drag is a *spatial* task, not a *content* task. The user doesn't need to read previews while dragging — they need to see structure and find the drop target. Collapsing everything to chips multiplies visible drop targets by ~5–6× and removes the #1 friction point in drag-and-drop UIs: scrolling mid-drag.

### Where it earns its place in this app

| Surface | Value of pattern | Notes |
|---|---|---|
| **Weekly view** | High — main use case | 40-week grid currently requires heavy scrolling for cross-week moves; collapse lets 4–6 weeks fit on screen during drag |
| **Daily view (section reorder)** | High | A single section's rich content can fill the viewport; reordering 10 sections without scrolling is otherwise painful |
| **Schedule view** | Low–medium | Typically one day visible at a time; drop targets are mostly already on screen. Build only after Weekly + Daily prove out. |

## 2. Universal Foundation

These pieces are shared across all three applications. Build them once.

### 2.1 Library choice: `@dnd-kit`

Use `@dnd-kit/core` + `@dnd-kit/sortable`. Reasons:
- Pointer, keyboard, and touch sensors are first-class
- `<DragOverlay>` portal handles the floating-chip-follows-cursor pattern
- Built-in `dropAnimation` config matches what you'd hand-write
- Active community, modern API, ~10KB gzipped

**Do not** use `react-beautiful-dnd` (unmaintained) or HTML5 native drag (too coarse for the polish required).

### 2.2 State machine: board-level, not card-level

A single source of truth at the surface level drives every peer's density. Per-card `isDragging` state is wrong — the whole surface needs to react to *any* drag.

```ts
type DragState =
  | { phase: 'idle' }
  | { phase: 'dragging'; activeId: string; overId: string | null; overIndex: number | null }
  | { phase: 'dropping'; activeId: string };  // brief, for drop animation

const [dragState, setDragState] = useState<DragState>({ phase: 'idle' });
const density: 'full' | 'compact' = dragState.phase === 'idle' ? 'full' : 'compact';
```

Every child receives `density` as a prop. **No child decides its own density.**

### 2.3 The card/section component shape

Build with one outer container and conditional children — this lets framer-motion's `layout` prop animate height changes smoothly. Do not swap between two separate components for the two densities; the visual identity must remain continuous through the transition.

```tsx
<motion.div
  layout
  layoutId={`item-${id}`}
  transition={{ duration: 0.2, ease: 'easeOut' }}  // matches app motion policy
  className="rounded-lg border bg-white overflow-hidden"
>
  {/* Header row — always rendered, identical markup in both densities */}
  <HeaderRow density={density} {...data} />

  {/* Rich content — only mounted in full density */}
  <AnimatePresence initial={false}>
    {density === 'full' && (
      <motion.div
        initial={{ opacity: 0, height: 0 }}
        animate={{ opacity: 1, height: 'auto' }}
        exit={{ opacity: 0, height: 0 }}
        transition={{ duration: 0.2, ease: 'easeOut' }}
      >
        <RichContent {...data} />
      </motion.div>
    )}
  </AnimatePresence>
</motion.div>
```

### 2.4 Motion tokens (use these exact values app-wide)

App motion policy is **200ms ease-out, no springs, respect `prefers-reduced-motion`.**

| Transition | Duration | Easing |
|---|---|---|
| Collapse on drag-start | 200ms | ease-out |
| Drop animation (chip → final slot) | 220ms | cubic-bezier(.2,0,0,1) |
| Re-expand on drop | 200ms | ease-out |
| Drop indicator slide between gaps | 150ms | ease-out (no spring) |
| Reduced-motion fallback | 150ms | opacity-only fade; no height or position animation |

Total drag→idle: ~250ms. Stay under 300ms or it starts feeling choreographed instead of responsive.

### 2.5 Accessibility (build in from day 1, not retrofit)

- **Keyboard:** dnd-kit `KeyboardSensor`. Space lifts, arrows move, Space drops, Esc cancels. Maps to the app's existing `j/k/h/l` navigation philosophy.
- **`aria-live="polite"` region:** announce "Picked up [name]. Currently between [before] and [after]." on every move. Critical for screen reader users.
- **`prefers-reduced-motion`:** 150ms opacity-only fade between full and compact density. No `layout` animation, no height or position transitions, no drop-animation spring. The dragged item still follows the cursor (that's input, not animation) but its arrival in the final slot is an instant snap with an opacity fade-in on the re-expanding peers.
- **Touch (iPad-critical):** explicit grab-handle affordance only — never long-press the body (conflicts with click-to-expand). Grab handle is 44×44 minimum even when the card is compressed to 28px tall (handle can visually exceed the card bounds).
- **Focus indicators:** visible 2px outline on the lifted item and the drop-target slot.

### 2.6 Persistence

- **Optimistic updates always.** Update local state on `onDragEnd` *before* the network call. The re-expand animation never waits on the server.
- **Reconcile on failure.** If the server rejects the move, revert with a toast: "Couldn't save that move — try again." Anchored, not global.
- **Snapshot expansion state at drag-start.** In Weekly view, multiple cards can be expanded simultaneously per `TeacherUIState`. Snapshot which were expanded before drag, collapse all to chips, restore the exact set on drop. Do not re-fetch.

### 2.7 Performance (will bite at scale)

- **Memoize each item with `React.memo`** on `(id, density)`. Without this, every item re-renders on every pointer move during drag (which fires onDragOver continuously). At 100+ items this drops below 60fps fast.
- **`content-visibility: auto`** on items far outside the viewport.
- **Fallback to instant density swap** (no `layout` animation) when item count exceeds threshold (suggested: 150 items visible). Framer-motion's `layout` is O(n) in reflow cost.
- **Throttle `onDragOver` collision detection** if the surface has many items. dnd-kit allows custom collision functions.

## 3. Application 1: Weekly View — Lesson Cards Across Cells

### 3.1 What's being dragged

A lesson card. The drop target is a cell in the (subject × day) grid, optionally on a different week visible in the same scrollable canvas.

### 3.2 What collapses

Every lesson card visible on the canvas — across all visible weeks — collapses simultaneously.

**Full state (default):**
- Height: 88px (Simple) / 64–72px (Advanced)
- 6px subject color stripe (left edge)
- Title (16px / 14px depending on mode)
- 2–3 line preview paragraph (140–200 chars from directions)
- Completion checkbox
- Standards count badge
- Resource icons
- Personal-fork indicator
- Master-has-updates indicator
- Comment count badge

**Compact (drag-active) state:**
- Height: 28px
- 6px subject color stripe — **stays full-height, non-negotiable** (this is how teachers orient across the grid; without it, compressed chips become indistinguishable text walls)
- **2-letter subject code prefix** in the title — `MA · Lesson title`, `SC · Lesson title`, `EN · Lesson title`, etc. Same weight as the title, neutral gray-600 color, separated by middle-dot. Only appears in chip mode; full-mode cards omit it. Codes are configured per subject in Settings → Subjects.
- Title: 13px, weight 500, line-clamp 1 (prefix included in the clamp)
- Tiny completion-status dot (4×4px) right of title
- Drag handle on left (still 44px wide for touch — yes, overflows the card bounds visually)
- Everything else hidden: preview paragraph, standards badge, resource icons, indicators, comments

### 3.3 Drop indicator

**Cell-level glow, not an insertion line.** Cells are semantic (subject × day) so the unit of drop is the cell, not a position between elements.

- Empty target cell: dashed 2px border + subject-tint glow (use the same affordance from the resource drag-over spec in §38 of the planning doc)
- Occupied target cell: same glow, plus a subtle "stack" hint badge ("Will stack with [existing lesson title]")

### 3.4 Within-day reordering (the exception)

When the drop target is the **same cell** the lesson came from, the user is reordering within the day (stacked multi-lessons). In this case:
- The cell stays expanded (don't collapse within-cell)
- A horizontal insertion line (2px purple) appears between stacked chips
- Insertion line uses `layoutId="insertion-line"` so it slides between positions

This means the surface-wide collapse fires on drag-start regardless, but within-cell reordering shows finer-grained affordance.

### 3.5 What stays anchored (does not collapse)

- Catch-up bar (top chrome)
- Master/Core mode banner (top chrome)
- Day headers (Sun, Mon, Tue, Wed, Thu)
- Week labels in the left margin
- Day Events banner at the bottom of each column — collapse OR hide entirely during drag (it's not a valid lesson drop target)

### 3.6 Expansion state preservation

Per `TeacherUIState`, weekly view supports multiple simultaneously-expanded inline lesson cards. The pattern must:

1. On drag-start: snapshot `expandedCardIds: Set<string>` from current local state
2. Collapse all (including expanded ones) to chips
3. On drop: re-expand all back to default density
4. Restore inline-expansion only for IDs in the snapshot

Do not write to `TeacherUIState` server-side during this transition — that state didn't actually change from the user's perspective.

### 3.7 Subject color + 2-letter prefix (resolved)

The chip mode ships with **both** the 6px subject stripe **and** a 2-letter subject code prefix in the title. This is belt-and-suspenders, intentionally:

- The stripe is the fast visual scan for sighted users with normal color vision
- The 2-letter prefix handles deuteranopia/protanopia (~8% of male users) and any palette pairs that may collapse at reduced chip scale
- Both together help *all* users scan a dense 30-chip grid faster than either alone

**Implementation notes:**
- Subject codes are configured per subject in **Settings → Subjects** (existing surface — add a new "Code" field, 2-character limit, uppercase, auto-suggested from subject name on creation). Provide a migration/backfill that auto-generates codes for existing subjects (first 2 chars of subject name, deduplicated).
- Codes only appear in chip mode (during drag). Full-mode cards never show them.
- If a teacher has not yet set a code, fall back to first 2 chars of subject name in chip mode rather than blocking the drag.
- The prefix counts against the 13px title's line-clamp budget — acceptable because chip titles already truncate aggressively in dense rows.

**Still worth doing post-launch:** run the 8-subject palette through a color-blindness simulator (Sim Daltonism, Stark, or Adobe's built-in check) to identify any palette pairs that should be hue-shifted regardless. The prefix is a fallback for accessibility; a palette that fails the sim test is still a palette worth fixing.

## 4. Application 2: Daily View — Lesson Section Reorder

### 4.1 What's being dragged

A **section** inside a lesson's right-pane detail view. A lesson can have up to 10 sections (up to 6 text sections like Directions, Notes; up to 4 list sections like Standards, Resources, Strategies, Accommodations).

The drag reorders sections within a single lesson's layout.

### 4.2 Section ordering — data model (resolved)

Section order follows a three-tier precedence model, consistent with the app's existing master/personal fork pattern:

1. **Template** (Settings → Subject template, or a custom template) — defines the default section order for any new lesson created under that subject/template.
2. **Master lesson instance** — when a teacher reorders sections in **master mode**, the change writes to that single master lesson instance only. The template is *not* modified. New lessons created afterward still follow the unchanged template until the template itself is updated in Settings.
3. **Personal lesson instance** — when a teacher reorders sections in **personal mode**, the change auto-forks (if not already forked) and writes to that teacher's personal copy of that single lesson. Same auto-fork pattern as any other personal-mode edit.

**Resolution precedence when rendering a lesson's sections:**

```
Personal copy override?  →  use personal order
  else: Master instance override?  →  use master instance order
    else: use Template order
```

**Template changes are Settings-only.** The reorder drag never modifies templates directly. If a teacher wants their reordering to become the new default for future lessons, they go to Settings → Subject template and update it there. (Worth considering a "Save this order as template default" action in the section header's overflow menu as a future enhancement — out of scope for this build.)

**Surfaces where the reorder is available:** anywhere a lesson's section list is rendered — Daily view right pane, Schedule view right pane (academic block selected), Weekly view inline-expanded cards, Subject view inline-expanded items, Unit view inline-expanded items. The pattern in this section applies identically across all of them; only the host pane geometry differs.

**Divergence indicator (recommended):** when a lesson's section order differs from the template (whether at the master or personal tier), show a subtle "↺ Custom order" chip near the section list header, with hover/tap revealing "Reset to template order." This mirrors the existing "Modified pill" pattern on lesson cards and gives a clear undo path.

### 4.3 What collapses

Every section in the right pane collapses to a one-line header chip.

**Full state (default):**
- Section header (icon + name, e.g., "📝 Directions", "🎯 Standards")
- Full rich-text content (Directions could be paragraphs)
- Embedded media, formatted lists, etc.
- Per-section actions (edit, share, delete from this lesson)

**Compact (drag-active) state:**
- Height: 40px (slightly taller than weekly chips because section types matter more here)
- Drag handle on left (44×44 touch target)
- Section type icon (16px Lucide) — `FileText` for text sections, `List` for list sections, `Target` for standards, `Paperclip` for resources, `Lightbulb` for strategies, `Accessibility` for accommodations
- Section name (14px, weight 500)
- Content summary chip on the right:
  - Text section: "245 words" or "Empty" (subtle gray if empty)
  - List section: "5 standards" / "3 resources" / "Empty"

The summary chip is meaningful — it tells the teacher what's in each section without expanding. Don't omit it.

### 4.4 Drop indicator

Vertical layout means a **horizontal insertion line** between sections (the opposite axis from Padlet's vertical line between cards in a row).

- 2px line, full width of right pane
- Color: purple-500 (same as weekly within-cell reorder)
- Slides between positions using `layoutId="section-insertion-line"`
- 8px vertical padding around the line so it has breathing room

### 4.5 What stays anchored

- Right pane header (lesson title, breadcrumb, close button)
- Right pane footer (comment thread can collapse to "💬 N comments" badge during drag — it's the largest content block and gets in the way)
- Left pane (the day's lesson list) — entirely untouched. This drag is scoped to the right pane only.

### 4.6 Empty sections

A teacher may have an "Accommodations" section configured but empty for this lesson. Show the empty state explicitly in chip mode:
- Type icon dimmed (50% opacity)
- "Empty" text in the summary chip (gray-400)
- Full opacity drag handle (still draggable — reordering an empty section is valid)

Empty sections often *are* the easiest ones to reorder (no content to scroll past), so don't hide them.

### 4.7 Persistence

On drop, write the new section order to the appropriate tier per §4.2:

- **Master mode active:** write to the `MasterLessonInstance.sectionOrder` field for that specific master lesson. Do not touch the template.
- **Personal mode active, personal fork already exists:** write to `PersonalLessonInstance.sectionOrder`.
- **Personal mode active, no personal fork yet:** auto-fork (create the `PersonalLessonInstance`), then write `sectionOrder`. Surface the existing "Now editing your personal copy" affordance the app already shows for other personal-fork triggers.

Optimistic update; revert with inline error anchored to the section list if the server rejects.

After the write succeeds (or even optimistically), update the divergence indicator from §4.2 if the saved order differs from the template's order.

### 4.8 Cross-application interaction note

A teacher in Daily view can drag a section to reorder, *and* drag the lesson itself from the left pane (within-day reorder). These must not conflict:
- Left-pane drag: lesson list, writes `EventDayOrderOverride`
- Right-pane drag: section list, writes to lesson instance

Both use the same collapse-on-drag visual pattern, but they're scoped to their own pane. The board-level `dragState` should be **per-pane**, not truly global, in the Daily view. Two independent state machines, one per pane.

## 5. Application 3: Schedule View — Deferred

Build only after Weekly and Daily prove out. The pattern applies — lessons collapse to chips during drag — but the value is much lower because Schedule view is typically a single-day timeline where all drop targets are already on screen.

If/when built: same chip dimensions as Weekly (28px height, subject stripe full-height). Time labels in the left axis stay anchored. Time blocks themselves don't move (they're set by master schedule); only lessons inside them move.

The right-click "Move to → time block" submenu probably handles 95% of Schedule view moves already. Don't gold-plate this view until users ask for it.

## 6. Component Contracts

### 6.1 Weekly view

```tsx
<WeeklyGrid>
  <DndContext sensors={sensors} onDragStart={...} onDragOver={...} onDragEnd={...}>
    {weeks.map(week =>
      <WeekRow density={density} key={week.id}>
        {subjects.map(subject =>
          <Cell subjectId={subject.id} day={day} density={density} expandedIds={expandedSnapshot ?? expandedIds}>
            {lessonsInCell.map(lesson =>
              <LessonCard {...lesson} density={density} />
            )}
          </Cell>
        )}
      </WeekRow>
    )}
    <DragOverlay>
      {dragState.phase === 'dragging' && <LessonCard id={activeId} density="compact" floating />}
    </DragOverlay>
  </DndContext>
</WeeklyGrid>
```

### 6.2 Daily view (right pane section reorder)

```tsx
<RightPane>
  <PaneHeader />
  <DndContext sensors={sensors} onDragStart={...} onDragOver={...} onDragEnd={...}>
    <SortableContext items={sectionIds} strategy={verticalListSortingStrategy}>
      {sections.map(section =>
        <LessonSection {...section} density={density} />
      )}
    </SortableContext>
    <DragOverlay>
      {dragState.phase === 'dragging' && <LessonSection id={activeId} density="compact" floating />}
    </DragOverlay>
  </DndContext>
  <CommentThread collapsed={dragState.phase !== 'idle'} />
</RightPane>
```

## 7. Shared Visual Tokens

| Token | Value |
|---|---|
| Chip height (weekly card) | 28px |
| Chip height (lesson section) | 40px |
| Subject stripe width | 6px (preserved full-height in chip mode) |
| Drag handle (touch target) | 44×44px minimum |
| Drag handle icon | Lucide `GripVertical`, 16px |
| Drop indicator color | purple-500 |
| Drop indicator thickness | 2px |
| Cell drop-glow color | subject tint, 30% opacity |
| Subject code prefix (chip mode only) | 13px, weight 500, gray-600, format `XX · Title` |
| Chip floating shadow | `0 8px 24px rgba(0,0,0,0.12)` + 2px purple-300 ring |
| Chip floating scale | 1.02 |
| Chip floating opacity | 0.95 |

## 8. Build Order

1. **Universal foundation (§2)** — install dnd-kit, set up the state machine, build the generic `<DragOverlay>` wrapper, wire the motion tokens. ~1 day.
2. **Weekly view application (§3)** — biggest user value. Wire it to actual lesson data, test on a 3-week grid with ~7 subjects, then expand to full 40-week range. ~2–3 days including the within-cell reorder edge case.
3. **Daily view section reorder (§4)** — depends on the §4.2 data-model decision being made. Once that's resolved, ~1–2 days because the surface is contained (single pane).
4. **Test with real teachers** before merging. The collapse pattern can register as "the grid breaking" to users coming from paper planbooks. If feedback is negative, fall back to collapsing only the dragged card's column/section rather than the whole surface.
5. **Schedule view** — only if users ask. May never need it.

## 9. Things to Test Early

- **Color-blindness on the 8-subject palette in chip mode.** At 28px height with reduced visual weight, similar hues may collapse. Test before locking the palette.
- **iPad drag-handle touch ergonomics.** Specifically: does the 44px handle on a 28px card feel awkward or natural? Try both with the handle overflowing the card vertically and with the card itself expanding to 44px minimum on touch devices.
- **Reduced-motion fallback feel.** Without animation, the density swap is instant — does that feel like a bug to users, or like a feature? Consider keeping a 150ms opacity fade even in reduced-motion mode.
- **Cross-week drag scroll behavior.** If the user drags a card to the bottom of the viewport, the canvas should auto-scroll to reveal more weeks. dnd-kit's `autoScroll` handles this but tune the activation zone (default is often too aggressive).
- **Multi-lesson stack reorder inside a single cell.** Edge case: 3 lessons stacked in one cell, user wants to reorder them without leaving the cell. Verify the horizontal insertion line appears correctly and that drag-out-of-cell still works.
