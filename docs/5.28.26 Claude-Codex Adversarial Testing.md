## Code Review Gate (Mandatory)

Before declaring any task "complete" or asking the user to review your work, you
MUST run an adversarial review of your changes through the OTHER agent, using the
command for your role below.

The review invocation itself is **exempt** from this gate: it runs read-only and
is not a "task," so a review never triggers another review.

### If you are Claude Code → review via Codex
Run Codex non-interactively under the read-only sandbox:

    codex exec --sandbox read-only "$REVIEW_PROMPT"

The `--sandbox read-only` flag is mandatory; never weaken it (no
`--dangerously-bypass-approvals-and-sandbox`, no removing the flag to "just see
if it works"). Earlier revisions of this doc included `--approval-policy never`;
that flag was removed in Codex CLI 0.133+ and the command now fails with
"unexpected argument" if it's present. `exec` mode is already non-interactive
and the read-only sandbox blocks every side effect that could otherwise trigger
an approval prompt, so the flag is unnecessary. See CLAUDE.md §4a for the full
invocation contract including the failure protocol when the sandboxed command
cannot run.

### If you are Codex → review via Claude Code
Run Claude Code headless, restricted to read-only tools so it cannot edit and
will not hang on permission prompts:

    claude -p "$REVIEW_PROMPT" --allowedTools "Read,Grep,Glob,Bash(git diff:*),Bash(git diff --cached:*)"

## Shared Review Rubric (`$REVIEW_PROMPT`)

Use this exact instruction as the prompt for the reviewer:

> Act as a strict, skeptical Senior Security & QA Engineer. Review the current
> uncommitted changes (run `git diff` and `git diff --cached` for context). For
> each issue report: file/line, severity (Critical / High / Medium / Low), the
> concrete failure scenario, and a suggested fix. Focus on logic errors, security
> flaws, race conditions, unhandled edge cases, broken error handling, and missing
> or wrong tests. Do not praise; report only problems. If nothing is Medium or
> above, output exactly: NO BLOCKING ISSUES.

---

## Acting on the result (both agents)

- The reviewer can be wrong or hallucinate. Evaluate each finding on its merits —
  do **not** blindly apply its suggestions.
- Fix every Critical and High finding you judge to be legitimate.
- For findings you dismiss, state in one line why (false positive / out of scope /
  intended behavior).
- Re-run the review after fixes. Stop when the reviewer outputs `NO BLOCKING
  ISSUES`, or when the only remaining items are Low/Medium nits you have
  explicitly justified.

---

## Scope (both agents)

Only invoke this gate for changes that affect logic, security, data handling, or
public interfaces. Skip it for trivial edits (comments, formatting, copy changes).

---

## Notes

- The gate is mandatory for in-scope changes — see §Scope above for which
  edits qualify. It is NOT advisory; do not skip "under pressure" or "in long
  sessions." If the sandboxed review command cannot run, report the blocker
  to the user and fall back to the strongest local verification (lint + tsc +
  build + the responsive probe) rather than bypassing the gate. For an
  always-fires version, wire the relevant `codex exec` / `claude -p` command
  into a Stop hook so it triggers automatically when a turn ends.
- The gate assumes both CLIs are installed, authenticated, and on PATH so each
  can invoke the other. Verify this once before relying on the gate.
- Known limitation (Windows, 2026-05-28): Codex's sandboxed `git diff` /
  `git diff --cached` calls can fail with `windows sandbox: spawn setup
  refresh`. Codex falls back to direct file reads + `git status --short`, so
  findings remain valid but the diff-centric prompt loses some context. If
  fidelity matters for a sensitive review, also paste the diff into the prompt
  as additional input.
