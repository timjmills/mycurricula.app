# 5.27.26 UI UX Review of CLaude (Codex)

This is a cross-review of `docs/5.27.26 UI UX Review (Claude).md` against `docs/5.27.26 UI UX Review (Codex).md` and the current repo. I am not merging plans here. This pass only confirms, disputes, or re-scopes audit claims.

## Findings I Confirm

- **Personal/Master consequence needs stronger pre-edit teaching.**
  Claude flags this in `Section 2 - Top 10 first-visit friction points`, item 1 (`docs/5.27.26 UI UX Review (Claude).md:45`) and recommends a first-flip tutorial popover in A1 (`docs/5.27.26 UI UX Review (Claude).md:71`). I agree with the underlying risk. The toggle labels are visible and the Master option has consequence copy in `ariaLabel` and `title`, but that copy is still not the same as an explicit pre-edit cue (`components/shell/top-bar.tsx:473-488`). The banner only appears after `editMode` is already `master` (`components/shell/master-banner.tsx:43-55`) and is mounted above the top bar in planner layout (`app/(planner)/layout.tsx:64-66`).
  My audit partially covered this under `### 14. Master/Core/Personal Terminology Is Inconsistent` and `### 5. "No Confirm Dialogs For Master Mode"` (`docs/5.27.26 UI UX Review (Codex).md:218-275`). I missed the first-flip teaching angle because I framed the risk around terminology and inline edit consequences, not first-use concept teaching.

- **Daily row reorder is local-only and can be misread as curriculum movement.**
  Claude calls this `Weekly-view lesson reorder is local-only` (`docs/5.27.26 UI UX Review (Claude).md:49`). The file cited is actually Daily, not Weekly, but the finding is real for the Daily left-pane row order. The code states that row reorder is "purely a personal viewing preference" and never writes to the planner store or moved flag (`components/daily/DailyView.tsx:54-62`), and the commit path writes only local state plus localStorage (`components/daily/DailyView.tsx:1297-1305`).
  My audit missed this. I focused on prototype trust and no-op controls, but did not inspect whether a visible drag action looked like a shared curriculum edit.

- **Daily narrow mode needs a stronger back affordance.**
  Claude flags the hidden splitter and non-sticky Back button at narrow widths (`docs/5.27.26 UI UX Review (Claude).md:51`) and recommends a sticky back row (`docs/5.27.26 UI UX Review (Claude).md:118`). I agree. The CSS removes the splitter and swaps list/detail panes under `max-width: 720px` (`components/daily/DailyView.module.css:997-1043`), and the Back button is rendered inside the detail pane (`components/daily/DailyView.tsx:1778-1785`) with no sticky positioning in the CSS.
  My audit partially covered Daily pane usability under `### 3. Daily Panes Can Resize To Unusable Widths` (`docs/5.27.26 UI UX Review (Codex).md:51-62`), but I missed this narrow-mode navigation failure.

- **The Subject route deliberately closes the left filter panel, breaking a shell pattern.**
  Claude flags this as a route exception (`docs/5.27.26 UI UX Review (Claude).md:53`). I confirm it. App state defaults `leftPanelOpen` to `true` (`lib/app-state.tsx:217-218`, `lib/app-state.tsx:294`), but `SubjectView` closes it once on mount if it is open (`components/subject/SubjectView.tsx:1016-1027`). The global filter panel also returns `null` when closed (`components/shell/left-filter-panel.tsx:155-157`).
  My audit missed this because I reviewed the top-bar and filter-panel burden at a structural level, not per-route mount behavior.

- **The Year chameleon header is visually meaningful but under-explained.**
  Claude flags the sticky Year header tint as a silent indicator (`docs/5.27.26 UI UX Review (Claude).md:55`) and recommends tooltip/aria enrichment (`docs/5.27.26 UI UX Review (Claude).md:73`). I agree. `QuarterMonthWeekHeader` applies a subject class when `subjectId` is present (`components/year/QuarterMonthWeekHeader.tsx:54-56`), but the accessible label remains the generic `"Year timeline header"` (`components/year/QuarterMonthWeekHeader.tsx:58-63`).
  My audit missed this. I covered the Year header only through disabled coming-soon actions under `### 5. Yearly View Shows Disabled "Coming Soon" Actions` (`docs/5.27.26 UI UX Review (Codex).md:82-88`).

- **Unit Health standards coverage needs explanation.**
  Claude flags the STANDARDS stat label (`docs/5.27.26 UI UX Review (Claude).md:57`) and recommends a tooltip/subtitle (`docs/5.27.26 UI UX Review (Claude).md:74`). I agree. The card shows `STANDARDS`, `SKIPPED`, and `WHEN` as terse stat labels with values (`components/subject/UnitHealthCard.tsx:166-194`), while the only tooltip in that area is for editing the "Don't miss" callout (`components/subject/UnitHealthCard.tsx:202-219`).
  My audit missed this because I focused on Curriculum resource discovery and mobile subject switching rather than Unit Health semantics.

- **Settings scope clarity is real, but more nuanced than tab-level labels.**
  Claude identifies scope clarity as a first-visit friction point (`docs/5.27.26 UI UX Review (Claude).md:18-19`) and proposes TEAM/YOU chips (`docs/5.27.26 UI UX Review (Claude).md:97-101`). I agree with the need. The Settings sidebar registry has no `scope` field (`app/settings/layout.tsx:34-49`). Curriculum cards do show a `Shared with your team` chip (`app/settings/curriculum/page.tsx:281-292`, `app/settings/curriculum/page.tsx:913-928`), but the sidebar itself does not expose scope before a teacher enters a page.
  My audit missed this exact scope-navigation issue. I touched related product-rule leakage through hard-coded Grade 5 and Sun-Thu assumptions, but not Settings scope hierarchy.

- **Rail drag discovery is weaker than the implemented drag behavior.**
  Claude recommends `cursor: grab/grabbing` plus first-session discovery (`docs/5.27.26 UI UX Review (Claude).md:79`). I agree with the discoverability gap. Drag is implemented with `useSortable` and explicit drag hints in rail tooltips (`components/shell/rail-icons.tsx:23-27`, `components/shell/rail-icons.tsx:235-237`), but the visible rail button cursor is still plain `pointer` (`components/shell/GlobalRail.module.css:70-82`).
  My audit partially covered this under `### 12. Global Rails Mix Navigation With Future Feature Markers` (`docs/5.27.26 UI UX Review (Codex).md:191-197`), but I missed the drag-specific discoverability problem.

- **Responsive collapse cliffs deserve a dedicated pass.**
  Claude flags the cumulative Daily collapse sequence at 1280/720/480 (`docs/5.27.26 UI UX Review (Claude).md:63`) and later asks for a 360/768/1280 cliff audit (`docs/5.27.26 UI UX Review (Claude).md:117-119`). I agree. The Daily CSS has multiple breakpoint regimes: schedule/right rail changes around 1280/960, narrow pane mode at 720, and week-strip compression at 480 (`components/daily/DailyView.module.css:499`, `components/daily/DailyView.module.css:997-1043`, `components/daily/DailyView.module.css:1848-1909`).
  My audit covered pieces of this through `### 1. App Chrome Is Doing Too Much`, `### 2. Weekly Right Rail Is Unreachable Below 1280px`, and `### 3. Daily Panes Can Resize To Unusable Widths` (`docs/5.27.26 UI UX Review (Codex).md:19-67`), but did not name the collapse-cascade problem clearly enough.

## Findings I Dispute

- **The in-grid Catch-up bar is not as under-taught as Claude says.**
  Claude says the flame badge and in-grid Catch-up bar do not carry enough concept teaching (`docs/5.27.26 UI UX Review (Claude).md:47`). I agree about the flame badge, but not the in-grid bar. The weekly Catch-up bar wraps its eyebrow in a tooltip explaining that these are lessons the teacher/team fell behind on and that `View all` opens triage (`components/weekly/CatchupWeekBar.tsx:46-60`). The flame badge is thinner: it says `Open Catch-up screen (N items not covered)` (`components/shell/catchup-flame-button.tsx:46-60`). So the finding should be narrowed to the top-bar flame and any non-tooltip touch path, not the whole Catch-up entry system.

- **The phone SOON-tab fallback finding is stale for current code.**
  Claude asks to verify SOON-flagged Yearly/Curriculum tabs in the More menu (`docs/5.27.26 UI UX Review (Claude).md:59`, `docs/5.27.26 UI UX Review (Claude).md:117`). Current `VIEWS` makes Daily, Weekly, Yearly, and Curriculum all live links, not SOON tabs (`components/shell/top-bar.tsx:66-90`). The More menu navigation renders live tabs and skips only `soon` or missing-href tabs (`components/shell/top-bar-more-menu.tsx:207-225`). Since no current primary tab is `soon`, there is no SOON badge fallback to preserve.

- **The proposed Catch-up consequence toast has the wrong scope.**
  Claude proposes a team-scoped toast for Catch-up on/off: "Catch-up cues turned OFF for everyone on your team" (`docs/5.27.26 UI UX Review (Claude).md:87-93`). Current code treats Catch-up cues as user-scoped. The Settings layout documents Catch-up as `USER catch-up cue toggle` (`app/settings/layout.tsx:7-11`), and catch-up state persists in browser localStorage under `mycurricula:catchup-enabled` (`lib/catchup-state.tsx:51-63`, `lib/catchup-state.tsx:134-140`, `lib/catchup-state.tsx:222-230`). A toast is useful, but the copy should say the cue changed for the current teacher unless the product intentionally changes the scope.

- **The HelpDrawer recommendation misses an existing, narrower help surface.**
  Claude proposes a new top-bar `?` help button and route-aware drawer (`docs/5.27.26 UI UX Review (Claude).md:80`). There is already a global `?` shortcuts overlay mounted in planner layout (`app/(planner)/layout.tsx:61-63`, `components/shell/global-shortcuts.tsx:19-37`), and the keyboard dispatcher opens it on `?` (`lib/use-keyboard-shortcuts.ts:199-201`). This does not solve route-aware help or visible first-visit discovery, but it means the implementation should extend or reconcile with the existing help/shortcut system, not create a parallel help surface without an integration decision.

- **"Schedule mode should be in the top-bar Grid/List toggle" is speculative.**
  Claude says teachers may expect the top-bar `Grid | List` toggle to show a timetable (`docs/5.27.26 UI UX Review (Claude).md:61`). I would not treat that as a confirmed UX bug without user testing. Current code labels the top-bar toggle explicitly as layout mode, with `Grid layout` and `List layout` tooltip copy (`components/shell/top-bar.tsx:501-533`). Schedule is intentionally handled as local view chrome and as a dedicated route/drawer (`components/shell/top-bar.tsx:56-59`, `components/daily/DailyView.tsx:1900-1903`, `components/daily/DailyView.tsx:2012-2024`). This is an information-architecture tradeoff, not a clear defect.

## Findings I Want To Reassess On My Own Audit

- **Downgrade `### 3. Mobile P | M Labels` from active product bug to docs cleanup.**
  My audit warned that collapsing Personal/Master to `P | M` is risky (`docs/5.27.26 UI UX Review (Codex).md:253-259`). Current code explicitly does not preserve the short-label behavior and keeps full `Personal` / `Master` labels at every width (`components/shell/top-bar.tsx:465-471`). The audit point is still valid as a standards-doc cleanup, but it should not read like current UI behavior.

- **Strengthen `### 1. App Chrome Is Doing Too Much` with a discovery requirement.**
  My audit says route-specific actions should move out of the top bar (`docs/5.27.26 UI UX Review (Codex).md:19-29`). Claude is right that discovery and concept teaching are structural, not just local tooltip issues (`docs/5.27.26 UI UX Review (Claude).md:14-21`). I would keep the chrome-simplification finding, but add that every removed or relocated global control needs a reliable discovery path, especially on first visit.

- **Adjust `### 5. "No Confirm Dialogs For Master Mode"` to allow a first-use teaching layer.**
  My audit argued that the dangerous moment is the shared edit, not toggling into Master mode (`docs/5.27.26 UI UX Review (Codex).md:269-275`). Claude's first-flip popover is not necessarily a confirm dialog; it can be a one-time concept-teaching layer (`docs/5.27.26 UI UX Review (Claude).md:71`). I would revise my note to say: no recurring confirmation on entering Master mode, but consider both a one-time first-flip explainer and persistent inline cues on shared editable fields.

- **Soften the tone of `### 1. "Every Interactive Control Needs A Tooltip"`.**
  My audit says the universal tooltip rule is too broad (`docs/5.27.26 UI UX Review (Codex).md:235-243`). I still believe that, but Claude correctly points out the app has invested heavily in a canonical tooltip primitive and onboarding voice (`docs/5.27.26 UI UX Review (Claude).md:12`, `docs/5.27.26 UI UX Review (Claude).md:30`). I would rewrite my finding to preserve the primitive and the concept-teaching value while narrowing the mandate for obvious text buttons.

- **Strengthen `### 6. Placeholder Controls Need Clear Prototype Treatment` with concept-teaching distinctions.**
  My audit grouped no-op controls, future markers, and disabled actions together (`docs/5.27.26 UI UX Review (Codex).md:97-112`). Claude separates discovery, concept teaching, consequence feedback, and scope clarity (`docs/5.27.26 UI UX Review (Claude).md:14-19`). That taxonomy is useful. I would keep my placeholder-control concern but not let it absorb all first-visit teaching problems.

## Real Disagreements Worth Surfacing To Tim

- **Tooltip-everywhere rule.**
  Claude's side: the broad tooltip mandate has produced real onboarding coverage, a canonical primitive, and consistent explanatory voice. Removing or weakening it risks a less teachable app.
  Codex side: requiring every button to speak creates noise and can make tooltips feel less meaningful. The rule should be progressive: mandatory for ambiguous, icon-only, disabled, destructive, high-consequence, or non-obvious controls; optional or terse for obvious text actions.

- **Personal/Master vocabulary.**
  Claude's side: `Personal` and `Master` are already embedded in the product doctrine and code, and scope chips can explain consequences without changing core terms.
  Codex side: `Master` and `Core Curriculum` are internal-feeling labels for teachers. The visible pair should be evaluated as teacher language, for example `Personal` / `Team Curriculum`, while implementation can keep `master` internally.

- **Master-mode teaching: one-time popover vs inline shared-edit cues.**
  Claude's side: a first-flip popover teaches the stakes before the first possible shared edit and can be persisted so it does not become recurring friction.
  Codex side: the highest-risk moment is editing a shared field, not toggling modes. Inline cues near editable fields in Master mode may be more durable than a one-time popover that teachers dismiss and forget.
  This is not mutually exclusive, but Tim should decide which one is primary.

- **Top-bar HelpDrawer vs top-bar simplification.**
  Claude's side: a visible `?` help button gives first-time teachers a clear route-aware support entry point.
  Codex side: the top bar is already overloaded. A route-aware help drawer may be right, but its trigger should be reconciled with existing `?` keyboard help and should not become another always-visible global responsibility without removing something else.

- **Schedule as local mode vs global mode.**
  Claude's side: teachers may expect Schedule to sit beside Grid/List because it changes how the planner represents the week/day.
  Codex side: Grid/List is a layout switch for the current surface; Schedule is a timetable surface. Keeping Schedule in local chrome, the rail drawer, and `/schedule` avoids making the global top bar carry every route-specific mode.

## Gaps Neither Audit Covered

- **The Tooltip/Button title fallback appears internally contradictory.**
  `Button` says its `tooltip` prop mirrors to native `title=` as a disabled-button and touch fallback (`components/ui/Button.tsx:24-30`, `components/ui/Button.tsx:140-148`). But `Tooltip` then strips the child `title` attribute when it clones the trigger (`components/ui/Tooltip.tsx:259-269`). This may be intentional to avoid duplicate OS popups, but it means the claimed native long-press fallback may not exist. Neither audit checked touch tooltip behavior against this code path.

- **Existing keyboard help is not integrated into the new-help discussion.**
  The app already has a command/shortcuts help layer: `GlobalShortcuts` mounts `CommandPalette` and `ShortcutsOverlay` (`components/shell/global-shortcuts.tsx:19-37`), and `?` opens the shortcuts overlay (`lib/use-keyboard-shortcuts.ts:199-201`). Claude proposes a new HelpDrawer, and my audit says the top bar is overloaded, but neither audit asks whether the current help surface should become route-aware or get a visible trigger.

- **Subject-to-Daily lesson links may not focus the requested lesson.**
  Subject List mode routes a row click to `/daily?lesson=${l.id}` (`components/subject/SubjectView.tsx:939-944`), but `DailyView` does not read `useSearchParams`; it initializes `selectedId` to the first not-done lesson for the active day (`components/daily/DailyView.tsx:1106-1113`) and changes selection only through internal day/lesson handlers (`components/daily/DailyView.tsx:1272-1285`). Neither audit covered whether cross-route "jump to lesson" links actually land on the intended lesson.

- **Comment drift is becoming an audit risk.**
  Several audit findings were influenced by comments as much as behavior. For example, top-bar comments still say Schedule was dropped from primary navigation and not surfaced in the More menu (`components/shell/top-bar.tsx:55-59`), while other top-bar CSS comments refer to a five-tab strip including Schedule (`components/shell/top-bar.module.css:517-523`). Neither audit explicitly calls out stale comments as a maintenance and audit-quality problem.

- **Neither audit includes completed assistive-tech or touch verification.**
  Claude proposes verification after each wave (`docs/5.27.26 UI UX Review (Claude).md:218-229`), and my audit explicitly says I could not complete a browser viewport pass (`docs/5.27.26 UI UX Review (Codex).md:9`). Neither document records an actual screen-reader pass, keyboard-only route traversal, or touch long-press tooltip test at 360/768/1280.
