## Recommendations for the next pass

1. **Code comments referencing MD files should use stable IDs, not filenames.**
   `lib/lesson-templates.ts` says "see Documents/Project Files/5.17.26
   Onboarding & Lesson-Flow Template Plan" — if that doc is ever renamed
   or moved, the comment silently rots. Same for the 4 "Plugin
   Directions §X" comments and the 1 "collapse_on_drag_pattern" comment.
   A short "see docs/historical/onboarding-template.md §3" with a stable
   relative path under a curated `docs/historical/` folder would be more
   resilient than the dated-filename pattern.
2. **Consider moving the 12 `Documents/Project Files/` historical-reference
   docs into `docs/historical/`** so the project's MD landscape collapses
   to two visible roots: `CLAUDE.md`/`BUILD_STANDARD.md`/`README.md` at
   the root, and everything else under `docs/`. The `Documents/` folder
   would then hold only the design handoff bundle, which is what its
   name actually implies.
3. **The 12 design-handoff READMEs are all under 200 lines each and
   describe the same kind of bundle.** A single
   `Documents/Claude Design/HANDOFF-INDEX.md` listing them with 1-line
   summaries would replace ~10 of the README.md files with a single
   index — but that requires a coordinated pass through each handoff
   folder and is more disruption than this audit was scoped for.
4. **`5.16.26 conversation_record.md` (1898 lines) is the largest
   single historical doc** and may be a candidate for compression in
   a future pass — the decisions it captures are largely re-stated in
   the planning_document and CLAUDE.md. Not deleted today because it
   is explicitly cited in CLAUDE.md §7.
5. **README.md is now ~50 lines and points at CLAUDE.md;** consider
   whether the dev-onboarding scripts table belongs in CLAUDE.md §3
   (so README can shrink to a 10-line "what / where" pointer) or
   whether the current split is good.
