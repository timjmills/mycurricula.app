# Planner Data → Supabase — Ultraplan

> Hook the **planner document** (lessons, units, resources, standards) up to
> Supabase, mirroring the proven Teach pattern (`lib/teach/{supabase-source,
> actions,client}.ts`). Dated 2026-06-01. Branch: `claude/ecstatic-bohr-1ovo3`.
> Companion runbook: `docs/SUPABASE_SETUP.md`. Owner verifies locally first.

## 0. Why this plan

The Teach surface already persists to Supabase end-to-end (schema + RLS + data
source + server-action bridge + client facade, all verified on a live local
stack). The **planner** (Weekly/Daily/Year/Subject/Catch-up) does NOT: every
view reads `lib/planner-store.tsx`, a 1731-line reducer document store seeded
straight from `lib/mock/` (153 lessons, 10 units, 3 standards frameworks, 10
subjects), read across 62 files. The DB tables already exist
(`master_core_lesson_events`, `units`, `resources`, `standards`, `subjects`,
`personal_core_lesson_event_copies`, …) — what's missing is the data-layer
**seam** + an **importer** to load real rows. This plan builds both behind a
flag so the mock stays the default and nothing regresses until the owner flips
it on.

## 1. Current-state facts (verified)

- `lib/planner-store.tsx`: `useReducer(historyReducer, INITIAL_REDUCER_STATE)`,
  structural sharing + undo/redo. Initial doc = `{ lessons: [...LESSONS],
  cellLayouts, … }` from `lib/mock`. Public API `usePlanner()` exposes
  `lessons`, `getLesson`, `getSections`, and ~30 mutators (moveLesson,
  editLesson, setLessonStatus, duplicate*, section CRUD, resource CRUD, undo/…).
- Domain `Lesson` (lib/types.ts) is FLAT: `{id, subject, unit, title, objective,
  preview, directions, notes, resources[], standards[], week, day, isPersonal,
  pendingMaster, modified, moved, status, tasks[], …}`.
- DB `master_core_lesson_events` is NORMALIZED + different names: `unit_id`,
  `subject_id`, `week_number`, `day_of_week` (enum `weekday`), `learning_objectives`
  jsonb, `resources` jsonb mirror, `standards uuid[]`, soft-delete `deleted_at`.
  Personal forks live in `personal_core_lesson_event_copies` (lazy, §4.3).
- `Unit` (flat: `{id, subject, name, weeks, shade}`) vs DB `units` (normalized).
- Sections (`LessonSectionContent` w/ `SectionResource[]`) come from
  `lib/lesson-flow.ts` derived per-lesson; the DB has a richer `resources` table
  (`owner_event_type`/`owner_event_id` polymorphic) + the jsonb mirror on the
  lesson row. v1 maps to the lesson-row jsonb mirror (matches the mock shape);
  the full `resources` table is a later normalization.
- **No id bridge exists yet** (mock uses slug ids like `m-12-0`; DB uses uuids).
  The Teach source left explicit hooks for one. We need a real bridge.
- RLS requires the signed-in user to have a `teachers` row AND a
  `teacher_grade_assignments` row (`can_read_grade`). The Claude bypass + Google
  login currently create only an auth user → **provisioning gap** (Part B).

## 2. Scope (owner-approved)

ALL planner data: lessons + units + resources + standards + subjects. Build the

## 3. Architecture (mirror Teach exactly)

```
planner-store.tsx (client, reducer)         ← unchanged mutators + undo/redo
  │  hydrate initial doc from ↓ ; persist mutations through ↓
lib/planner/client.ts   (teachClient-style facade; flag picks mock vs server)
  │                         server actions (only when flag on)
lib/planner/actions.ts  ("use server"; generic dispatch to the source)
  │
lib/planner/supabase-source.ts (server-only; supabase/server.ts + RLS)
lib/planner/mock-source.ts     (wraps lib/mock — the default, isomorphic)
lib/planner/source.ts          (PlannerDataSource interface + isConfigured + bridge)
```

- **PlannerDataSource interface** (LEAD owns — the frozen contract). Async,
  Promise-returning, mirrors what `usePlanner` needs to hydrate + persist:
  - `listLessons(gradeLevelId, ownerId): Lesson[]` (personal-fork-resolved,
    soft-deletes excluded)
  - `listUnits(gradeLevelId): Unit[]`, `listSubjects(gradeLevelId): Subject[]`,
    `listStandards(gradeLevelId): Standard[]`
  - `getSections(lessonId): LessonSectionContent[]`
  - mutators the reducer commits: `updateLesson(id, patch)`, `moveLesson(...)`,
    `setLessonStatus(...)`, `createLesson(...)`, `softDeleteLesson(id)`,
    `setSections(lessonId, sections)`, resource add/edit/remove.
  - **Forking semantics (§4.3):** an edit in personal mode writes a
    `personal_core_lesson_event_copies` row (lazy fork); reads resolve personal
    over master. The source owns this; the reducer is unaware.
- **Hydration:** `PlannerProvider` gains an optional `initialDoc` prop. A server
  component (or a load effect, flag-gated) fetches via the source and seeds the
  reducer; absent the flag it uses `[...LESSONS]` exactly as today.
- **Persistence:** mutators stay synchronous reducer updates for snappy UX
  (optimistic), then fire-and-await the matching source call; on error, surface
  a toast + reload (no silent divergence). Undo/redo stays client-side; a
  committed undo re-issues the inverse persist.
- **Mapping module** (`lib/planner/map.ts`): pure row↔domain mappers (snake↔camel,
  `day_of_week` enum↔0–4 index, `week_number`↔`week`, jsonb objectives/resources,
  `standards uuid[]`↔slug[] via the id bridge). Unit-tested, no I/O.
- **Id bridge** (`lib/planner/id-bridge.ts`): slug↔uuid resolution shared by
  planner + Teach sources (Teach already wants it). Backed by a per-request
  cache of `subjects`/`units` natural keys.

## 4. Privacy + grade-scoping (hard rules)

- §11.4: planner rows carry NO student names (they don't today — assert in the
  importer + a grep gate). Names stay in the local groups store.
- Every query is grade-scoped (`grade_level_id`) and owner/RLS-scoped — never a
  bare table scan. Multi-grade-ready: no single-grade assumption in any mapper.

## 5. The importer (so the DB isn't empty)

`scripts/import-mock-planner.mjs` (service-role, local/seed only): idempotent
upserts of `lib/mock` → DB. Order: subjects → units → standards frameworks +
standards → master_core_lesson_events (+ jsonb resources). Deterministic uuids
(uuid v5 from the mock slug) so re-runs are stable AND the id bridge is trivial.
Also provisions a dev teacher + grade assignment (closes the Part-B gap for
local testing). Prints a per-table inserted/updated count. NEVER run against
prod without the runbook's explicit step.

## 6. Auth/teacher provisioning gap (Part B — prerequisite)

Login/bypass creates an auth user but no `teachers` + `teacher_grade_assignments`
row, so RLS denies everything (verified: insert → "violates RLS"). Fix in the
auth callback + claude-bypass: on first authenticated request, upsert a
`teachers` row (id = auth uid, default school/grade from config or first
grade) + a `teacher_grade_assignments` row. Idempotent. This is REQUIRED before
either Teach or planner persistence works for a real user.

## 8. Risks / decisions

- **Big store, structural sharing:** do NOT rewrite the reducer. Only (a) seed
  its initial doc, (b) tee mutators to the facade. Undo/redo stays in-memory.
- **Forking model must not break (CLAUDE.md §2):** personal edits write the
  copies table, never master; completion never forks; master needs explicit
  mode. The source enforces; reducer flags (`isPersonal`/`pendingMaster`/
  `modified`) map to/from the copies table.
- **Perf:** hydrate once per session; mutations are per-row, optimistic. No
  N+1 — batch the initial lesson+sections+resources fetch.
- **Stop conditions:** schema lacks a needed column → add an additive migration
  (like `20260531120000`), don't hack the mapper; RLS shape ambiguous → surface,
  don't weaken; mock-default regression → block.
