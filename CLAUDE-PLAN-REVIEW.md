## VERDICT: **GO-WITH-CHANGES**

The plan is **fundamentally sound and should proceed** — but **not as written**. Three findings are blocking-before-coding (one Critical, two High), and the plan's headline framings ("re-skin / preserve plumbing", "complete the backend source", "reuse backend as-is") are **over-claims contradicted by code** that will mislead implementers into under-budgeting and into regressing live-prod paths.

What's genuinely de-risked (verified): **the chosen base builds clean** (tsc + lint + `next build` all green at HEAD `882a1f4`, no error suppression); **the persistent non-remounting shell already exists** (`app/(planner)/layout.tsx`) so D4's hard part is free; **the backend is real and prod-live** (not a stub); and **D1's subject remap is safe** (color is a derived slug — no data migration). The risk is concentrated where the plan calls things easy: the appearance-engine port (its own "critical path") and the "presentation-only" rebuild of the core view primitives.

## Detail on the blocking three

**F1 (Critical) — silent theme reset for every existing user.** Independently surfaced by TWO lenses (appearance-engine + feasibility). The current app stores `data-theme ∈ {paper,cloud,night,mint,sky,blossom}` in `localStorage` (`mycurricula:user:theme`) and `teacher_preferences.theme` (CHECK-constrained). The v2 model uses a different value set AND the three v2 sources don't agree on it (bundle keeps `night` as a theme + derives tone; `modes.css` drops `night`/`off` and makes tone the dark axis; `themes.css` re-adds `night`). The plan (§0 D2) says "`data-theme` splits into `data-theme`+`data-tone`" but never publishes the resulting value lists or a migration. On first post-deploy load, the boot script (`theme-init.tsx:42`) falls back to default for any unrecognized value → **paper/cloud/night users silently lose their theme**, and a CHECK change without a row migration rejects/strands existing rows. This is the single must-fix-first.

**F2 (High) — the "re-skin / preserve presentation" premise is half-true.** "Preserve the data/state/hook layer" is verified-feasible; "rebuild only the CSS" is contradicted for the core surfaces, which weld material into TSX inline-styles, JS-measured px, dnd DOM shape, and the `data-style` axis v2 deletes. `lesson-chip.tsx` and `card-stack.tsx` prove the separation is *achievable* — the rest of the app simply wasn't built to it. The 20%/30%/50% effort split is optimistic; the "re-skin" bucket contains rewrites.

**F4 (High) — the product differentiator is absent from the source design.** The three-tier forking cue exists in v2 only as one gated Plan-header pill + spec prose; Tier 3 isn't implemented at all, and the v2 lesson data model carries no fork/move state. The plan's "carry it into each frame by hand" is the right call but is hand-waved — it must be a scheduled, verified deliverable, sourced from current code (which has all three tiers), or the rebuild silently ships without the feature.

## Addendum — final deep-dive lens reports (folded in)

The five lenses' final reports corroborated the findings above and sharpened five points (all VERIFIED-AGAINST-CODE at HEAD `882a1f4`):

- **A1 [High] — porting `modes.css` faithfully SILENTLY LOSES the photo-grading system (sharpens F1).** `modes.css` is only ~⅓ of the v2 engine: it uses `data-frame`/`wash` and keys frame *materials*, but the **canonical bundle** (the behavioral source of truth per the handoff README) uses `data-version`/`ambient` + `data-canvas`/`data-veil`/`data-dim`/`data-mode`/`data-zoom`, and the premium 4-layer photo grading (auto-luminance/duotone/brightness) lives in `themes.css` keyed on **those axes `modes.css` lacks**. D2 adopts the `modes.css` vocabulary the canonical source doesn't implement → an implementer who "ports `modes.css`" gets frame materials but drops photo grading. **Reconcile the engine vocabulary to the bundle before Phase 0; treat `modes.css` as ⅓ of the engine.**
- **A2 [High] — the §7 "presentation-scoped diff" verify gate is unachievable.** The plan's `pipeline(SURFACES, build, verify)` asserts each surface diff is "presentation-scoped, no data/state change." For Day/Week/Year/Card that is false (F2: they need markup/state changes), so the gate **false-positives on every one of those surfaces** → it gets disabled (losing its value) or the rebuild stalls. Down-scope it to "no change to the data/logic core (stores/hooks/types)", not "no state change."
- **A3 [Medium, sharpens F6] — the Night block must be DECOMPOSED, not "split" in one bullet.** `data-tone="dark"` (`modes.css:333-337`) redefines the exact neutrals the current `[data-theme="night"]` block sets (`tokens.css:1385-1417`), and that block fuses accent + neutral + subject/tag/highlight `color-mix` recipes (`tokens.css:1498-1594`). The split must move neutrals→`data-tone`, keep accent on `data-theme`, and **re-key the subject/tag/hl recipes onto `data-tone`** — or last-wins-by-source-order silently corrupts dark mode.
- **A4 [Medium, sharpens F3] — `theme_style` is a deployed-prod `NOT NULL DEFAULT 'vivid'` column.** Dropping `data-style` can't just stop writing it; needs a follow-up migration (nullable/drop). A new engine that fails the stale CHECK makes `theme-sync` **silently stop** (it swallows errors by design, `theme-sync.ts:137-139`).
- **A5 [Low] — two plan-accuracy fixes.** (a) `YearView.tsx` is **RETIRED dead code** (`:5-8`); §8 critical-files should target `TimelineYear`/`RoadmapView` (and Year Frame C is net-new constellation anyway). (b) Drop the "v2's hardcoded `repeat(5)` must be adopted" framing — the `--day-count` school-week contract **already exists** (`WeeklyGrid.module.css:189`); v2's mockup must adopt IT, not the reverse.

These refine, but do not change, the **GO-WITH-CHANGES** verdict and the must-resolve set (A1 attaches to F1, A2 is a new orchestration must-fix alongside F11).
